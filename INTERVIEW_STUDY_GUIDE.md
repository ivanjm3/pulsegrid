## System Overview

```
                                   ┌──────────────┐
                                   │    Client    │
                                   └──────┬───────┘
                                          │ POST /videos/upload (multipart, ≤10GB)
                                          ▼
                            ┌─────────────────────────┐
                            │      API (cmd/api)       │   stateless, horizontally
                            │  :8080 HTTP + /metrics   │   scaled (2 replicas)
                            └───┬─────────────┬────────┘
                     stream     │             │ 1. INSERT status=submitting (tx)
                     upload     │             │ 2. publish Kafka message
                                ▼             │ 3. UPDATE status=submitted, COMMIT
                          ┌──────────┐        ▼
                          │    S3    │  ┌──────────────┐        ┌───────────────┐
                          │  source  │  │  Postgres     │        │  Kafka topic   │
                          │  bucket  │  │  (RDS)        │        │ transcoding-   │
                          └──────────┘  └──────────────┘        │ jobs           │
                                                                  └──────┬────────┘
                                                                         │ consumer group
                                                                         │ pulsegrid-workers
                                                                         ▼
                                                          ┌───────────────────────────┐
                                                          │   Worker (cmd/worker)      │
                                                          │  :8081 /metrics only       │
                                                          │  scaled 0-30 by KEDA        │
                                                          │  (Kafka consumer lag)       │
                                                          └───┬──────────────┬─────────┘
                                                  download    │              │ upload outputs +
                                                  source       ▼              ▼ manifest.json
                                                          ┌──────────┐  ┌──────────┐
                                                          │    S3    │  │    S3    │
                                                          │  source  │  │  output  │
                                                          │  bucket  │  │  bucket  │
                                                          └──────────┘  └──────────┘

              failure path: worker classifies error (retryable / permanent / constraint)
              → re-enqueue to transcoding-jobs (retry_count++) | publish to transcoding-dlq | os.Exit(1)
```

**Two Go binaries, one shared library.** `cmd/api` and `cmd/worker` are independent `package main` entrypoints; everything they share — models, S3/Kafka/Postgres clients, retry logic, error classification, metrics, logging, ffmpeg invocation — lives in `pkg/`. This is the single most important structural decision in the repo, and it's the first thing worth being able to explain (see [Part 1](#project-layout--packages)).

**The critical unresolved gap** (documented in `implementation-audit.md` and `USER_GUIDE.md`'s "Known Limitations"): the worker never writes job completion status back to Postgres. It has no `pkg.DBClient` at all — `cmd/worker/main.go` imports `pulsegrid/pkg` but never calls `pkg.NewPostgresClient`. This single gap is a goldmine for interview discussion because it's a real, honest, still-open architectural seam — not a contrived example. It's referenced throughout this guide wherever relevant.

## Part 1 — Go Engineering

### Project Layout & Packages

| | |
|---|---|
| **What it is** | Two `cmd/` entrypoints (`api`, `worker`, plus `load-test`) sharing one `pkg/` library, with `db/` for migrations and `pkg/queue/` as a second, more abstract Kafka wrapper living alongside `pkg/kafka.go`. |
| **Why it exists** | Go has no built-in module boundary weaker than a package; the convention (`cmd/<binary>` + shared `pkg/` or `internal/`) is how the ecosystem fakes "shared library, multiple executables" without a monorepo build tool. |
| **Why this approach** | API and worker have almost disjoint responsibilities (HTTP request handling vs. long-running subprocess orchestration) but need identical wire formats (`pkg.Job`, `pkg.KafkaMessage`), identical retry semantics, and identical error classification. Splitting them into two binaries lets them scale independently (2 API replicas vs. 0-30 KEDA-scaled workers) and have wildly different resource profiles (API: 1-2 CPU/1-2Gi mem; worker: 2-4 CPU/4-8Gi mem, `kube/api-deployment.yaml` vs. `kube/worker-deployment.yaml`) and even different container images (`Dockerfile.api` has no ffmpeg; `Dockerfile.worker` does). |
| **Alternatives** | (1) `internal/` instead of `pkg/` — would prevent external modules from importing this code, which doesn't matter here since nothing external consumes it, but is the idiomatic choice when you *do* want to prevent that. (2) A monorepo with separate Go modules per service — adds `go.mod` per service, better isolation, worse for a two-service repo. (3) One binary with a `--mode=api|worker` flag — simpler deploy story, but couples resource requests/limits and blast radius (an OOM in transcoding shouldn't be able to take down the HTTP tier). |
| **Tradeoffs** | Two binaries sharing `pkg/` means a `pkg/` change potentially needs two rebuilds/redeploys and two sets of tests passing. It also means the shared types (`pkg.Job`, `pkg.KafkaMessage`) become a de facto contract between two independently-deployed services — a schema change to `pkg.KafkaMessage` deployed to the API before the worker rolls out is a live compatibility hazard. |
| **Common mistakes** | Importing `cmd/api` types from `cmd/worker` or vice versa (Go allows internal cross-`cmd/` imports since both are `package main`, but you can't import symbols from another `main` package — this is actually why the integration test in `test/integration/full_lifecycle_test.go` **re-implements** `handleUpload`/`handleGetJob` as `apiServer` methods locally rather than importing them from `cmd/api`, see line 251-260: "Since cmd/api is package main, we reimplement the upload+status handlers"). This is a real Go language constraint, not sloppiness, and it's worth being able to explain in an interview. |
| **Performance implications** | None directly — but two independently-scaled binaries with independent build artifacts means Docker layer caching and CI build parallelism benefit from the split (see `.github/workflows/ci.yml`'s separate `Build API` / `Build Worker` steps). |
| **Production considerations** | Coordinated schema changes to `pkg/` types require careful rollout order (usually: worker first if it's tolerant of new fields, consumer-before-producer for wire format additions). |
| **Security considerations** | Package-private (`s3Uploader`, `dbClient` etc. are lowercase package-level vars in `cmd/api/main.go`) keeps global mutable state out of `pkg/`'s public API — nothing in `pkg/` reaches back into `cmd/api` globals. |
| **Scalability considerations** | This is *the* structural enabler of independent scaling — see [KEDA autoscaling](#keda--horizontal-worker-autoscaling). |
| **Failure scenarios** | A panic in worker's ffmpeg subprocess handling cannot cascade into the API process — they're separate OS processes/pods by construction, not by discipline. |
| **How this repo implements it** | `cmd/api/main.go`, `cmd/worker/main.go`, `cmd/load-test/{main,harness}.go` are the three entrypoints. `pkg/` holds all shared logic: `models.go` (types), `s3client.go`, `kafka.go`, `postgres.go`, `retry.go`, `errors.go`, `metrics.go`, `logger.go`, `transcode.go`, `manifest.go`, `jobid.go`, `queue_depth.go`. `pkg/queue/kafka.go` is a *separate*, more abstract `MessageQueue` interface (`Publish`/`Consume`/`Commit`/`SendDLQ`) that duplicates most of `pkg/kafka.go`'s `KafkaProducer` — worth asking about (see interview questions). `db/` is its own package for migrations, imported only by `cmd/api`. |
| **Relevant files** | `cmd/api/main.go`, `cmd/worker/main.go`, `pkg/*.go`, `pkg/queue/kafka.go`, `db/migrate.go` |

### Interfaces & Dependency Injection

| | |
|---|---|
| **What it is** | Every external dependency (S3, Kafka, Postgres) is behind a small Go interface (`S3Uploader`, `S3Downloader`, `S3OutputUploader`, `KafkaProducer`, `DBClient`), and the concrete implementation is assigned to a package-level `var` that is `nil` unless the corresponding environment variable is set. |
| **Why it exists** | Go has no constructor injection framework by convention (no Spring, no Guice) — the idiomatic pattern is "accept an interface, return a struct," and wire concrete types at `main()`. |
| **Why this approach** | Two forces converge on this design: (1) **testability** — every handler and worker function can be tested by substituting a mock struct that implements the same tiny interface (see `test/integration/full_lifecycle_test.go`'s `mockS3Uploader`, `mockKafkaProducer`, `mockDB`); (2) **local-dev fallback** — the interfaces are nil-checked everywhere (`if s3Uploader != nil { ... } else { ... local mode ... }`), so a developer with no AWS/Kafka/Postgres can still exercise the HTTP layer end-to-end. This is a real, deliberate design choice (`USER_GUIDE.md` calls it out explicitly under "Local development fallback"). |
| **Alternatives** | A DI container (`google/wire`, `uber/fx`) — generates or runtime-resolves the dependency graph. Overkill for two services with ~4 dependencies each; adds a build step (`wire`) or reflection-based runtime cost (`fx`) for no real benefit at this scale. |
| **Tradeoffs** | Package-level mutable `var`s (`s3Uploader`, `kafkaProducer`, `dbClient`, `apiMetrics` in `cmd/api/main.go:27-37`) are global state — convenient for a small `main` package, actively hostile to unit-testing handler functions in isolation (you can't parallelize tests that mutate these without a mutex or `t.Parallel()` hazard) and impossible to have two independently-configured instances in the same process. The `test/integration` package works around this by not importing `cmd/api` at all — it *reimplements* the handlers as methods on a struct (`apiServer`) that holds injected fields, which is strictly better DI than what `cmd/api/main.go` itself does. |
| **Common mistakes** | Forgetting a nil-check before calling into one of these interfaces — a scan of `cmd/api/main.go` shows every call site is properly guarded (`if s3Uploader != nil`, `if kafkaProducer != nil`, `if dbClient != nil`), but this is a manual discipline, not compiler-enforced; a new handler that forgets the nil-check will panic in local-dev mode. |
| **Performance implications** | Interface method calls in Go go through an itable indirection (roughly the cost of one extra pointer dereference vs. a concrete-type call) — irrelevant here since these calls are all I/O-bound (network to S3/Kafka/Postgres), where microsecond-level dispatch overhead is noise. |
| **Production considerations** | In prod, all four env vars are set (`AWS_REGION`, `DATABASE_URL`, `KAFKA_BROKERS`), so the nil-mode branches are dead code in production but load-bearing in CI/local dev and in `test/integration`. |
| **Security considerations** | N/A directly, but see [Secrets](#secrets--credential-management) — the same env-var-driven pattern is how credentials flow in. |
| **Scalability considerations** | N/A — this is a per-process wiring concern, not a scaling concern. |
| **Failure scenarios** | If `AWS_REGION` is set but AWS creds are wrong, `s3Uploader` is non-nil but every call fails at request time — this is a *worse* failure mode than "not configured" because it's silent until traffic arrives; `GET /health` is the mitigation (it actively pings each dependency). |
| **How this repo implements it** | Interfaces: `pkg.S3Uploader`, `pkg.S3Downloader`, `pkg.S3OutputUploader` (`pkg/s3client.go:28,104,110`), `pkg.KafkaProducer` (`pkg/kafka.go:15`), `pkg.DBClient` (`pkg/postgres.go:53`). All satisfied by concrete types `*pkg.S3Client`, `*pkg.KafkaClient`, `*pkg.PostgresClient`. Wired in `main()` of each binary based on env var presence. |
| **Relevant files** | `pkg/s3client.go`, `pkg/kafka.go`, `pkg/postgres.go`, `cmd/api/main.go:27-131`, `cmd/worker/main.go:25-91` |

### Error Handling & Error Wrapping

| | |
|---|---|
| **What it is** | Go's `error` is a value, not an exception. This repo layers three custom error *types* (`TranscodingError`, `SourceNotFoundError`, `ResourceConstraintError`, `PermanentError`) on top of stdlib `errors.New`/`fmt.Errorf("...: %w", err)`, and a single classification function (`pkg.ClassifyError`) that inspects an error chain to decide retry/DLQ/pod-exit behavior. |
| **Why it exists** | Go deliberately has no exceptions — errors are ordinary return values, forcing every call site to explicitly decide what to do with failure instead of letting it propagate implicitly up a stack. |
| **Why this approach** | The worker's retry/DLQ/pod-exit decision (`handleJobOutcome` in `cmd/worker/main.go:198-276`) needs to distinguish three fundamentally different failure classes with different remediation, and doing that via string-matching on error messages *alone* would be fragile. So the repo defines typed errors for the unambiguous cases (`*pkg.ResourceConstraintError` → pod must exit; `*pkg.PermanentError`/`*pkg.SourceNotFoundError` → DLQ immediately) and falls back to substring matching on `err.Error()` for everything else (`pkg/errors.go:97-129` — patterns like `"unsupported codec"`, `"out of memory"`, `"no space left on device"`). `errors.As` walks the wrapped chain to find a typed error even if it's several `fmt.Errorf("...: %w", err)` layers deep — e.g. `processJob` wraps `s3Downloader.DownloadSourceFromS3`'s error with `fmt.Errorf("source download failed: %w", err)` (`cmd/worker/main.go:314`), and `ClassifyError` still finds the underlying `*pkg.SourceNotFoundError` via `errors.As`. |
| **Alternatives** | (1) Sentinel errors only (`errors.Is(err, ErrSourceNotFound)`) — simpler, but loses per-instance context (which job, which S3 URI) that `SourceNotFoundError`'s fields carry. (2) Error codes as plain strings/ints on a single error struct — what many Go services do; less idiomatic, loses the ability to `errors.As` into a specific concrete type when you need its fields. (3) A full "problem details" (RFC 7807) error object threaded through every layer — heavier, more explicit, arguably better for the HTTP-facing side (which this repo does approximate with `ErrorResponse` in `cmd/api/main.go:40-46`) but overkill for internal worker-to-worker error flow. |
| **Tradeoffs** | Substring matching on `err.Error()` (`pkg/errors.go:98-126`) is inherently brittle — a vendor library changing its error message text (e.g., AWS SDK v2 wording "no space left on device" differently in a future version) silently reclassifies a resource constraint as retryable, causing infinite retry loops instead of a clean pod exit. This is a deliberate pragmatic tradeoff (typed errors for the cases the code controls, string matching for the cases it doesn't — `exec.Command` stderr, syscall-level disk-full errors) rather than an oversight. |
| **Common mistakes** | Losing the error chain by using `fmt.Errorf("...: %v", err)` instead of `%w` (breaks `errors.As`/`errors.Is` downstream) — this repo consistently uses `%w`, worth checking in review. Another common mistake this repo *avoids*: comparing errors with `==` instead of `errors.Is`/`errors.As` (pointer equality breaks the moment an error is wrapped). |
| **Performance implications** | `errors.As` walks the `Unwrap()` chain linearly — O(depth), negligible for the 2-3 levels of wrapping in this codebase. |
| **Production considerations** | `ClassifyError`'s substring list is the single most important tuning surface for retry economics in production — adding a new permanent-failure signature (e.g., a specific ffmpeg exit message for an unsupported container format) is a one-line change but directly controls DLQ growth rate. |
| **Security considerations** | Error messages containing `err.Error()` are surfaced directly to HTTP clients in some paths (`writeErrorWithRequestID(..., err.Error())` in `handleVideoUpload`, `cmd/api/main.go:296`) — this leaks internal details (e.g. raw Postgres/S3 SDK error text) to the client. Acceptable for an internal API, a real information-disclosure smell for a public-facing one. |
| **Scalability considerations** | N/A directly. |
| **Failure scenarios** | A DB connection string typo produces a Postgres error deep in `pgxpool` — wrapped through `NewPostgresClient` → `fmt.Errorf("postgres connection failed: %w", err)` → `log.Fatalf` in `main()`. The API pod crash-loops with a clear log line rather than starting in a half-initialized state. |
| **How this repo implements it** | `pkg/errors.go` defines the type hierarchy and `ClassifyError`. Every I/O call site wraps with `%w` and job-id context. `cmd/worker/main.go:213-227` is the consumption point. |
| **Relevant files** | `pkg/errors.go`, `pkg/errors_test.go` (15 `TestClassifyError_*` cases per `implementation-audit.md`), `cmd/worker/main.go:198-276` |

### Context, Cancellation & Timeouts

| | |
|---|---|
| **What it is** | `context.Context` propagates deadlines, cancellation signals, and (unused here) request-scoped values through call chains. |
| **Why it exists** | Without `context`, a goroutine blocked on network I/O (an S3 GET, a Postgres query) has no standard way to be told "give up" from outside — you'd need ad-hoc channels/timers per call site. |
| **Why this approach** | Every I/O boundary in this repo takes a `context.Context` as its first argument (Go convention). Three distinct patterns show up: (1) **request-scoped** — `handleHealth` derives `context.WithTimeout(r.Context(), 5*time.Second)` so a slow Postgres/Kafka/S3 ping can't hang a health check indefinitely (`cmd/api/main.go:165-166`); (2) **operation-scoped** — `TranscodeSingleRendition`/`TranscodeHLS` wrap the parent context in a 30-minute timeout (`DefaultTranscodeTimeout`, `pkg/transcode.go:23,61,139`) so a hung/looping ffmpeg process is killed rather than blocking a worker slot forever; (3) **poll-loop-scoped** — the worker's main loop creates a fresh `context.WithTimeout(context.Background(), 10*time.Second)` per `FetchMessage` call (`cmd/worker/main.go:139-141`) specifically so the blocking Kafka fetch can be interrupted periodically to check the `shutdownRequested` atomic flag. |
| **Alternatives** | Passing a raw `chan struct{}` "done" signal predates `context` in Go and is what `context` replaced — strictly worse because it doesn't compose (no built-in deadline, no value propagation, no standard cancellation-reason). |
| **Tradeoffs** | `context.WithTimeout` allocates a timer and a small heap object per call — negligible at this request volume, but the poll-loop's per-iteration `context.WithTimeout(context.Background(), 10*time.Second)` (`cmd/worker/main.go:139`) means up to 10 seconds of latency between a SIGTERM and the worker noticing it mid-fetch, which is why the deployment sets `terminationGracePeriodSeconds: 1800` (30 min, `kube/worker-deployment.yaml:25`) — that grace period has to cover both this polling latency *and* an in-flight 30-minute transcode. |
| **Common mistakes** | Forgetting `defer cancel()` after `context.WithTimeout`/`WithCancel` — leaks the internal timer goroutine until the parent context is itself cancelled. Every call site in this repo does call `defer cancel()` correctly (`cmd/api/main.go:166`, `cmd/worker/main.go:141` via explicit `cancel()` not defer since it's in a loop, `pkg/transcode.go:62,140`). Note `cmd/worker/main.go:141` calls `cancel()` directly (not deferred) because it's inside a `for` loop — deferring inside a loop body would leak until function return, not iteration end; the code correctly avoids that trap. |
| **Performance implications** | `context.Background()` in `processJob` (`cmd/worker/main.go:303`) is a **deliberate root context**, not derived from the poll loop's per-fetch context — meaning once a message is pulled off Kafka, the actual transcode work is *not* cancelled if the worker receives SIGTERM mid-job. This is intentional: the 30-minute `terminationGracePeriodSeconds` exists specifically to let in-flight jobs finish rather than being killed mid-transcode (partial output, wasted work, retry from scratch). |
| **Production considerations** | The `SessionTimeout: 30 * time.Minute` on the Kafka reader (`cmd/worker/main.go:99`) and `terminationGracePeriodSeconds: 1800` are the same number for a reason — both must exceed the worst-case single-job transcode time, or Kafka will trigger a consumer group rebalance (redelivering the "in progress" message to another consumer) or Kubernetes will SIGKILL the pod before the job finishes. |
| **Security considerations** | N/A directly. |
| **Scalability considerations** | Long session timeouts trade rebalance safety for slower failure detection — if a worker pod actually dies (not gracefully) mid-job, Kafka won't reassign that partition's messages to another consumer until the 30-minute session timeout expires, meaning a crashed worker can silently stall a partition's throughput for up to 30 minutes. |
| **Failure scenarios** | Worker crashes (OOM-killed, not graceful SIGTERM) mid-transcode → Kafka doesn't get a heartbeat → after `SessionTimeout` (30 min) the consumer group rebalances and another worker re-fetches the *uncommitted* offset → job reprocessed from scratch. This is correct at-least-once behavior but means a crash has a 30-minute blast radius on that one partition before recovery starts. |
| **How this repo implements it** | See patterns above. `r.Context()` is used for all HTTP-handler-scoped calls (auto-cancelled if client disconnects); `context.Background()` is used for the worker's core processing (deliberately shutdown-independent). |
| **Relevant files** | `cmd/api/main.go:165-166,240-241` (`r.Context()` propagation via `http.MaxBytesReader`+multipart parsing), `cmd/worker/main.go:139-141,303`, `pkg/transcode.go:61-62,139-140` |

### Goroutines, Channels, Worker Pools & Synchronization

| | |
|---|---|
| **What it is** | Lightweight (2KB initial stack) green threads scheduled by the Go runtime (M:N scheduler onto OS threads), coordinated via channels (CSP-style message passing) or `sync` primitives (mutexes, atomics). |
| **Why it exists** | Goroutines let a single process handle massive I/O concurrency (thousands of blocked network calls) without OS-thread-per-connection cost; channels give a way to communicate between them without shared-memory locking discipline for every access. |
| **Why this approach — and an important honest note** | **This repo does not implement a worker pool in the classic sense** (a fixed-size pool of goroutines pulling from a shared work channel within one process). Instead, it achieves worker-pool-like behavior at the **pod level**: each worker pod runs a single-threaded polling loop (`cmd/worker/main.go:133-179`) that processes one Kafka message at a time, sequentially, and KEDA scales the *number of pods* (0-30) based on consumer lag. Concurrency comes from horizontal pod count, not from goroutines within a pod. This is a legitimate and common pattern for CPU-heavy, long-running work (ffmpeg transcodes are CPU/IO-bound for tens of minutes — running N of them concurrently *within* one pod would require N cores and N×memory anyway, so pod-level parallelism maps 1:1 to resource allocation more cleanly than in-process goroutine pools). |
| **Where goroutines *are* used** | (1) The metrics HTTP server runs in a background goroutine (`go func() { metricsServer.ListenAndServe() }()`, `cmd/worker/main.go:114-119`) so it doesn't block the polling loop. (2) The shutdown-signal listener runs in a background goroutine (`cmd/worker/main.go:122-128`) that blocks on `<-shutdownCh` and flips an `atomic.Bool`. (3) `StartQueueDepthPoller` (`pkg/queue_depth.go:30-51`) launches a background goroutine with a `time.Ticker` that polls Kafka every 30s and updates a Prometheus gauge — this *is* a small worker-style background task. (4) The load-test harness's `metricsSampler.Start` (`cmd/load-test/harness.go:731-751`) is the same ticker-goroutine pattern. |
| **Alternatives** | An in-process worker pool (N goroutines reading from a buffered channel of jobs) would let one pod process multiple videos concurrently, better amortizing pod startup/S3-connection-setup cost, at the expense of needing per-job resource isolation (a stuck/leaking ffmpeg process in one goroutine could starve others sharing the pod's CPU/memory limits) and much harder OOM attribution (`kube/worker-deployment.yaml`'s `pulsegrid_pod_resource_constrained` metric currently attributes an OOM to "the pod," which is unambiguous with one job per pod but would become ambiguous with N concurrent jobs per pod). |
| **Tradeoffs** | Pod-per-job-at-a-time is simpler to reason about (one job, one process, one set of resource limits) but has higher fixed overhead per job (pod scheduling latency, TCP/TLS handshake to S3/Kafka/Postgres on every pod start if pods are short-lived) — mitigated here because pods aren't recycled per job, they poll continuously and only scale down on sustained idle (KEDA `cooldownPeriod: 300`, `kube/worker-deployment.yaml:78`). |
| **Common mistakes** | The `atomic.Bool` shutdown flag (`cmd/worker/main.go:106`) is correct precisely *because* it's read/written from two different goroutines (main loop reads, signal-listener goroutine writes) without a mutex — using a plain `bool` here would be a data race (`go test -race` would catch it; `make test-race` target exists in the `Makefile` for exactly this). |
| **Performance implications** | `atomic.Bool.Load()` is a single CPU instruction (no lock contention) — appropriate for a flag checked every polling iteration (up to every ~10s or on every message). |
| **Production considerations** | The metrics server and the polling loop share the process but not a lock — if the polling loop panics unrecovered, the whole pod (including its metrics endpoint) goes down, which is actually *desired* here: Kubernetes' liveness probe on the metrics port (`kube/worker-deployment.yaml:50-56`) then correctly detects the pod as dead and restarts it. |
| **Security considerations** | N/A directly. |
| **Scalability considerations** | This is the crux of PulseGrid's whole scaling story — see [KEDA](#keda--horizontal-worker-autoscaling). Sequential-per-pod processing means throughput is purely `(number of pods) × (1 / avg transcode time)`, which is a very simple, very predictable capacity-planning model — a real advantage over trying to reason about per-pod goroutine-pool saturation. |
| **Failure scenarios** | If the background metrics-server goroutine's `ListenAndServe()` returns a non-`ErrServerClosed` error, it's logged (`logger.Error(...)`) but the goroutine simply exits — the polling loop keeps running with a dead metrics endpoint, meaning Prometheus scraping starts failing but job processing continues silently. This is a real gap: nothing crashes the pod or alerts on this specific failure mode beyond the eventual "target down" Prometheus meta-alert (not present in `kube/prometheus-rules.yaml`). |
| **How this repo implements it** | `sync/atomic.Bool` for the shutdown flag; unbuffered nothing — no channels are used for job dispatch at all (a notable absence for a "worker pool" concept — worth flagging explicitly in an interview since it's easy to *assume* a `chan Job` pattern exists here and it doesn't). |
| **Relevant files** | `cmd/worker/main.go:106-179` (poll loop + shutdown), `pkg/queue_depth.go:30-51` (ticker goroutine) |

### Memory, Escape Analysis & Garbage Collection

| | |
|---|---|
| **What it is** | Go's compiler decides per-variable whether it can live on the stack (freed automatically on function return) or must "escape" to the heap (referenced after the function returns, e.g. via a returned pointer or captured closure) — heap-allocated objects are reclaimed by Go's concurrent, tri-color mark-and-sweep garbage collector. |
| **Why it matters here** | The worker is the memory-sensitive half of this system by a wide margin — `kube/worker-deployment.yaml:43-49` requests 4Gi/limits 8Gi per pod, vs. the API's 1Gi/2Gi, because ffmpeg subprocesses and in-flight video buffers are the dominant memory consumers, not Go heap objects. |
| **Why this approach** | The upload path is explicitly designed to avoid loading whole videos into memory: `handleVideoUpload` streams the multipart file part directly into `s3Uploader.UploadSourceToS3` (`cmd/api/main.go:294`, `io.Reader` passed straight through), and `S3Client.UploadSourceToS3` uses the AWS SDK's `manager.Uploader` with `PartSize: 10MB, Concurrency: 5` (`pkg/s3client.go:43-46`) — meaning at most ~50MB (5 concurrent 10MB parts) is buffered in memory regardless of whether the source file is 50MB or 10GB. `USER_GUIDE.md` states this explicitly ("streamed directly to S3 with no local disk buffering" — actually more precise: no local disk buffering on the *upload* side; the *download* side, `DownloadSourceFromS3`, does write the full file to `/tmp/{jobID}/original.mp4` via `io.Copy`, `pkg/s3client.go:192`, because ffmpeg needs a local file path, not a stream). |
| **Alternatives** | Reading the whole upload into a `[]byte` before uploading — simplest code, catastrophic memory profile for a 10GB file limit (`MaxUploadSize`, `cmd/api/main.go:25`); would require gigabytes of heap per concurrent upload, defeating the point of having `http.MaxBytesReader` as a size guard at all. |
| **Tradeoffs** | Streaming multipart uploads means Go's `ParseMultipartForm(32 << 20)` (`cmd/api/main.go:243`) buffers only the first 32MB of the multipart body in memory; anything beyond that spills to temp files on disk automatically (this is `net/http`'s built-in behavior, not custom code) — meaning even the "streaming" path has an implicit disk-spill fallback the code doesn't explicitly manage. |
| **Common mistakes** | Capturing a loop variable by reference in a closure launched per-iteration is the classic Go escape-analysis/GC-adjacent bug (pre-Go-1.22, all iterations would share one variable) — not present in this codebase's loops, but worth knowing since `go.mod` targets **Go 1.25** (per `USER_GUIDE.md`'s tech stack table; Dockerfiles pin `golang:1.22-alpine` — a real, checkable **version drift** between `go.mod`'s declared minimum and the Docker build image, good interview flag) and Go 1.22+ fixed loop-variable-per-iteration semantics by default, so this class of bug is now much harder to write accidentally regardless. |
| **Performance implications** | ffmpeg subprocesses (`exec.CommandContext`, `pkg/transcode.go:64,142`) run entirely outside the Go heap/GC — their memory (video frame buffers, encoder state) is managed by the OS and ffmpeg's own C allocator, invisible to Go's GC and to `pprof` heap profiles. This means the worker pod's memory limit (`8Gi`) has to be sized empirically against real ffmpeg behavior, not Go's memory model. |
| **Production considerations** | `cmd.CombinedOutput()` (`pkg/transcode.go:65,143`) buffers **all** of stdout+stderr into memory before returning — for a 30-minute ffmpeg run producing verbose progress logging, this buffer could grow to tens of MB; the code caps what it *logs* (`stderr[:500]`, `pkg/transcode.go:69-70`) but does not cap what it *buffers* during collection, meaning `CombinedOutput()` itself is a latent unbounded-memory risk if ffmpeg becomes unusually chatty. |
| **Security considerations** | N/A directly. |
| **Scalability considerations** | Because per-job memory is dominated by the ffmpeg subprocess (not Go heap), horizontal pod scaling (KEDA) is a cleaner lever than tuning `GOGC`/`GOMEMLIMIT` — there's very little Go-GC-tuning surface area in this workload. |
| **Failure scenarios** | Out-of-memory on the worker pod (large video + multiple renditions + HLS segmenting all writing to `/tmp` and holding process memory simultaneously) is exactly the case `ResourceConstraintError`/`ErrorTypeConstraint` exists for (`pkg/errors.go:46-55,115-126`) — worker deliberately calls `os.Exit(1)` rather than retrying in-place (`cmd/worker/main.go:234`), because retrying on the same resource-constrained pod is expected to fail identically; Kubernetes then reschedules onto potentially different node capacity. |
| **How this repo implements it** | Streaming upload via `io.Reader` + multipart-part; buffered download to local disk; ffmpeg subprocess memory is opaque to Go; `defer cleanupTempDir` (`cmd/worker/main.go:319`) ensures `/tmp/{jobID}` disk usage doesn't accumulate across jobs even though it isn't a memory concern per se. |
| **Relevant files** | `pkg/s3client.go:41-52,58-89,128-215`, `pkg/transcode.go:64-77,142-155`, `cmd/worker/main.go:278-297` (cleanup) |

### JSON

| | |
|---|---|
| **What it is** | Go's `encoding/json` reflects over struct tags (`` `json:"field_name,omitempty"` ``) to marshal/unmarshal between Go structs and JSON. |
| **Why this approach** | JSON is the wire format for three independent boundaries in this system, each with different constraints: (1) HTTP request/response bodies (`cmd/api/main.go`'s `ErrorResponse`, `UploadResponse`, `JobStatusResponse`, `ListJobsResponse`); (2) the Kafka message payload (`pkg.KafkaMessage`, `pkg.DLQMessage` — `pkg/kafka.go:25-44`); (3) the `requested_renditions` and `manifest.json` — both JSON stored *inside* Postgres as `JSONB` or written as a file, respectively. Using the same encoding library and largely overlapping struct definitions (`pkg.Job` embeds `Renditions []Rendition` which is reused verbatim in `pkg.KafkaMessage`) keeps the schema consistent across all three boundaries without a schema-definition language (no protobuf/Avro here). |
| **Alternatives** | Protobuf/Avro for the Kafka message — gives you compact binary encoding, schema evolution tooling (backward/forward compatibility checks), and a schema registry story; JSON gives you human-readable messages (debuggable with `kafka-console-consumer` and eyeballs), zero codegen step, and — crucially — a wire format the same Go structs can share with the HTTP layer with zero translation code. For a system at PulseGrid's scale (documented target: tens of jobs/sec at peak, not millions), JSON's CPU/bandwidth overhead vs. protobuf is not the bottleneck; schema evolution safety is the real tradeoff being made away from. |
| **Tradeoffs** | `pkg.KafkaMessage`'s JSON schema has **no version field** — a consumer (worker) deployed with an older struct definition than the producer (API) will silently drop unknown fields on unmarshal (Go's default behavior) rather than fail loud. This is fine for additive changes but silently *breaks* on renamed/removed fields with no compile-time or runtime signal, which is exactly the schema-evolution safety protobuf/Avro would buy back. |
| **Common mistakes** | Forgetting `omitempty` on optional fields causes them to serialize as JSON `null`/zero-value instead of being omitted — `pkg.Job.FailureReason` and `pkg.Job.OutputFiles` correctly use `omitempty` (`pkg/models.go:67-68`); `pkg.Rendition`'s HLS-only fields (`Type`, `SegmentDuration`, `BaseResolution`) also correctly use `omitempty` so non-HLS renditions don't carry spurious zero-value fields in the wire payload (`pkg/models.go:40-42`). |
| **Performance implications** | `json.Marshal`/`Unmarshal` use reflection — noticeably slower than codegen'd (un)marshalers (`easyjson`, protobuf) at very high throughput, but for job-submission-rate workloads (not per-video-frame), this is not a measurable bottleneck. `GenerateManifest` explicitly round-trips the manifest through `json.Unmarshal` immediately after marshaling it (`pkg/manifest.go:93-97`, "Validate JSON by unmarshalling back") purely as a correctness check, at the cost of doing the encode work twice — a deliberate belt-and-suspenders tradeoff for a file that's the durable record of what a job produced. |
| **Production considerations** | `JobFilter`/`QueryJobs` marshal/unmarshal `requested_renditions` as `JSONB` in Postgres (`pkg/postgres.go:107-109,173-175`) — this means the DB cannot efficiently query/filter *by* rendition contents (e.g., "find all jobs that requested HLS") without a JSONB containment query or a separate normalized table; the schema as built optimizes for write-simplicity over query flexibility on that column. |
| **Security considerations** | `json.Unmarshal` into a fixed struct type (not `map[string]interface{}` or `interface{}`) is itself a mild security good-practice — it rejects/ignores unexpected fields rather than deserializing arbitrary attacker-controlled structure, limiting JSON-based injection surface. `parseRenditions` (`cmd/api/main.go:569-591`) does validate renditions JSON structurally (non-empty array, each entry has `id` or `type`) before use — reasonable input validation at the JSON boundary. |
| **Scalability considerations** | N/A beyond throughput above. |
| **Failure scenarios** | A worker receiving a malformed Kafka message (`json.Unmarshal` fails, `cmd/worker/main.go:157-168`) does **not** crash or retry — it logs the error and explicitly commits the offset to skip the poison message ("Commit to skip malformed message," `cmd/worker/main.go:163-166`). This is a deliberate, correct choice: a malformed message will *never* successfully unmarshal no matter how many times it's retried, so treating it as retryable would create an infinite reprocessing loop; skipping it is effectively an implicit permanent-failure classification that bypasses `ClassifyError` entirely (worth noting as an inconsistency — this failure mode never reaches the DLQ, unlike every other permanent failure in the system, meaning there's no audit trail for messages dropped this way beyond the log line). |
| **How this repo implements it** | Standard `encoding/json` throughout, no third-party JSON library. Struct tags define wire names; `omitempty` used consistently for optional fields. |
| **Relevant files** | `pkg/models.go`, `pkg/kafka.go:25-44`, `pkg/manifest.go:22-30,87-97`, `cmd/api/main.go:39-54,382-388,505-514,569-591` |

### HTTP Server & Middleware

| | |
|---|---|
| **What it is** | The API is built entirely on `net/http`'s standard library `http.ServeMux` — no Gin/Echo/Chi/Fiber. |
| **Why this approach** | Five routes (`/videos/upload`, `/jobs/`, `/jobs`, `/health`, `/metrics`), no path-parameter routing needs beyond a single `{job_id}` segment (parsed manually via `strings.TrimPrefix(r.URL.Path, "/jobs/")`, `cmd/api/main.go:523`), and no complex middleware chain — this is genuinely within the sweet spot where `net/http` alone is the right call, not a missed opportunity for a framework. `USER_GUIDE.md`'s tech stack table states this explicitly: "Go standard library `net/http` (no external web framework)." |
| **Alternatives** | Chi/Gin — buy you named path parameters (`/jobs/{job_id}` instead of manual prefix-trimming), a formal middleware chain (auth, logging, recovery, CORS composable as `Use()` calls), and often better default panic recovery. Worth the dependency once you have >~10 routes, nested resource paths, or need per-route middleware composition — none of which apply here yet. |
| **Tradeoffs — the big one: no middleware at all** | **There is no middleware in this codebase.** No panic-recovery middleware (an unrecovered panic in any handler crashes the whole `http.ListenAndServe` goroutine — actually, in Go's `net/http`, a panicking handler is recovered *per-connection* by the server automatically, logging the panic and closing that connection, so it does *not* crash the whole process — but there's no custom logging/observability around that recovery, so a panic is invisible in the structured logs/metrics, only in stderr). No request-logging middleware (there is no access log at all — no line printed per HTTP request, which is a real production gap: you cannot reconstruct request volume, latency distribution, or client IPs from logs, only from the three Prometheus metrics that exist). No auth middleware (see [Security](#authentication--authorization)). No CORS handling (fine for a server-to-server internal API, a gap if browsers ever call it directly). |
| **Common mistakes** | Writing to `http.ResponseWriter` after `WriteHeader` — not present here (each handler writes headers, then body, in the correct single-writer order); calling `w.WriteHeader` twice — checked, not present. A different mistake that *is* present: `handleVideoUpload` calls `pkg.GenerateJobID()` **twice** per request — once for `requestID` (`cmd/api/main.go:233`) and again for `jobID` (`cmd/api/main.go:283`) — two independent UUIDs for what could arguably be the same identifier, doubling `crypto/rand` reads for no functional benefit (minor, but a real "why two IDs" question worth being ready for). |
| **Performance implications** | `http.ServeMux`'s routing is a simple longest-prefix-match trie lookup — O(path segments), negligible overhead vs. any router library at this route count. The real perf-relevant HTTP-layer decision is `http.MaxBytesReader(w, r.Body, MaxUploadSize)` (`cmd/api/main.go:240`), which enforces the 10GB cap by erroring the read *as it happens* rather than buffering unboundedly first — this is the single most important line for protecting the API tier's memory from a malicious/broken client claiming an enormous `Content-Length`. |
| **Production considerations** | No configured server timeouts (`http.Server{}` uses `http.ListenAndServe(addr, mux)` directly with zero `ReadTimeout`/`WriteTimeout`/`IdleTimeout`, `cmd/api/main.go:142`) — this is a real production risk: a slow-loris-style client (opens connection, trickles bytes) can hold a connection (and its `MaxUploadSize`-bounded but still potentially large buffer) open indefinitely, exhausting file descriptors / connection slots under sustained attack or even just flaky client networks. The worker's metrics server (`cmd/worker/main.go:113`) has the same gap. |
| **Security considerations** | See [Input Validation](#input-validation) — validation is thorough within handlers, but the *transport* layer (no timeouts, no rate limiting, no request-size-independent of `MaxBytesReader`) is unhardened. |
| **Scalability considerations** | Stateless handlers + `http.ServeMux` mean the API tier scales horizontally trivially (2 replicas today, `kube/api-deployment.yaml:32`) — no session affinity, no in-memory state to synchronize. |
| **Failure scenarios** | A panic in `handleListJobs` (e.g., a nil-pointer if `dbClient.QueryJobs` returned a malformed result) would be caught by `net/http`'s per-request panic recovery, close that one connection with a logged stack trace to stderr, and the server keeps running — degraded observability (no metric/structured log for the panic) but not a full outage. |
| **How this repo implements it** | `http.NewServeMux()` + `mux.HandleFunc(...)` for each route, `promhttp.Handler()` mounted at `/metrics` (`cmd/api/main.go:133-138`). No middleware chain of any kind. |
| **Relevant files** | `cmd/api/main.go:133-145`, `cmd/worker/main.go:111-119` |

### Configuration

| | |
|---|---|
| **What it is** | 100% environment-variable driven — no config file, no flags (except `cmd/load-test`, which is a CLI tool and correctly uses `flag.NewFlagSet`, `cmd/load-test/harness.go:123`). |
| **Why this approach** | This is the [Twelve-Factor App](https://12factor.net/config) config pattern — config lives in the environment, not in code or a checked-in file, so the same container image runs unmodified across dev/staging/prod by varying only its env vars (`kube/configmap.yaml` + `kube/secrets.yaml` supply these in Kubernetes). |
| **Alternatives** | A config file (YAML/TOML) + a struct-binding library (Viper) — better for deeply nested config or config that needs hot-reload; adds a dependency and a file-mount concern in Kubernetes (ConfigMap-as-volume vs. ConfigMap-as-env, `envFrom: configMapRef` is simpler, which is what's used, `kube/api-deployment.yaml:54-57`). |
| **Tradeoffs** | Every config value is optional and silently falls back to a default or a "not configured" mode (`if os.Getenv("DATABASE_URL") != ""`) — this maximizes local-dev friendliness at the cost of making misconfiguration silent rather than a fail-fast startup error. A typo'd `KAFKA_BROKER` (missing the `S`) in a real deployment would not error — it would just leave `kafkaProducer` nil and jobs would stop being enqueued, discoverable only via `GET /health`'s `kafka: not_configured` status or the complete absence of `pulsegrid_queue_depth_jobs` metric activity. |
| **Common mistakes** | Two different env var naming schemes exist for the *same* concepts between the two binaries: API uses `KAFKA_TOPIC`/`KAFKA_DLQ_TOPIC`/`KAFKA_CONSUMER_GROUP` (`cmd/api/main.go:95,99,119`); worker uses `JOB_TOPIC`/`DLQ_TOPIC`/`CONSUMER_GROUP` (`cmd/worker/main.go:50-52`). `kube/configmap.yaml` has to define **both** sets of names pointing at the same values (lines 11-13 and 20-22) purely to satisfy both binaries — this is real, checkable config duplication risk: updating the topic name requires editing two ConfigMap keys in lockstep, and nothing enforces they stay equal. |
| **Performance implications** | N/A — config is read once at startup (`os.Getenv` calls happen only in `main()`, not per-request). |
| **Production considerations** | Output bucket name (`pulsegrid-output`) and Kafka `RequiredAcks`/`MaxAttempts` are hardcoded constants (`pkg/s3client.go:24`, `pkg/kafka.go:60-61`), not environment-configurable — `USER_GUIDE.md` calls this out explicitly. This means changing the output bucket name requires a code change + redeploy, not a config change — an intentional scope-limiting choice (fewer knobs = fewer misconfiguration modes) but a real operational constraint if you ever need per-environment output buckets (dev/staging/prod sharing one hardcoded name would be a real bug). |
| **Security considerations** | Secrets (`DATABASE_URL` with embedded password, AWS keys) flow through the same `envFrom: secretRef` mechanism as non-secret config (`kube/*-deployment.yaml`) — see [Secrets](#secrets--credential-management) for why this specific delivery mechanism matters. |
| **Scalability considerations** | N/A. |
| **Failure scenarios** | Missing `DATABASE_URL` at worker deploy time isn't actually possible to misconfigure into a *worse* state than intended, because the worker **never reads `DATABASE_URL` at all** — it has no DB client. This is the config-level fingerprint of the "worker never writes status back" gap: if you go looking for why, `cmd/worker/main.go`'s env var list (`brokers`, `topic`, `dlqTopic`, `groupID`, `awsRegion`) conspicuously has no `DATABASE_URL`. |
| **How this repo implements it** | `os.Getenv`/a small `getEnv(key, fallback)` helper (`cmd/worker/main.go:398-403`) throughout; defaults hardcoded per call site. |
| **Relevant files** | `cmd/api/main.go:56-131`, `cmd/worker/main.go:43-91`, `kube/configmap.yaml`, `kube/secrets.yaml` |

### Testing, Property Testing & Benchmarking

| | |
|---|---|
| **What it is** | Go's built-in `testing` package for unit/table-driven tests; `testing/quick` for a small subset of true property-based (randomized-input) tests; no `testing.B` benchmarks exist anywhere in the repo. |
| **Why this approach** | Table-driven tests are the idiomatic Go default — a `[]struct{name string; input X; want Y}` slice iterated with `t.Run(tt.name, func(t *testing.T) {...})` gives readable, independently-reportable subtests without a third-party framework. This repo uses that pattern extensively (e.g., `pkg/errors_test.go`'s 15 `TestClassifyError_*` cases per `implementation-audit.md`). |
| **What "property testing" means here vs. what's implemented** | A genuine property-based test generates *random* inputs (e.g., `quick.Check` calling your function hundreds of times with generated values) and asserts an invariant holds for all of them — as opposed to an example-based test asserting one specific input produces one specific output. `implementation-audit.md` is unusually candid about this: the original task spec (`.kiro/specs/pulsegrid/tasks.md`) called for `testing/quick`/`gopter`-style property tests on ~8 different subsystems (manifest schema, retry count increment, DLQ entry, queue schema, temp cleanup, error-logging context), but **only three files actually use `testing/quick`**: `pkg/postgres_test.go`, `pkg/kafka_test.go`, `pkg/jobid_test.go` (confirmed via repo grep). The rest are implemented as thorough example-based unit tests that *cover the same logic* but don't generate randomized inputs — a real, honestly-documented gap between spec and implementation, and a good discussion point on the difference between "well-tested" and "tested exactly as specified." |
| **Alternatives to `testing/quick`** | `gopter` (more powerful generators, shrinking on failure) — heavier dependency, steeper API, not used here. `testing/quick` is stdlib-only, has no shrinking (a failing case isn't automatically minimized to the smallest reproducing input, unlike `gopter` or QuickCheck-proper), and is arguably under-maintained in the Go ecosystem — a fair critique of the choice, but consistent with this repo's general "minimize dependencies" posture. |
| **Tradeoffs** | Property tests catch edge cases example-based tests don't think to write (e.g., a UUID generator property-tested for "always 36 characters, always valid hyphen positions, always version-4 bit pattern" across thousands of random calls, vs. one hardcoded example) — but they're harder to write meaningfully for I/O-heavy code (there's no natural "random Postgres connection string" generator that's useful), which is exactly why this repo's actual `testing/quick` usage clusters around pure, deterministic functions (`GenerateJobID`, message schema round-trips) rather than the DB/Kafka integration-shaped tests. |
| **Common mistakes** | Table-driven tests without `t.Run` per case lose the ability to see *which* case failed at a glance and can't be run individually via `go test -run`; this repo consistently uses `t.Run`. A more subtle mistake avoided here: mutating shared state across subtests without `t.Parallel()`-safe isolation — the mock structs in `test/integration/full_lifecycle_test.go` all use `sync.Mutex`-guarded fields specifically because they *could* be accessed concurrently even though the current tests don't call `t.Parallel()`. |
| **Performance implications / benchmarking** | **No `func Benchmark*(b *testing.B)` exists anywhere in this repo.** For a system whose core cost driver is ffmpeg transcode duration (external process, not benchmarkable via Go's `testing.B` model) and S3 upload/download throughput (network-bound, also a poor `testing.B` fit), this is a defensible omission — but pure-Go hot paths like `pkg.RetryWithBackoff`'s delay computation, `pkg.ClassifyError`'s string matching, or `db.RunMigrations`'s file-sorting logic *would* be legitimate benchmark targets that simply aren't covered. Worth naming directly if asked "what's missing from the test suite." |
| **Production considerations** | `make test-race` (`Makefile:62-63`) runs the full suite with Go's race detector (`-race`) — this is the primary defense against the goroutine/atomic-flag concurrency patterns in the worker being subtly wrong; CI (`.github/workflows/ci.yml:30-31`) runs `go test ./... -v -race -count=1` on every push/PR, so race conditions are caught before merge, not just available as an opt-in local target. |
| **Security considerations** | N/A directly, though thorough input-validation unit tests (`cmd/api/main_test.go`, per audit table's "2.1 Unit tests: HTTP validation ✅") are a meaningful defense-in-depth layer against malformed-request handling regressions. |
| **Scalability considerations** | N/A. |
| **Failure scenarios** | N/A — this section is about verifying failure scenarios, not causing them. |
| **How this repo implements it** | Standard `testing` package, `t.Run` table-driven pattern throughout, `testing/quick` in exactly 3 files, `httptest` for HTTP-layer tests, in-process mocks (not `testcontainers`) for external dependencies everywhere. See [Part 8](#part-8--testing-strategy) for the full breakdown by test type. |

### Structured Logging & Graceful Shutdown

| | |
|---|---|
| **What it is** | `go.uber.org/zap` production JSON logger, wrapped in a thin `pkg.Logger` type that injects `pod_id` and `event_type` on every call; graceful shutdown via signal handling + an atomic flag + a bounded `terminationGracePeriodSeconds`. |
| **Why this approach** | Zap is chosen for near-zero-allocation structured logging (`zapcore.NewJSONEncoder`) — appropriate for a worker that may log dozens of structured events per job, at scale across up to 30 pods. Every log call is JSON (`pkg/logger.go:40-44`), which is what lets a downstream log aggregator (not present in this repo's manifests — no Fluentd/Loki/CloudWatch-agent sidecar defined, meaning logs currently only reach `kubectl logs`/stdout capture) parse fields like `job_id`, `error_type`, `retry_count` without regex scraping. |
| **Why graceful shutdown this way** | `signal.Notify(shutdownCh, syscall.SIGTERM, syscall.SIGINT)` (`cmd/worker/main.go:107-108`) + a background goroutine that flips `shutdownRequested.Store(true)` on receipt (`cmd/worker/main.go:122-128`), checked at the top of every poll-loop iteration (`cmd/worker/main.go:134-137`) and again after a `FetchMessage` timeout (`cmd/worker/main.go:148-151`). This is the standard Go idiom for graceful shutdown: never call `os.Exit` directly from a signal handler; instead, set a flag/close a channel and let the main loop unwind naturally, so in-flight work (`processJob`) finishes before the loop exits and cleanup (`metricsServer.Shutdown`, `reader.Close()`) runs. |
| **Alternatives** | `context.WithCancel` derived from signal context (`signal.NotifyContext`, Go 1.16+) is the more modern idiom — would let `processJob`'s inner context also observe cancellation directly rather than needing a separate polled atomic flag; this repo's approach (atomic flag, `context.Background()` for the actual job work) is a deliberate choice to make in-flight jobs shutdown-*immune* (see [Context](#context-cancellation--timeouts)), which `signal.NotifyContext` would make shutdown-*aware* by default — the repo's choice is arguably safer for this specific workload (never abandon a 25-minutes-in transcode because a rolling deploy started), at the cost of the up-to-10-second detection latency during idle polling. |
| **Tradeoffs** | The current design cannot cancel an in-flight ffmpeg transcode on shutdown even if you wanted to (no cancellable context reaches `TranscodeSingleRendition`/`TranscodeHLS` from the signal path) — the *only* thing bounding shutdown time is `DefaultTranscodeTimeout` (30 min) and `terminationGracePeriodSeconds` (30 min), which not coincidentally match. |
| **Common mistakes** | Logging secrets/PII in structured fields — checked, not present (job IDs, S3 URIs, error messages, but no credentials logged). A real, present gap: `LogTranscodeFailure` truncates `ffmpeg_stderr` to 500 chars (`pkg/logger.go:109-112`) consistently with the same truncation in `pkg/transcode.go:69-70,147-148` — good defensive consistency, but there's no test verifying this truncation happens in the *logger* specifically (`implementation-audit.md` Task 20.1: "No test file exists for `pkg/logger.go` at all"). |
| **Performance implications** | Zap's structured-field API (`zap.String(...)`, `zap.Int(...)`) avoids `fmt.Sprintf`-style string formatting until (and unless) the log line is actually encoded — meaningfully faster than `log.Printf`-style logging at high volume, though the API's `defer logger.Sync()` (`cmd/worker/main.go:46`) is important and correctly present: `Sync()` flushes buffered log entries, and skipping it risks losing the last few log lines on process exit. |
| **Production considerations** | `podID := logger.PodID()` (`cmd/worker/main.go:53`) reads `HOSTNAME` (Kubernetes injects the pod name here automatically) — this is what makes `pod_id` in every log line traceable back to a specific pod for debugging a specific worker's behavior, and it's the same mechanism `GenerateManifest` uses for `WorkerPodID` (`pkg/manifest.go:36`), giving a consistent identity thread from logs → manifest → (eventually, if wired) DLQ `pod_id` field. |
| **Security considerations** | N/A beyond the "don't log secrets" point above, which holds. |
| **Scalability considerations** | Structured JSON logs are what make log-based alerting/dashboards *possible* to build even without the metrics gap noted elsewhere — but only if something actually ships these logs off the pod (currently nothing does; this is worth naming as a real observability gap alongside the missing access log). |
| **Failure scenarios** | If SIGTERM arrives while `processJob` is deep inside `TranscodeHLS`, the poll loop doesn't even notice until that call returns — `shutdownRequested.Load()` isn't checked mid-`processJob`, only between iterations (`cmd/worker/main.go:134` and `148-151`). This is consistent with the "never abandon in-flight work" design intent but means a *second* SIGTERM (or SIGKILL after grace period expiry) is the only way to force-stop a genuinely hung job. |
| **How this repo implements it** | `pkg.NewLogger()` (`pkg/logger.go:19-52`) wraps `*zap.Logger`; `Info`/`Error`/`LogJobEvent`/`LogJobError`/`LogTranscodeFailure` are the five logging entry points, each injecting consistent context fields. Graceful shutdown: atomic flag + signal goroutine + bounded poll timeout + explicit `metricsServer.Shutdown`/`reader.Close()` on exit (`cmd/worker/main.go:181-194`). |
| **Relevant files** | `pkg/logger.go`, `cmd/worker/main.go:43-53,105-137,181-195` |

## Part 2 — Distributed Systems

### Message Queues & Why Kafka

| | |
|---|---|
| **What it is** | A durable, partitioned, append-only log that decouples producers (API) from consumers (workers) in time and rate. |
| **Why it exists** | Without a queue, the API would have to call the worker synchronously (HTTP RPC) and block for the full transcode duration (minutes), or spawn transcoding in-process (coupling API scaling to transcode load, defeating the whole point of independent tiers). A queue lets the API return `202 Accepted` in milliseconds and lets worker capacity scale independently of submission rate. |
| **Why Kafka specifically** | Kafka gives three properties this system leans on: (1) **consumer groups with per-partition offset tracking** — exactly the mechanism KEDA uses to measure "lag" for autoscaling (`kube/worker-deployment.yaml`'s `ScaledObject`); (2) **replay** — a consumer group can be reset to re-process history, useful for a DLQ-recovery workflow or backfilling; (3) **ordering within a partition** — `Balancer: &kafka.Hash{}` (`pkg/kafka.go:59`) partitions by `job_id`, guaranteeing all messages for one job (including retries) land on the same partition and are processed in order relative to each other. |
| **Alternatives** | **SQS** — simpler ops (fully managed, no broker/ZooKeeper/KRaft to run), built-in DLQ as a first-class resource, but no consumer-group-lag concept (KEDA's SQS scaler uses queue depth, a coarser signal than per-consumer-group lag) and no partition-level ordering guarantee (SQS FIFO queues exist but cap throughput and complicate the "3-broker cluster" story entirely away). **RabbitMQ** — strong routing/exchange model, weaker at replay/ordering-at-scale than Kafka. `aws-architecture-review.md` explicitly recommends SQS as a cost/ops simplification for this workload's actual usage pattern (single queue, no fan-out) — a legitimate "we may be over-engineered here" critique worth being able to articulate both sides of. |
| **Tradeoffs** | Self-hosted Kafka (3-broker StatefulSet in `kube/`, not shown in this guide's file list but referenced by `USER_GUIDE.md` and `aws-architecture-review.md`) means PulseGrid owns broker upgrades, disk provisioning, ZooKeeper/KRaft operations, and rebalancing — real operational weight for a single-topic, single-consumer-group use case. `aws-architecture-review.md`'s estimate: this is one of the three biggest cost/complexity drivers not buying capability actually used. |
| **Common mistakes** | Using a random/round-robin partitioner instead of a key-based one for a workload requiring per-entity ordering (retries out of order with the original message) — avoided here via `Balancer: &kafka.Hash{}` keyed on `job_id` (`pkg/kafka.go:56-59,101-104`). Setting `RequiredAcks` too low (`kafka.RequireOne` or `RequireNone`) trades durability for latency — this repo uses `RequireAll` (`pkg/kafka.go:60,68`), meaning a publish only succeeds once all in-sync replicas have the message, the safest but slowest option. |
| **Performance implications** | `MaxAttempts: 1` on both writers (`pkg/kafka.go:61,69`) is a deliberate, easy-to-miss design choice: kafka-go's own internal retry is disabled, and *all* retry logic is delegated to `pkg.RetryWithBackoff` (5 attempts, 1s base, 16s cap) wrapping every publish call. This avoids double-retry-logic (kafka-go retrying *and* the app retrying, compounding backoff unpredictably) at the cost of needing to get the app-level retry parameters right for Kafka-specific failure modes (e.g., a leader election taking longer than the backoff schedule allows). |
| **Production considerations** | `SessionTimeout: 30 * time.Minute` (`cmd/worker/main.go:99`) is unusually long for a Kafka consumer (defaults are typically 10-45 *seconds*) — necessary because a consumer that doesn't heartbeat during a 25-minute ffmpeg call would otherwise be evicted from the group mid-job by a normal timeout. This is the correct fix for "long-running message processing" in Kafka, but it means a genuinely dead consumer (crashed, not just busy) is invisible to the group for up to 30 minutes. |
| **Security considerations** | No TLS/SASL configuration visible in `pkg/kafka.go`'s `kafka.TCP(brokers...)` — plaintext broker connections. Acceptable only because Kafka runs inside the same VPC/cluster as the API/worker pods (no cross-network hop), a common but real internal-trust-boundary assumption worth naming explicitly. |
| **Scalability considerations** | Partition count (not shown in these files — provisioned by whatever creates the `transcoding-jobs` topic, likely the Kafka StatefulSet's init or manual `kafka-topics` command, not present in the reviewed file set) is the hard ceiling on *consumer* parallelism — you cannot usefully run more active consumers in one group than there are partitions; KEDA can scale worker pods to 30, but if the topic has, say, 6 partitions, only 6 of those pods will ever be actively consuming, the rest idle. This is a real, checkable capacity-planning question to ask about a system you're onboarding onto. |
| **Failure scenarios** | Broker unreachable during publish → `RetryWithBackoff` retries 5x over up to ~31s total (1+2+4+8+16) → if still failing, `EnqueueJob` returns an error → `handleVideoUpload` rolls back the DB transaction (`txHandle.Rollback`, `cmd/api/main.go:337`) → client receives a `500` and the job never existed from their point of view. This is the **orphan-prevention** mechanism — see [Idempotency](#idempotency) below. |
| **How this repo implements it** | Two parallel implementations exist: `pkg.KafkaClient` (used by both `cmd/api` and `cmd/worker` today) and `pkg/queue.KafkaQueue` (a cleaner `MessageQueue` interface, seemingly unused by either binary — worth asking "why does this exist and is it dead code" in an interview). |
| **Relevant files** | `pkg/kafka.go`, `pkg/queue/kafka.go`, `cmd/worker/main.go:93-103`, `kube/worker-deployment.yaml` (KEDA trigger), `kube/configmap.yaml` |

### Consumer Groups & Offsets

| | |
|---|---|
| **What it is** | A consumer group is a named set of consumers that Kafka load-balances partitions across; an offset is the position (integer index) within a partition that a group has confirmed processing up to. |
| **Why this approach** | `GroupID: groupID` (default `pulsegrid-workers`, `cmd/worker/main.go:97`) means every worker pod is a member of the *same* group — Kafka automatically assigns each active pod a disjoint subset of partitions, so adding pods (KEDA scale-up) increases parallelism without any application-level coordination code. This is the entire mechanism by which "0-30 pods" turns into "0-30x throughput" (bounded by partition count, see above). |
| **Why commit-after-success, not commit-after-receive** | `reader.CommitMessages(ctx, msg)` is called **only** inside `handleJobOutcome`, after `processJob` has either fully succeeded, been re-enqueued, or been sent to the DLQ (`cmd/worker/main.go:201,245,257(no-commit on constraint!),272`) — never immediately upon `FetchMessage`. This is what gives the system its at-least-once (not at-most-once) delivery guarantee: if the pod crashes between fetch and commit, the message is redelivered to another consumer on rebalance. |
| **The one deliberate exception** | On `ErrorTypeConstraint` (resource exhaustion), the code calls `os.Exit(1)` **without committing** (`cmd/worker/main.go:230-234`) — this is intentional: the message must be redelivered (to a hopefully-healthier pod/node) rather than treated as handled, since a resource-constrained failure says nothing about whether the *job* is processable, only that *this pod* currently can't process it. |
| **Alternatives** | Auto-commit (kafka-go and most clients support periodic auto-commit on a timer, independent of processing outcome) — much simpler code, but breaks the delivery guarantee: a message could be marked "consumed" before processing actually completed, silently dropping work on a crash. Manual commit-on-success (what this repo does) is strictly more correct for this workload and is the standard pattern for "processing has real side effects that must complete before acknowledging." |
| **Tradeoffs** | Manual commit means a slow consumer (mid-transcode) holds its offset uncommitted for the full job duration — during that window, if the *broker* considers the consumer's session dead (session timeout), it will reassign the partition and redeliver the same message to another consumer, causing duplicate processing (see [Exactly-once vs. at-least-once](#exactly-once-vs-at-least-once-vs-ordering)). |
| **Common mistakes** | Committing before processing (defeats the guarantee) — not present here. Committing the wrong message/offset in a batch — N/A, this reader processes one message per `FetchMessage`/`CommitMessages` pair, no batching. |
| **Performance implications** | `HeartbeatInterval: 3 * time.Second` (`cmd/worker/main.go:100`) keeps the consumer's group membership alive independently of message processing rate — kafka-go sends heartbeats on a background timer regardless of whether `FetchMessage` is blocked, so a long-running `processJob` call doesn't itself starve heartbeats (this is a client-library-internal detail, not app code, but important to know it's *not* a risk here). |
| **Production considerations** | The queue-depth poller (`pkg/queue_depth.go`) independently computes `end_offset - committed_offset` per partition every 30s to feed `pulsegrid_queue_depth_jobs` — this is functionally the same lag calculation KEDA's Kafka scaler does internally, duplicated here purely for the API tier's own Prometheus visibility (KEDA doesn't expose its internal lag calculation as an app-visible metric). |
| **Security considerations** | N/A beyond the general Kafka trust-boundary note above. |
| **Scalability considerations** | Consumer count > partition count means idle consumers (a partition can only be owned by one consumer in a group at a time) — the practical ceiling on useful KEDA scale-up. |
| **Failure scenarios** | Rebalance mid-processing: pod A is transcoding job X (offset 47, uncommitted), pod A's session times out (network partition, GC pause exceeding session timeout, etc.), Kafka reassigns partition to pod B, pod B fetches from the last **committed** offset (46, not 47) and reprocesses job X's message from scratch — pod A, unaware of the rebalance, may *also* still be finishing job X and eventually calls `CommitMessages` on an offset it no longer owns (this either no-ops or errors depending on kafka-go's behavior; not explicitly handled/tested in this codebase). This is the mechanical root cause of why the system is at-least-once, not exactly-once, and why [Idempotency](#idempotency) at the S3-write layer matters. |
| **How this repo implements it** | `kafka.NewReader` with `GroupID` set (`cmd/worker/main.go:94-102`); explicit `CommitMessages` calls gated on outcome in `handleJobOutcome`. |
| **Relevant files** | `cmd/worker/main.go:94-103,198-276`, `pkg/queue_depth.go` |

### Retries & Exponential Backoff

| | |
|---|---|
| **What it is** | `pkg.RetryWithBackoff(ctx, maxAttempts, baseDelay, fn)` — retries `fn` up to `maxAttempts` times, sleeping `baseDelay * 2^attempt` (capped at 16s) between attempts, respecting context cancellation. |
| **Why it exists** | Distributed systems fail transiently constantly (network blips, broker leader elections, S3 throttling, DB connection pool exhaustion) — retrying immediately in a tight loop would amplify load on an already-struggling downstream (retry storm); exponential backoff spaces out attempts so transient failures have time to resolve before the next attempt, and permanent failures fail fast-ish rather than looping forever. |
| **Why this specific implementation** | A single generic helper (`pkg/retry.go`) used identically for S3 uploads/downloads, Kafka publishes, and Postgres connection establishment — one retry policy, one place to reason about it, rather than bespoke retry loops duplicated per client. Every caller supplies its own `maxAttempts`/`baseDelay` (all currently use 5 attempts / 1s base in this codebase, but the function doesn't hardcode that — it's a parameter, `pkg/retry.go:12`). |
| **Alternatives** | A retry *library* (e.g. `cenkalti/backoff`, `avast/retry-go`) — adds jitter (randomized delay to avoid thundering-herd synchronization across many clients retrying in lockstep), circuit-breaking, and more configurable backoff curves (this repo's is a fixed doubling with a hard cap, no jitter at all — a real, checkable gap: **thundering herd** across many simultaneously-failing worker pods retrying an S3 outage in near-lockstep is a real risk this implementation doesn't guard against). |
| **Tradeoffs** | No jitter + identical backoff parameters across all worker pods means a transient S3/Kafka blip affecting *all* pods simultaneously (e.g., an AZ networking event) produces near-synchronized retry waves at the 1s/2s/4s/8s/16s marks across potentially 30 pods — worse for the recovering downstream than randomized jitter would be. |
| **Common mistakes** | Retrying non-idempotent operations — this repo is careful here: `EnqueueJob`/`ReenqueueWithRetry` retries are safe because Kafka publish is naturally at-least-once anyway (a duplicate publish just means a duplicate message, handled by the broader at-least-once design, not a correctness violation); S3 `PutObject` retries are safe because S3 PUT is idempotent by nature (same key, same content, overwritten harmlessly). Retrying a *permanent* failure (retrying a 403 Forbidden 5 times before giving up) wastes ~31 seconds per request — this repo explicitly avoids that via `isPermanentS3Error` short-circuiting (`pkg/s3client.go:217-228,251-257`) and the `permanentErr`-then-`return nil`-to-stop-retrying pattern inside the retry closure itself (`pkg/s3client.go:160-179`, `SourceNotFoundError` case) — a clever but slightly unusual idiom worth understanding: the closure returns `nil` (success, from `RetryWithBackoff`'s point of view) while setting an outer `permanentErr` variable, deliberately short-circuiting the retry loop without making `RetryWithBackoff` itself permanent-error-aware. |
| **Performance implications** | Worst case, a single S3 call that's genuinely down costs `1+2+4+8+16 = 31` seconds before giving up (5 attempts) — this directly adds to job processing latency and, transitively, to how long a Kafka consumer holds a partition without committing (interacting with the 30-minute session timeout discussed above). |
| **Production considerations** | The **retry_count on the Kafka message** (`kafkaMsg.RetryCount`, incremented in `handleJobOutcome`, `cmd/worker/main.go:262`) is a *different* retry concept from `RetryWithBackoff`'s in-process attempts — this is job-level retry (re-enqueue the whole job for a fresh worker pickup after a transcode failure), capped at `maxRetries = 3` (`cmd/worker/main.go:41`) before DLQ. Two independent retry mechanisms exist at two different layers (I/O-call-level via `RetryWithBackoff`, job-level via Kafka re-enqueue) — a good interview question is being able to clearly separate these two and explain why both are needed (transient I/O blips shouldn't burn a job-level retry; a job that fails after exhausting I/O retries genuinely needs a fresh attempt, possibly on different infrastructure). |
| **Security considerations** | N/A directly, though unbounded retry against an auth failure (if `isPermanentS3Error`/`ClassifyError`'s permanent-pattern list ever missed a case) could look like a low-grade credential-stuffing-style hammering pattern against S3/Kafka from a monitoring perspective. |
| **Scalability considerations** | See thundering-herd note above — becomes a real concern only at higher worker-pod counts (closer to 30 than to 1). |
| **Failure scenarios** | `ctx.Done()` during a backoff sleep returns immediately with a wrapped cancellation error (`pkg/retry.go:32-36`) rather than waiting out the full delay — correct responsiveness to shutdown/timeout. |
| **How this repo implements it** | `pkg/retry.go`'s single `RetryWithBackoff` function, called from `pkg/kafka.go` (4 call sites), `pkg/s3client.go` (3 call sites), `pkg/postgres.go` (1 call site, for initial connection). |
| **Relevant files** | `pkg/retry.go`, `pkg/retry_test.go`, `pkg/s3client.go:69-82,153-201,235-259`, `pkg/kafka.go:82-112,142-154,164-176,196-208` |

### Dead-Letter Queue & Idempotency

| | |
|---|---|
| **What it is** | DLQ: a separate Kafka topic (`transcoding-dlq`) that receives messages the system has given up retrying, preserving them for manual inspection/replay rather than silently dropping them. Idempotency: the property that processing the same message twice produces the same end state as processing it once. |
| **Why DLQ exists** | Without it, a permanently-failing message (corrupted video, unsupported codec) would either loop forever (if naively retried) or be silently dropped (if simply discarded on failure) — neither preserves the failure for debugging or lets an operator manually fix-and-replay. |
| **Why this approach** | Two paths lead to DLQ (`cmd/worker/main.go:236-247,251-259`): immediate DLQ for `ErrorTypePermanent` (no point retrying a corrupted file), and DLQ-after-exhaustion for `ErrorTypeRetryable` once `retry_count >= maxRetries` (3). Both call `SendDLQFromMessage`, which wraps the original `KafkaMessage` with failure metadata (`DLQMessage` embeds `KafkaMessage` + adds `FailureReason`, `FailureTimestamp`, `PodID`, `DLQEntryTimestamp` — `pkg/kafka.go:38-44`) — the full original job spec is preserved, not just an error summary, which is exactly what you need to *replay* a DLQ entry after fixing whatever caused it. |
| **Why idempotency matters here** | Because delivery is at-least-once (not exactly-once, see below), the same Kafka message *can* be processed twice (rebalance-during-processing, worker crash-after-processing-before-commit). `UploadOutputsToS3`'s S3 key structure (`{jobID}/{rendition}/{filename}`, deterministic from job ID + rendition, no random component) means a duplicate full reprocessing of the same job produces an **idempotent overwrite** at the S3 layer — the second run's `PutObject` calls just overwrite the same keys with equivalent content, not create duplicates. This is a real, load-bearing idempotency property, even though nothing in the code explicitly labels it as such. |
| **Alternatives** | An idempotency-key + dedup-table pattern (record processed message IDs in Postgres, check-and-skip on redelivery) — stronger guarantee (true exactly-once *effect*), more machinery (a table, a transaction per message, cleanup/TTL policy for the dedup table). This repo relies on the weaker-but-simpler "S3 overwrite is naturally idempotent" property instead, which works *for the S3 side* but does **not** cover Postgres — and since the worker doesn't write to Postgres at all currently (the known gap), there's no double-write-to-DB risk today, but if/when that gap is closed, whoever wires in `RecordStatusEvent` calls needs to either make those calls idempotent (e.g., `ON CONFLICT DO NOTHING`) or accept duplicate `job_status_events` rows on redelivery. |
| **Tradeoffs** | DLQ messages currently have **no automated recovery path** — nothing in this repo consumes `transcoding-dlq` and retries/reprocesses it; it's a manual-inspection-only sink today (`kube/prometheus-rules.yaml`'s `DLQBacklog` alert is the only automated signal that DLQ is growing). |
| **Common mistakes** | Sending to DLQ without enough context to debug or replay — avoided here (full original message + failure reason + pod + timestamps preserved). A real gap: `SendDLQ` (the older, `Job`-based variant, `pkg/kafka.go:116-154`) sets `PodID: ""` unconditionally (`pkg/kafka.go:134`) — dead code today since `handleJobOutcome` only calls `SendDLQFromMessage`, but if `SendDLQ` were ever wired in, DLQ entries from that path would always have an empty pod attribution. |
| **Performance implications** | DLQ publish uses the same `RetryWithBackoff` (5x, 1s-16s) as the main topic — a DLQ publish itself failing (broker down for *both* topics) means the job is neither retried nor DLQ'd; it's simply lost (offset still gets committed after the DLQ attempt regardless of DLQ success/failure, `cmd/worker/main.go:244-247`) — a real, if narrow, at-least-once guarantee violation under compound-failure conditions (worth flagging as an edge case). |
| **Production considerations** | `pulsegrid_queue_depth_dlq_jobs` is referenced by `kube/prometheus-rules.yaml`'s `DLQBacklog` alert (line 46) but **no code in this repo ever populates that metric** — `pkg/queue_depth.go`'s poller only measures `transcoding-jobs`, not `transcoding-dlq`. This is a real, checkable observability gap: the alert rule exists, the metric it depends on does not, so `DLQBacklog` can never fire. Excellent "find the bug" interview material. |
| **Security considerations** | DLQ messages contain `S3URI`s and `FailureReason` strings (which may echo back ffmpeg stderr content, though not directly — `FailureReason` here is `processErr.Error()`, `cmd/worker/main.go:241,257`, and `TranscodingError.Error()` does *not* include `Stderr` in its string form, `pkg/errors.go:31-33` — only `Message`, so raw ffmpeg output does *not* leak into DLQ messages, only into structured logs via `LogTranscodeFailure`). |
| **Scalability considerations** | A DLQ topic under sustained high failure rate could itself need partition/retention tuning independent of the main topic — not configured distinctly anywhere in the reviewed files. |
| **Failure scenarios** | See "compound failure" note above (both topics down simultaneously). |
| **How this repo implements it** | `pkg.DLQMessage` type, `SendDLQFromMessage`/`SendDLQ` methods on `KafkaClient`, called from `handleJobOutcome`'s permanent and max-retries-exceeded branches. |
| **Relevant files** | `pkg/kafka.go:37-44,114-208`, `cmd/worker/main.go:236-259`, `kube/prometheus-rules.yaml:45-52` |

### Event-Driven Architecture

| | |
|---|---|
| **What it is** | Components communicate by publishing/consuming discrete events (Kafka messages, DB status-event rows) rather than direct synchronous calls. |
| **Why this approach** | Decouples the API's request-handling lifecycle from the worker's processing lifecycle in both time (async) and failure domain (a worker crash doesn't propagate an error back to the original HTTP caller, who already got their `202`). |
| **Two event streams exist** | (1) The **Kafka job event** (submit → maybe-retry → maybe-DLQ) — this is the operational, at-least-once, ordered-per-job event stream driving actual work. (2) The **`job_status_events` Postgres table** (a TimescaleDB hypertable, `db/migrations/002_create_job_status_events.sql`) — intended as an audit/analytics event log of job lifecycle transitions (`submitted`, presumably `processing`/`completed`/`failed` if ever wired up). Only the API side currently writes to it (`dbClient.RecordStatusEvent(r.Context(), job.JobID, "submitted")`, `cmd/api/main.go:351-353`) — the worker never does, consistent with the broader "worker has no DB client" gap. |
| **Alternatives** | A full event-sourcing model (job state *derived* from replaying `job_status_events`, rather than `jobs.status` being directly mutated) — `job_status_events` being a TimescaleDB hypertable suggests this was at least contemplated (hypertables are built for high-volume time-series event data, exactly what an event-sourced audit log needs), but the actual `jobs` table is mutated directly (`UPDATE jobs SET status = $1`, `pkg/postgres.go:297`) as the source of truth, with `job_status_events` as a secondary, currently-incomplete audit trail rather than the system of record. This is a hybrid, not pure event sourcing. |
| **Tradeoffs** | The hybrid model gives fast, simple point-queries (`GetJobByID` reads one row) without needing to replay an event log — at the cost of the audit trail being only as complete as the (currently incomplete) write-path into `job_status_events`. |
| **Common mistakes** | N/A specific to this repo beyond the incompleteness noted. |
| **Performance implications** | TimescaleDB hypertables optimize for time-range queries and high insert throughput on time-partitioned data — a reasonable choice *if* every job lifecycle transition eventually gets recorded there (target: multiple events per job × thousands of jobs/day), underused currently since only one event type (`"submitted"`) is ever inserted. |
| **Production considerations** | N/A beyond what's covered. |
| **Security considerations** | N/A. |
| **Scalability considerations** | Hypertables' chunk-based partitioning (by `event_timestamp`) scales insert/query performance well as the table grows — a legitimate scale-ahead decision even though current write volume doesn't yet stress a plain table. |
| **Failure scenarios** | N/A beyond incomplete audit trail as a "failure to fully investigate an incident" risk — if `job_status_events` only ever shows `"submitted"`, it cannot help answer "when did this job start processing vs. finish" during an incident, forcing reliance on worker pod logs (which rotate/are ephemeral) instead. |
| **How this repo implements it** | `RecordStatusEvent(ctx, jobID, eventType)` (`pkg/postgres.go:257-269`), called once per job from the API on successful submission. |
| **Relevant files** | `db/migrations/002_create_job_status_events.sql`, `pkg/postgres.go:257-269`, `cmd/api/main.go:351-353` |

### Exactly-Once vs. At-Least-Once vs. Ordering

| | |
|---|---|
| **What it is** | Delivery semantics describe how many times a consumer might observe/act on a given message under failure: at-most-once (may be lost, never duplicated), at-least-once (never lost, may be duplicated), exactly-once (never lost, never duplicated — the hardest and most expensive to guarantee end-to-end). |
| **Why this repo is at-least-once, not exactly-once** | Kafka itself supports exactly-once *semantics* (transactional producers + read-committed consumers) but this repo doesn't use them (`MaxAttempts: 1` + manual `RetryWithBackoff`, no Kafka transactions API usage) — and more fundamentally, exactly-once end-to-end would require every *side effect* (S3 writes, DB writes) to participate in the same transactional boundary as the offset commit, which Kafka's exactly-once guarantees don't extend to external systems by default. This repo instead achieves **effectively-once outcomes** for S3 (via idempotent overwrite, discussed above) on top of at-least-once *delivery* — a very common, pragmatic real-world pattern: don't fight for true exactly-once, make your side effects naturally idempotent instead. |
| **Why ordering matters, and where it's guaranteed vs. not** | Ordering is guaranteed **per-partition**, and partition assignment is by `job_id` hash (`Balancer: &kafka.Hash{}`) — so all messages for one job (original submission + any retries) are strictly ordered relative to each other. There is **no ordering guarantee across different jobs** (nor is one needed — jobs are independent units of work). |
| **Alternatives** | True exactly-once via Kafka transactions (`Transactional.ID`, `InitTransactions`, etc. in the underlying protocol) — kafka-go supports a subset of this; not used here. Would remove the *need* for idempotent S3 writes as a safety net, at real cost: transactional producers add latency and complexity, and still wouldn't cover Postgres writes without a separate two-phase-commit-style mechanism (outbox pattern, change-data-capture) — which is a bigger architectural undertaking than this system's current scale justifies. |
| **Tradeoffs** | At-least-once + idempotent writes is simpler to build and reason about than true exactly-once, at the cost of *relying on every downstream side effect independently being idempotent* — a property that has to be manually verified for each new side effect added (as noted above, this holds for S3 today but would need explicit attention if/when Postgres writes are added to the worker). |
| **Common mistakes** | Assuming "Kafka guarantees exactly-once" as a blanket property without checking which specific guarantees are actually configured (a very common interview trap — many candidates conflate "Kafka *can* do exactly-once" with "this system *has* exactly-once" without checking `MaxAttempts`, transactional config, or idempotent-producer settings). |
| **Performance implications** | At-least-once with idempotent side effects is cheaper (lower latency, simpler code) than exactly-once machinery — a deliberate scale-appropriate tradeoff. |
| **Production considerations** | Duplicate processing (rare, but structurally possible) means a job could theoretically be transcoded twice and uploaded twice (harmless due to key-overwrite idempotency) but would also double-count `pulsegrid_job_completed_total`/`pulsegrid_transcode_duration_seconds` observations if both duplicate runs reach `handleJobOutcome`'s success path — a real, if rare, metrics-accuracy risk worth naming. |
| **Security considerations** | N/A directly. |
| **Scalability considerations** | Per-partition ordering is what makes partition count the parallelism ceiling (discussed under Consumer Groups above) — a direct consequence of choosing ordering-per-key over global ordering or no ordering. |
| **Failure scenarios** | Covered extensively above (rebalance mid-processing, compound DLQ+main-topic outage). |
| **How this repo implements it** | Hash-partition by `job_id`; manual commit-after-outcome; no Kafka transactions; idempotent S3 key structure as the practical exactly-once-*effect* mechanism. |
| **Relevant files** | `pkg/kafka.go:56-59,101-104`, `pkg/s3client.go:270-349` (deterministic key structure) |

### Backpressure & Horizontal Scaling

| | |
|---|---|
| **What it is** | Backpressure: a mechanism by which a system under load signals upstream producers to slow down (or absorbs the load gracefully) rather than falling over. Horizontal scaling: adding more instances of a stateless component to increase aggregate capacity. |
| **Why this approach** | PulseGrid's backpressure mechanism is entirely **implicit and queue-based**, not an explicit rate-limit/rejection at the API: the API always accepts uploads (up to the 10GB size cap) and always publishes to Kafka regardless of current queue depth — Kafka's durable log *absorbs* burst load, and the actual backpressure signal is **consumer lag**, which KEDA reads to scale worker pod count up in response (`lagThreshold: "10"`, `kube/worker-deployment.yaml:87` — scale up once average lag per partition exceeds 10 messages). |
| **Alternatives** | Explicit backpressure at the API (e.g., reject new uploads with `503` once queue depth exceeds a threshold, or apply a token-bucket rate limiter per client) — protects the API/Kafka from unbounded queue growth proactively; this repo instead lets the queue grow unboundedly (bounded only by broker disk) and relies entirely on worker autoscaling to catch up, which is simpler but means a sustained submission rate exceeding max worker throughput (30 pods × per-pod throughput) has **no circuit breaker** — the queue just keeps growing, and `HighQueueDepth` (`kube/prometheus-rules.yaml:14-21`, fires at >100 jobs for 5 min) is purely an *alert*, not an automatic corrective action. |
| **Tradeoffs** | No explicit admission control means the system is simple to reason about (accept everything, let the queue+autoscaler handle it) but has no defense against a genuine capacity-exceeding sustained load spike beyond `maxReplicaCount: 30` — once 30 workers can't keep up, queue depth grows without bound until disk-full on the Kafka brokers, a real, unmitigated failure mode. |
| **Common mistakes** | Conflating "the queue can absorb bursts" with "the queue can absorb *sustained overload*" — a burst (temporary spike, then return to normal) is exactly what a queue+autoscaler handles well; sustained overload (average submission rate genuinely exceeds max scaled-out throughput) is a capacity-planning failure no amount of queueing fixes, only more workers or fewer jobs. |
| **Performance implications** | `pollingInterval: 15` + `cooldownPeriod: 300` (`kube/worker-deployment.yaml:77-78`) means KEDA checks lag every 15s but waits 5 minutes of sustained low-lag before scaling *down* — asymmetric responsiveness (fast scale-up, slow scale-down) is a deliberate and common autoscaling pattern that avoids flapping (rapid up/down oscillation) at the cost of over-provisioning during the cooldown window after a burst subsides. |
| **Production considerations** | `minReplicaCount: 0` (`kube/worker-deployment.yaml:79`) means the worker fleet can scale to **zero** during idle periods — real cost savings, but introduces cold-start latency for the *first* job after idle (KEDA has to detect lag, scale up from 0, and the new pod has to start, connect to Kafka/S3, before the first job even begins processing) — worth quantifying if SLA-sensitive. |
| **Security considerations** | N/A directly, though unbounded queue growth from a malicious high-volume submission client is effectively a resource-exhaustion vector with no rate-limit defense at the API layer (see [Security](#input-validation)). |
| **Scalability considerations** | This whole section *is* the scalability story — the honest summary: horizontal scaling here is real and well-designed for burst absorption, but has no explicit backpressure/admission-control layer, relying entirely on "add more workers" as the single lever, bounded by `maxReplicaCount` and ultimately by AWS account/node-group capacity. |
| **Failure scenarios** | Kafka broker disk fills from unbounded queue growth under sustained overload → publishes start failing → API's `EnqueueJob` retries exhaust → DB transaction rolls back → clients start seeing `500`s — this is the system's actual failure mode under sustained overload, and it's a *cascading* one (queue failure becomes API failure) rather than a clean, isolated degradation. |
| **How this repo implements it** | KEDA `ScaledObject` with a Kafka trigger reading consumer group lag; no explicit rate limiting or admission control anywhere in `cmd/api`. |
| **Relevant files** | `kube/worker-deployment.yaml:66-87`, `kube/prometheus-rules.yaml:14-21` |

## Part 3 — Database

### Postgres, Transactions & Consistency

| | |
|---|---|
| **What it is** | RDS-managed PostgreSQL 15, accessed via `pgx/v5` + `pgxpool` connection pooling; the atomic DB-then-Kafka write pattern is the system's core consistency mechanism. |
| **Why Postgres** | Strong relational consistency (ACID transactions), mature tooling, and — specifically — the ability to combine a transactional `INSERT` with a subsequent conditional `UPDATE`+`COMMIT` around an external side effect (the Kafka publish), which is exactly the "atomic-ish" submission pattern this repo builds. `USER_GUIDE.md`/`aws-architecture-review.md` both note Postgres (RDS `db.t4g.micro`) as a "required resource — no cheaper managed Postgres exists at this tier." |
| **Why this transaction pattern specifically** | `InsertJobTx` (`pkg/postgres.go:317-358`) begins a transaction and inserts the job row with `status = 'submitting'`, returning a `TxHandle` — **the transaction is deliberately left open** across the Kafka publish call, which happens *outside* any DB call, back in `handleVideoUpload` (`cmd/api/main.go:326-353`). If the Kafka publish fails, the handler calls `txHandle.Rollback` — the DB row is never persisted, so from the client's perspective (and any concurrent reader's) the job never existed. If Kafka succeeds, `UpdateStatusAndCommit` updates status to `'submitted'` and commits in one round-trip. This is a real, if unusual, pattern: **holding an open Postgres transaction across a network call to a different system** (Kafka) — a well-known anti-pattern in most contexts (holds a lock/connection for the duration of an unrelated slow external call, risking connection pool exhaustion and lock contention) that this repo accepts deliberately to get "no orphaned Kafka messages without a DB row" as a guarantee. |
| **Alternatives** | **Outbox pattern** — write the job row *and* a "pending Kafka message" row in the *same* local transaction (no cross-system network call inside the transaction boundary), then a separate relay process/CDC reads the outbox table and publishes to Kafka asynchronously, marking rows as published. This is the textbook correct solution to the exact problem this repo is solving, and it avoids holding a DB transaction open across a network call — at the cost of needing a relay process (more moving parts) and *eventual* (not immediate) consistency between "job exists in DB" and "job exists in Kafka." The repo's own code comments acknowledge the tradeoff directly: "If DB commit fails after Kafka: log ALERT (orphan possible), still return 202" (`cmd/api/main.go:324`) — this is the *exact* failure mode the outbox pattern eliminates, present here as an accepted, documented risk instead. |
| **Tradeoffs** | Holding the transaction open across the Kafka call means: (1) the DB connection is checked out of the pool for the full duration of a Kafka publish (including its up-to-~31s retry window) — under load, this could exhaust `pgxpool`'s connection limit faster than expected; (2) any lock the `INSERT` takes (row-level, minimal here since it's a fresh row with a generated PK) is held for that same duration. In exchange: no separate relay process, no outbox table, no eventual-consistency window — a real, working, honest simplicity-for-safety tradeoff, clearly reasoned about (not accidental) based on the code comments. |
| **Common mistakes** | Not handling the "Kafka succeeded, DB commit failed" case explicitly — this repo *does* handle it, just not by preventing it (can't, by definition, once you're past the Kafka call) but by loud logging (`log.Printf("ALERT: ...")`, `cmd/api/main.go:346`) — the honest acknowledgment that this is an operator-facing, not code-fixable, edge case given the chosen pattern. |
| **Performance implications** | pgx's `pgxpool` handles connection pooling automatically (see [Connection Pooling](#connection-pooling) below); the transaction-held-open-across-Kafka pattern is the main latency amplifier on the write path — worth being able to estimate: best case (Kafka publish succeeds first try) adds Kafka's round-trip latency (single-digit ms typically) to the transaction hold time; worst case (Kafka retries all 5 attempts) adds up to ~31 seconds of held-open transaction time, which is a meaningfully long time to hold a DB connection under load. |
| **Production considerations** | `RecordStatusEvent` (`cmd/api/main.go:351-353`) is called **after** the transaction commits, as a separate, unguarded ("best-effort," per its own comment) statement — if it fails, the job submission has already succeeded and returned to the client; only a `log.Printf("WARN: ...")` records the miss. This is a reasonable choice (an audit-log write shouldn't be allowed to fail an otherwise-successful submission) but means `job_status_events` can silently under-report even the one event type it currently tracks. |
| **Security considerations** | `DATABASE_URL` embeds the password in the connection string — see [Secrets](#secrets--credential-management). All queries use `$1`/`$2`-style parameterized placeholders (`pgx` native parameter binding, e.g. `pkg/postgres.go:120-133,338-351`) — **no string-concatenated SQL anywhere in this codebase**, meaning SQL injection is structurally prevented by the client library's API shape, not by manual escaping discipline. Worth confirming explicitly and citing as a real strength. |
| **Scalability considerations** | Single-writer Postgres (no read replicas configured, `db_multi_az = false` by default, `terraform/variables.tf:171-174`) — all reads (`GetJobByID`, `QueryJobs`) and writes hit the same primary instance; fine at current scale, a real future bottleneck if job-status-query volume (e.g., from many polling clients, exactly what `cmd/load-test` simulates) grows independently of write volume. |
| **Failure scenarios** | RDS failover (if Multi-AZ were enabled) or a connection drop mid-query surfaces as a `pgx` error, wrapped and returned up the call chain — `handleGetJob`/`handleListJobs` return `500 INTERNAL_ERROR` in that case (`cmd/api/main.go:541,489`), not a `503`, meaning clients can't easily distinguish "transient DB blip, retry me" from "this specific request is malformed" — a minor API-design gap. |
| **How this repo implements it** | `pgxpool.Pool` wrapped in `PostgresClient`; `InsertJobTx`/`pgTxHandle` implement the held-open-transaction pattern; every other query (`GetJobByID`, `QueryJobs`, `RecordStatusEvent`) is a single autocommit statement. |
| **Relevant files** | `pkg/postgres.go`, `pkg/postgres_test.go`, `db/migrations/001_create_jobs_table.sql` |

### Indexes & Schema Design (including a real bug)

| | |
|---|---|
| **What it is** | B-tree indexes on `jobs.status`, `jobs.submission_time DESC`, `jobs.completion_time DESC` (`db/migrations/001_create_jobs_table.sql:21-23`); a `CHECK` constraint restricting `status` to four literal values; a separate TimescaleDB hypertable for time-series events. |
| **Why these specific indexes** | They map directly to `QueryJobs`' actual query shape (`pkg/postgres.go:181-255`): filtering by `status IN (...)` (uses `idx_jobs_status`), filtering/sorting by `submission_time` (uses `idx_jobs_submission_time`, and the query's `ORDER BY submission_time DESC` — descending index matches descending sort exactly, avoiding a sort step), and — less directly used today — `completion_time` for duration-based queries. This is textbook "index what you query by," not speculative indexing. |
| **A real, checkable bug worth knowing cold** | The `jobs.status` column has `CHECK (status IN ('submitted', 'processing', 'completed', 'failed'))` (`db/migrations/001_create_jobs_table.sql:6`) — but `pkg.JobStatusSubmitting = "submitting"` is a **fifth** status value defined in Go (`pkg/models.go:9`), and `InsertJobTx` explicitly inserts `string(JobStatusSubmitting)` as the initial row status (`pkg/postgres.go:340`, hardcoded, not even parameterized from the caller's `job.Status`). **Against a real Postgres instance with this migration applied, every single job submission would fail its `INSERT` with a CHECK constraint violation**, which `InsertJobTx` would then wrap and return as an error, which `handleVideoUpload` would surface as `500 INTERNAL_ERROR` to every client, on every upload, permanently. This is presumably why the repo's own tests never caught it — `implementation-audit.md` Task 25.1 confirms **no test exercises the real, migrated schema**; every DB test in this repo runs against a mock `DBClient`, not a live Postgres instance with `db/migrate.go` actually applied. This is one of the best interview artifacts in the whole repo: it demonstrates concretely *why* "the mocks all pass" is not the same claim as "the system works," and it's real, not contrived for this guide. |
| **Why this approach (schema design generally)** | `requested_renditions JSONB` stores the rendition spec array without a separate `renditions` table — a deliberate denormalization: renditions are always read/written as a unit with their parent job, never queried independently ("find all jobs with a 1080p rendition" isn't a supported query), so JSONB avoids a join for zero real query-flexibility cost given current access patterns. |
| **Alternatives** | A normalized `job_renditions` table (one row per rendition per job) — enables querying/aggregating by rendition type directly in SQL (e.g., "how many 1080p transcodes this week") without JSONB path operators; costs a join on every job read and write-side complexity (insert N rows instead of one JSONB blob) for a query pattern not currently needed. The JSONB choice is defensible at current scale/requirements. |
| **Tradeoffs** | JSONB means renditions are opaque to indexing/constraints — a malformed rendition spec (e.g., missing `id`) is caught at the *application* layer (`parseRenditions`, `cmd/api/main.go:583-587`) before it ever reaches Postgres, not by the schema itself; the schema trusts the app layer entirely for that column's internal structure. |
| **Common mistakes** | Missing an index that a hot query path actually needs — not observed here (indexes match query shape well). The CHECK-constraint mismatch above is the standout schema mistake in this codebase. |
| **Performance implications** | Three B-tree indexes on a `jobs` table add write-amplification (every `INSERT`/`UPDATE status` maintains three index structures, not just the heap row) — a reasonable cost given `jobs` is a moderate-write, read-heavy-relative-to-write table (one insert + a handful of status updates per job's lifetime, but potentially many status-polling reads per job from clients). |
| **Production considerations** | `updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP` (`db/migrations/001_create_jobs_table.sql:18`) has **no trigger** to actually update it on row modification — it will forever hold the `INSERT` time, never reflecting subsequent `UPDATE`s (like the status-flip from `submitting`→`submitted`, or any future worker-side status write-back). This is a second real, checkable schema gap: the column exists, implies "last modified," but nothing maintains that invariant. |
| **Security considerations** | `job_id UUID PRIMARY KEY` — using UUIDs (vs. auto-increment integers) as the primary key avoids leaking sequential job-count/rate information to clients who can see job IDs (a real, if minor, information-disclosure consideration; auto-increment IDs let an observer estimate total job volume by comparing two IDs' numeric gap). |
| **Scalability considerations** | `job_status_events` as a TimescaleDB hypertable (`create_hypertable(...)`, `db/migrations/002_create_job_status_events.sql:14`) is specifically a scale-ahead decision — hypertables auto-partition by time into chunks, keeping index sizes and query performance stable as the table grows to millions/billions of rows, versus a plain table's index bloat over time. Good foresight *if* the write path into this table is ever completed. |
| **Failure scenarios** | The CHECK-constraint bug above is the headline failure scenario: **100% of job submissions fail against a real, freshly-migrated database.** This is worth stating plainly if asked "what would you fix first" in an interview — it's a one-line migration or one-line code fix (`ALTER TABLE jobs ... CHECK (status IN (..., 'submitting'))` or simply never persisting the `submitting` interim state and inserting directly as `submitted`... but that would reopen the orphan-prevention problem the transaction pattern exists to solve — so the *correct* fix is almost certainly widening the CHECK constraint, not changing the Go logic). |
| **How this repo implements it** | See above. |
| **Relevant files** | `db/migrations/001_create_jobs_table.sql`, `db/migrations/002_create_job_status_events.sql`, `pkg/models.go:5-14`, `pkg/postgres.go:317-358` |

### Migrations & Connection Pooling

| | |
|---|---|
| **What it is** | A minimal embedded-SQL-file migration runner (`db/migrate.go`) using Go's `embed.FS` to bundle `.sql` files into the compiled binary, tracked via a `schema_migrations` table; `pgxpool.Pool` for connection pooling. |
| **Why this approach** | `//go:embed migrations/*.sql` (`db/migrate.go:14-15`) bakes migration files *into the API binary itself* at compile time — no separate migration-file deployment step, no risk of the binary and its expected schema drifting apart because they're literally built together. This is a deliberate simplicity choice over a standalone migration tool (`golang-migrate`, `goose`, `flyway`) — no new dependency, full control over the (intentionally minimal) runner logic, at the cost of reimplementing what those tools already solve well (down-migrations, dry-run, checksums-of-applied-migrations-to-detect-drift — none of which this runner has). |
| **Why run on API startup, not a separate job/init-container** | `db.RunMigrations` is called directly in `cmd/api`'s `main()` (`cmd/api/main.go:82-87`), gated by `DATABASE_URL` being set — meaning every API pod that starts up attempts to run pending migrations. With 2 API replicas (`kube/api-deployment.yaml:32`), **both pods race to run migrations on every deploy** — the per-migration transaction + `schema_migrations` existence check (`pkg/postgres.go`... actually `db/migrate.go:51-61`) provides some protection (a migration already recorded is skipped), but there's a real TOCTOU (time-of-check-to-time-of-use) race: two pods could both see a migration as "not yet applied," both begin a transaction, and one would fail on the `schema_migrations` primary-key insert (`version VARCHAR(255) PRIMARY KEY`, `db/migrate.go:24`) after the other commits first — the losing pod's migration transaction rolls back entirely (transactional DDL, correctly implemented, `db/migrate.go:69-88`), and that pod would then `log.Fatalf` and crash-loop until it retries and sees the migration as already applied. This is a real, if self-healing, deploy-time race condition worth being able to explain. |
| **Alternatives** | A dedicated migration Job/init-container run once before the Deployment rolls out (standard Kubernetes pattern: `Job` with `restartPolicy: Never`, Deployment's rollout waits on it) — eliminates the multi-pod race entirely, cleanly separates "apply schema" from "serve traffic" as distinct lifecycle phases. This repo's choice (migrate-on-boot, both pods racing) is simpler to deploy (no extra manifest) but a real production risk under multi-replica rolling deploys, exactly the kind of tradeoff worth naming unprompted in an interview. |
| **Tradeoffs** | Migrate-on-boot also means a **rollback** (deploying an older binary after a bad migration) does not automatically un-apply the migration — `db/migrate.go` has no down-migration concept at all, matching many lightweight migration tools' philosophy ("migrations are forward-only; write a new migration to undo a mistake") but worth stating explicitly as a chosen constraint, not an oversight. |
| **Common mistakes** | Non-transactional migrations (a multi-statement migration where a mid-way failure leaves partial schema changes applied) — avoided here: each migration runs inside `pool.Begin(ctx)` / `tx.Exec` / commit-or-rollback (`db/migrate.go:70-88`), so a failing migration leaves the schema exactly as it was before the attempt. |
| **Performance implications** | Migrations run once at startup, not per-request — no steady-state performance impact; startup latency for the API pod increases by however long pending migrations take (negligible for this repo's two small migrations, could matter for a future large `ALTER TABLE` on a big table). |
| **Production considerations** | Only `cmd/api` runs migrations; `cmd/worker` does not (confirmed: no `db.RunMigrations` call anywhere in `cmd/worker/main.go`, and indeed the worker has no DB client to run them with) — `USER_GUIDE.md`'s FAQ states this explicitly. This means a schema change **must** be deployed via the API first (or at minimum, simultaneously) for the worker (if it ever gains DB writes) to rely on new columns/tables existing. |
| **Security considerations** | Migration SQL is embedded at compile time from the repo — no runtime-supplied SQL execution path, meaning there's no user-input-driven migration-injection surface (migrations are a build-time artifact, not a runtime one). |
| **Scalability considerations** | `pgxpool`'s default pool sizing (not explicitly configured anywhere in `pkg/postgres.go` — no `MaxConns`/`MinConns` set on the parsed `pgxpool.Config`, meaning it uses `pgxpool`'s library defaults) — worth knowing this is *unconfigured*, a real "what would you tune" interview answer: pool size should scale with expected concurrent request volume × query latency, and currently relies entirely on library defaults rather than a value chosen for this workload's actual concurrency. |
| **Failure scenarios** | Two API pods racing on first-deploy migration (discussed above) — self-heals via crash-loop-and-retry, but produces confusing logs/pod restarts during exactly the moment (a fresh deploy) when clear signal matters most. |
| **How this repo implements it** | `embed.FS` + `schema_migrations` tracking table + per-file transaction; called once from `cmd/api`'s `main()`. |
| **Relevant files** | `db/migrate.go`, `db/migrations/*.sql`, `cmd/api/main.go:82-87` |

## Part 4 — AWS

This section deliberately doesn't re-derive the resource inventory or cost table — that's `aws-architecture-review.md` in full. This section explains the *engineering reasoning* behind each choice and cross-references specific numbers from that audit.

### S3 (Source & Output Storage)

| | |
|---|---|
| **What it is** | Two buckets (`{env}-source`, `{env}-output`) — durable, versioned-optional, encrypted, lifecycle-managed object storage. |
| **Why S3** | The workload's core artifact (video files, up to 10GB) is large, immutable-once-written, and needs to be durably stored and later streamed/downloaded by a different process (the worker) potentially on a different node — exactly S3's design point. `aws-architecture-review.md` calls it a "required resource, no cheaper AWS-native substitute." |
| **Why two buckets, not one** | Separates source (write-once by API, read-once by worker, then largely cold) from output (write-once by worker, read-many by end consumers/CDN in a hypothetical future) — different lifecycle policies could diverge (they don't currently — both use identical Glacier-at-90-days/expire-at-365-days rules, `terraform/main.tf:493-531,569-607`), and different IAM scoping is cleaner with separate ARNs. |
| **Alternatives** | EBS/EFS-backed shared storage — would require the API and worker to be co-located or share a network filesystem, defeating the independent-pod-scheduling model; S3's HTTP-API-based access model needs no shared filesystem at all, which is what lets 30 ephemeral worker pods each independently download/upload without any shared mount. |
| **Tradeoffs** | S3 GET/PUT latency (tens-to-hundreds of ms per request) is higher than local/EBS disk I/O — irrelevant here since the dominant cost is the transcode itself (minutes), not the S3 round-trip. |
| **Common mistakes** | Buffering large files fully in memory before upload — avoided (streaming multipart upload, see [Memory](#memory-escape-analysis--garbage-collection)). Not setting `abort_incomplete_multipart_upload` — this repo does (`days_after_initiation = 1`, both buckets, `terraform/main.tf:497-505,573-581`), preventing orphaned multipart upload parts (which silently accrue storage cost) from a client that starts an upload and never completes it. |
| **Performance implications** | `manager.Uploader` with `Concurrency: 5` (`pkg/s3client.go:44-46`) parallelizes multipart upload parts — meaningfully faster wall-clock upload time for large files than sequential parts, at the cost of 5x the concurrent bandwidth usage per upload. |
| **Production considerations** | `StorageClassStandard` explicitly set on every upload (`pkg/s3client.go:75,249`) — not left to bucket default — ensures immediate-access performance for freshly uploaded/produced files, with the *lifecycle rule* (not upload-time class selection) handling the cost-optimized transition to Glacier at 90 days. This is the correct division of concerns: hot-path code shouldn't need to reason about storage-class economics, lifecycle policy should. |
| **Security considerations** | `aws_s3_bucket_public_access_block` on both buckets, all four sub-settings `true` (`terraform/main.tf:533-540,609-616`) — blocks any accidental public ACL/policy regardless of what an individual object's ACL might request; server-side encryption (`AES256`, `terraform/main.tf:483-491,559-567`) at rest on both buckets. **The IAM story is the real gap** — see [IAM & IRSA](#iam--irsa) below. |
| **Scalability considerations** | S3 scales essentially without limit for this workload's request rate — not a bottleneck at any realistic PulseGrid scale. |
| **Failure scenarios** | Source object deleted/never-existed → `GetObject` returns `NoSuchKey`/`NotFound` → mapped to `*pkg.SourceNotFoundError` → classified `ErrorTypePermanent` → immediate DLQ, no wasted retries (`pkg/s3client.go:160-179`, `pkg/errors.go:91-95`). |
| **How this repo implements it** | `pkg.S3Client` wraps AWS SDK v2's `s3.Client` + `manager.Uploader`; Terraform provisions buckets with versioning-toggle, SSE, lifecycle, and public-access-block. |
| **Relevant files** | `pkg/s3client.go`, `terraform/main.tf:462-616`, `terraform/variables.tf:185-189` |

### IAM & IRSA

| | |
|---|---|
| **What it is** | AWS Identity and Access Management (roles, policies, trust relationships); IRSA (IAM Roles for Service Accounts) — an EKS-specific mechanism that lets individual Kubernetes pods assume a scoped IAM role via OIDC federation, instead of every pod on a node sharing the node's IAM role (or worse, static credentials). |
| **Why IRSA exists / why it matters here** | Without IRSA, AWS credentials for pod-level access are either (a) the EC2 node's instance-profile role — shared by *every* pod on that node, meaning a compromised low-privilege pod inherits the same S3/other permissions as every other pod co-located on the same node, or (b) static IAM access keys injected as Secrets — long-lived, rotatable only manually, leakable via `kubectl describe`/logs/env-dump. IRSA solves both: each Kubernetes ServiceAccount gets a distinct, narrowly-scoped IAM role, assumed via short-lived STS tokens automatically injected by EKS's OIDC webhook — no long-lived credentials to leak, and per-workload (not per-node) privilege scoping. |
| **What this repo actually does — the real finding** | `terraform/main.tf` **does** provision an EKS cluster with an OIDC-issuable identity (`aws_eks_cluster.main` implicitly creates an OIDC issuer URL once IRSA is set up on top of it — `aws-architecture-review.md` confirms `eks_cluster_oidc_issuer` is already output by Terraform) — but **no IAM role trust policy scoped to a Kubernetes ServiceAccount exists anywhere in this repo's Terraform**, and `kube/secrets.yaml` instead holds **static `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`** delivered as a plain Kubernetes Secret via `envFrom: secretRef` (`kube/api-deployment.yaml:57-58`, `kube/worker-deployment.yaml:36-37`). Instead, S3 access is granted via `aws_iam_role_policy.eks_nodes_s3` (`terraform/main.tf:371-396`) attached to the **node role** — meaning every pod scheduled on an EKS worker node (API or worker tier, both share node-level trust in this configuration) has the *node's* S3 permissions available via the instance profile, layered *underneath* the static-key credentials the app actually uses. `aws-architecture-review.md` calls this out plainly: "IRSA is set up for nothing" and recommends it as the #1 "stop the bleeding first" migration step. |
| **Why this matters as an interview topic** | This is a genuinely common real-world anti-pattern (static keys in a Secret, despite IRSA infrastructure being half-provisioned) — being able to explain *why* it's wrong (long-lived credential rotation burden, broader blast radius on leak, no per-workload scoping) and *what the fix looks like* (an `aws_iam_role` with a trust policy conditioned on the OIDC provider + a specific `system:serviceaccount:pulsegrid:pulsegrid-api` subject claim, annotated onto the k8s ServiceAccount via `eks.amazonaws.com/role-arn`) is exactly the kind of judgment a staff-level AWS/Kubernetes interview probes for. |
| **Alternatives** | Node-role-only (current node IAM policy) — simplest, worst isolation (any pod on the node effectively has node-level AWS access via the instance metadata service, regardless of which app it is). Static keys (current app-level credential, layered on top) — worst of both worlds: still broad (same keys work from anywhere the Secret is readable, not scoped to a specific pod's runtime identity) *and* long-lived/leakable. IRSA — correct answer, not implemented. |
| **Tradeoffs** | IRSA requires slightly more Terraform (an `aws_iam_openid_connect_provider` + per-workload `aws_iam_role` + trust policy + ServiceAccount annotation) — genuinely more setup than "attach a policy to the node role and inject static keys," which is presumably why the simpler (wrong) path was taken. |
| **Common mistakes** | Exactly what this repo has: provisioning the *infrastructure* IRSA needs (OIDC issuer) without completing the *workload-level* wiring (trust policy + ServiceAccount annotation), leaving both the broad node-role grant and static keys as parallel, redundant, worse-than-IRSA access paths. |
| **Performance implications** | IRSA's STS `AssumeRoleWithWebIdentity` token exchange adds a small one-time cost per credential refresh (cached and auto-refreshed by the AWS SDK, not per-request) — negligible. |
| **Production considerations** | Static keys in `kube/secrets.yaml` must be manually rotated — no automatic expiry, no automatic rotation — a real operational burden and standing risk that IRSA's short-lived STS tokens eliminate by design. |
| **Security considerations** | This is the single most significant, concretely fixable security finding in the whole repo. Full detail in [Part 9](#part-9--security). |
| **Scalability considerations** | N/A directly, though IRSA scales cleanly to many workload identities without proliferating IAM users/access keys, which static-key-per-service approaches do not. |
| **Failure scenarios** | Leaked static keys (checked into a fork, exfiltrated via a compromised pod, accidentally logged) grant S3 access from *anywhere on the internet*, indefinitely, until manually rotated — IRSA-issued STS tokens are scoped to the AWS account's OIDC trust and expire automatically (typically ~1 hour), making an equivalent leak far less damaging. |
| **How this repo implements it (as-is)** | Node IAM role + inline S3 policy (broad, node-wide) layered under static keys in a k8s Secret (app-level, long-lived). No IRSA. |
| **Relevant files** | `terraform/main.tf:296-396` (node role + policy), `kube/secrets.yaml`, `kube/api-deployment.yaml:57-58`, `kube/worker-deployment.yaml:36-37`, `aws-architecture-review.md` §1, §5, §9 (step 1) |

### EKS, Networking & Security Groups

| | |
|---|---|
| **What it is** | A managed Kubernetes control plane (EKS) with two managed node groups (`api`: t3.small, `worker`: t3.large/SPOT-capable), a VPC spanning 2 AZs with public+private subnets, one NAT Gateway, and three security groups (cluster, nodes, RDS). |
| **Why this approach** | EKS offloads control-plane operations (etcd, API server HA, upgrades) to AWS; two node groups let API and worker tiers have independently-sized/independently-scaled instance pools with different instance types matched to their actual resource profiles (API: light HTTP handling → t3.small; worker: ffmpeg CPU/memory-heavy → t3.large) — `aws_eks_node_group.worker` even carries a `taint { key = "workload", value = "transcoding", effect = "NO_SCHEDULE" }` (`terraform/main.tf:445-449`) to keep non-transcoding pods off worker-tier nodes (though no matching `toleration` is visible in `kube/worker-deployment.yaml` in the reviewed files — worth checking whether the Deployment actually tolerates its own dedicated node group's taint, or whether this taint currently prevents the worker Deployment from scheduling at all — a good "trace this end to end" exercise). |
| **Security group design** | Correctly layered: `eks_cluster` SG allows HTTPS (443) only from within the VPC (`terraform/main.tf:186-192`); `eks_nodes` SG allows node-to-node (`self = true`) and cluster-API-to-node ephemeral ports (1025-65535, sourced from the cluster SG specifically, not `0.0.0.0/0`, `terraform/main.tf:223-229`) plus CoreDNS TCP/UDP self-referencing rules; `rds` SG allows Postgres (5432) **only** from the `eks_nodes` SG (`terraform/main.tf:268-274`) — RDS is not reachable from the public internet or from any security group outside the node group, correct least-privilege network segmentation. All three SGs allow unrestricted egress (`0.0.0.0/0`), a common (if not maximally strict) default. |
| **Alternatives** | `aws-architecture-review.md` §6 lays this out directly: plain EC2 + Docker Compose/systemd, or ECS-on-EC2, would eliminate the $73/mo EKS control-plane fee and the operational overhead of node groups/RBAC/manifests for a workload of exactly two service types with no need for Kubernetes-specific primitives (no StatefulSets beyond the in-cluster Kafka, no complex scheduling beyond the taint above, no multi-team/multi-tenant isolation need). |
| **Tradeoffs** | EKS buys you KEDA (Kubernetes-native autoscaling on Kafka lag) essentially for free once you're on Kubernetes — replicating KEDA's exact behavior (poll Kafka lag, scale a fleet of workers) on plain EC2 would require hand-rolling a scaling controller (e.g., a Lambda + Auto Scaling Group API calls triggered by a CloudWatch custom metric) — a real, non-trivial engineering cost the audit's "just use EC2" recommendation somewhat understates. This is a good nuanced point to raise if asked to defend or critique the AWS review's recommendation. |
| **Common mistakes** | Overly broad SG rules (`0.0.0.0/0` on a database port) — not present here, RDS SG is correctly scoped. A real, if minor, mistake: `enable_nat_gateway` defaults `true` (`terraform/variables.tf:53-57`) even though the variable's own description states disabling saves ~$32/mo and is a supported toggle — the *default* is the more expensive option, not necessarily wrong, but worth noting defaults encode a "safety/simplicity over cost" bias here. |
| **Performance implications** | Single NAT Gateway (not one-per-AZ) is a minor availability/performance tradeoff — if that NAT Gateway's AZ has an outage, private-subnet egress from the *other* AZ's nodes is also affected (NAT Gateways are AZ-scoped in AWS), a real cross-AZ availability gap in the current topology. |
| **Production considerations** | `endpoint_public_access = true` on the EKS cluster (`terraform/main.tf:330`) means the Kubernetes API server is reachable from the public internet (protected by IAM/RBAC auth, not network isolation) — a common but real attack-surface consideration; `endpoint_private_access = true` is also enabled, so private (VPC-internal) access works too, meaning public access could be disabled entirely for a stricter posture at the cost of needing a bastion/VPN for `kubectl` access. |
| **Security considerations** | Covered above; full list in [Part 9](#part-9--security). |
| **Scalability considerations** | Node group `scaling_config` min/max/desired (`api_node_min_size=1/max=3`, `worker_node_min_size=0/max=5`, `terraform/variables.tf:79-95,114-124`) bounds the *node* count — separately from KEDA's *pod* count (0-30) — meaning pod-level autoscaling is itself bounded by whether the underlying node group has capacity to schedule those pods; a worker node group capped at 5 nodes of t3.large (2 vCPU/8GB each) cannot actually host 30 worker pods each requesting 2-4 CPU/4-8Gi (`kube/worker-deployment.yaml:43-49`) simultaneously — **this is a real, checkable capacity mismatch**: KEDA's `maxReplicaCount: 30` vastly exceeds what `worker_node_max_size: 5` t3.large nodes can physically schedule (roughly 1-2 worker pods per t3.large node given the resource requests, so ~5-10 pods max, not 30) unless cluster-autoscaler is also configured to grow the node group beyond its Terraform-set max (not evidenced in the reviewed files) or the max sizes are reconciled. Excellent "find the inconsistency across three different files" interview material (`terraform/variables.tf` node caps vs. `kube/worker-deployment.yaml` KEDA cap vs. actual pod resource requests). |
| **Failure scenarios** | Node group `desired_size` capacity exhausted (all nodes full) while KEDA wants to scale pods further → new worker pods stay `Pending` (unscheduled) → Kafka lag continues growing despite KEDA "trying" to help → without cluster-autoscaler, this is a silent capacity ceiling with no clear operator signal beyond `kubectl get pods` showing `Pending` and (if watched) `HighQueueDepth` firing without recovery. |
| **How this repo implements it** | Two managed node groups, taint on the worker group, layered SGs, single NAT Gateway, dual public/private endpoint access. |
| **Relevant files** | `terraform/main.tf:181-461`, `terraform/variables.tf`, `kube/worker-deployment.yaml` |

### Terraform (Infrastructure as Code)

| | |
|---|---|
| **What it is** | Declarative infrastructure provisioning; state stored remotely in S3 with DynamoDB locking, bootstrapped by a separate `terraform/state-bootstrap` configuration. |
| **Why Terraform** | Reproducible, diffable, version-controlled infrastructure — `terraform plan` shows exactly what will change before it happens, essential for a system with real cost implications (EKS control plane, NAT Gateway, RDS) where an accidental `apply` could be expensive or destructive. |
| **Why a separate state-bootstrap config** | The main config's `backend "s3"` block (`terraform/main.tf:16-22`) needs the S3 bucket + DynamoDB table to *already exist* before `terraform init` can use them as a backend — a classic chicken-and-egg problem, solved by a minimal, separately-applied bootstrap configuration (`terraform/state-bootstrap/`) that provisions just those two resources with local (or a different remote) state, run once per AWS account/environment before the main config is ever touched. |
| **Alternatives** | Terraform Cloud/Spacelift/Atlantis-managed state (no self-managed S3+DynamoDB backend needed) — offloads the bootstrap problem entirely to a SaaS/managed runner, at the cost of a new external dependency and (for Terraform Cloud specifically) potential cost; self-managed S3+DynamoDB (what's implemented) is the standard "roll your own" approach and is free beyond negligible S3/DynamoDB usage costs (`aws-architecture-review.md`: "<$1/mo"). |
| **Tradeoffs** | DynamoDB locking prevents concurrent `terraform apply` runs from corrupting state (two engineers running `apply` simultaneously) — essential team-safety mechanism, adds one more piece of infrastructure to reason about. |
| **Common mistakes** | Not enabling state encryption — checked, `encrypt = true` is set (`terraform/main.tf:20`). Hardcoding secrets into `.tfvars` committed to git — `db_password` is correctly passed via `-var` on the CLI (`USER_GUIDE.md`'s deployment steps: `terraform apply ... -var="db_password=<secret>"`), not stored in the checked-in `environments/*.tfvars` files, avoiding secret-in-git-history. |
| **Performance implications** | N/A — a provisioning-time concern, not runtime. |
| **Production considerations** | `environments/dev.tfvars` and `environments/prod.tfvars` separate sizing per environment (worth checking their actual diffs — likely node instance types/counts, `db_multi_az`, `enable_nat_gateway`) — the standard Terraform workspace-per-environment pattern via separate var files rather than separate state files/workspaces (both are valid approaches; var-files-only means both environments *could* accidentally share state if not carefully separated by backend `key`, worth confirming `terraform/main.tf`'s backend `key` is parameterized per environment in practice, which it is not shown to be in the static `key = "infrastructure/terraform.tfstate"`, `terraform/main.tf:18` — a real risk if dev and prod are ever applied against the same backend key without careful operator discipline). |
| **Security considerations** | State files can contain sensitive data in plan/refresh output (e.g., `db_password` flows through state) — `encrypt = true` on the S3 backend mitigates at-rest exposure; state file *access* (who can read the S3 object) is an IAM concern not detailed in these files. |
| **Scalability considerations** | N/A for the tool itself; the *infrastructure it provisions* has the scalability properties discussed elsewhere in this guide. |
| **Failure scenarios** | A `terraform apply` interrupted mid-run (network drop, `Ctrl+C`) can leave the DynamoDB lock held — `USER_GUIDE.md`'s troubleshooting table notes exactly this: "`terraform apply` fails on state locking or backend → Ensure `terraform/state-bootstrap` has been applied first." |
| **How this repo implements it** | Standard AWS provider, S3 backend + DynamoDB lock table, `environments/*.tfvars` for per-environment values, `variables.tf` for defaults/validation (note the `validation` blocks on `environment` and `az_count`, `terraform/variables.tf:16-19,47-50` — real input-validation discipline at the infra-as-code layer, not just the application layer). |
| **Relevant files** | `terraform/main.tf`, `terraform/variables.tf`, `terraform/outputs.tf`, `terraform/environments/*.tfvars`, `terraform/state-bootstrap/` |

### CloudWatch, Cost Optimization & Why Some AWS Services Were NOT Used

| | |
|---|---|
| **What it is** | This repo makes a deliberate, minimal-footprint AWS choice: no ALB/CloudFront/Route53/Lambda/ECS/API Gateway/SQS/SNS/EventBridge/Secrets Manager/KMS/CloudWatch-SDK usage anywhere — confirmed by `aws-architecture-review.md`'s explicit audit. |
| **Why this matters as a topic, not just an absence** | Being able to articulate *why* a service was **not** used is a stronger signal in a staff-level interview than listing services that were — it shows you understand the tradeoff space, not just the happy path. Below is the "why not," per service, synthesizing what's evidenced in the repo. |
| **Why no ALB/API Gateway** | The API Service is `ClusterIP` only (`kube/api-deployment.yaml:10`) — no `LoadBalancer` type Service, no Ingress resource in the reviewed `kube/` files, meaning **the API currently has no external entry point at all** in this configuration — worth flagging as a real gap (something has to front this for real external traffic; today, nothing in the manifests does, only internal cluster access). An ALB would be the natural fix (path-based routing, TLS termination, WAF integration) but wasn't added — possibly deliberately deferred as out-of-scope for "distributed video transcoding platform" plumbing, possibly a genuine gap. |
| **Why no Lambda/ECS** | Everything runs on EKS as long-lived pods, not event-triggered functions — appropriate given the worker's workload shape (long-running subprocess invocation up to 30 minutes, far beyond Lambda's 15-minute hard limit, and needing persistent local disk for ffmpeg's temp files, which Lambda's ephemeral `/tmp` — 512MB-10GB depending on config — would constrain awkwardly for large multi-rendition jobs). |
| **Why no SQS/SNS (replacing Kafka)** | Covered in depth under [Message Queues](#message-queues--why-kafka) — the tradeoff is ordering/replay/consumer-group-lag semantics (Kafka) vs. operational simplicity and native autoscaling metrics (SQS). `aws-architecture-review.md` recommends SQS as the pragmatic choice *if* this workload doesn't actually need Kafka's ordering/replay guarantees — worth being able to argue **both directions**: this repo *does* use partition-ordering for a real reason (in-order retry processing per job), so a wholesale SQS switch isn't free, even though it would remove real operational weight. |
| **Why no Secrets Manager/KMS** | Secrets flow through plain Kubernetes Secrets (`kube/secrets.yaml`), base64-encoded (not encrypted) at rest in etcd by default unless EKS envelope encryption is separately configured (not evidenced in `terraform/main.tf`) — Secrets Manager would add automatic rotation, audit logging (CloudTrail), and encryption-at-rest by default, at the cost of ~$0.40/secret/month plus API call costs and added application-code complexity (fetching secrets at runtime via SDK calls instead of env-var injection). This repo's choice trades security posture for simplicity — a legitimate startup-stage tradeoff, a real gap at any regulated/compliance-sensitive scale. |
| **Why no CloudWatch (metrics/logs)** | The observability stack is self-hosted Prometheus/Grafana (`kube/prometheus-rules.yaml`, `kube/grafana-dashboard.json`) instead of CloudWatch metrics/Container Insights — avoids CloudWatch's per-metric and per-GB-ingested costs at the expense of running/maintaining the Prometheus/Grafana stack yourself (not shown to be provisioned anywhere in `kube/` as its own Deployment in the reviewed files, meaning either it's assumed pre-installed on the cluster via a separate mechanism like the `kube-prometheus-stack` Helm chart, or this is itself a gap — the `PrometheusRule`/dashboard JSON are *config for* Prometheus/Grafana, not the Prometheus/Grafana Deployments themselves). |
| **Cost optimization, summarized (full detail in `aws-architecture-review.md`)** | Three biggest levers: EKS control plane ($73/mo fixed) vs. EC2, self-hosted 3-broker Kafka vs. SQS, and NAT Gateway ($32/mo fixed) vs. NAT instance or public subnets. Total estimated current cost: ~$200-210/mo at idle scale; minimal-footprint alternative: ~$60-90/mo. Being able to walk through *why* each of these three is expensive (fixed control-plane fee regardless of load; a full distributed system — ZooKeeper/KRaft, broker replication, PVCs — for a single-topic use case; a managed NAT gateway billed per-hour-plus-per-GB regardless of whether it's needed for a workload that could run in public subnets with tight SGs) is the actual cost-optimization skill being tested, not the dollar figures themselves. |
| **Relevant files** | `aws-architecture-review.md` (primary source — read in full), `terraform/main.tf`, `kube/*.yaml` |

## Part 5 — Infrastructure

### Docker & Multi-Stage Builds

| | |
|---|---|
| **What it is** | `Dockerfile.api`/`Dockerfile.worker` — two-stage builds (`golang:1.22-alpine` builder → `alpine:3.19` runtime), non-root user, static binary compilation. |
| **Why multi-stage** | The builder stage needs the full Go toolchain (~350MB+ image) but the *runtime* only needs the compiled binary + minimal OS packages — multi-stage builds let the final image `COPY --from=builder` just the artifact, discarding the entire build toolchain from the shipped image. Concretely: `Dockerfile.api`'s final stage is `alpine:3.19` + `ca-certificates`/`tzdata` + the ~binary — likely tens of MB, not hundreds. |
| **Why Alpine + `CGO_ENABLED=0`** | `CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build` (`Dockerfile.api:20-23`) produces a **statically linked** binary with no dynamic libc dependency — this is *why* Alpine (which uses musl libc, not glibc) works at all as a runtime base for a Go binary without needing musl-compatible dynamic linking; a CGO-enabled build would need a glibc-compatible base or careful musl-linking, adding real fragility this repo avoids entirely. |
| **Why worker's image has ffmpeg and api's doesn't** | `Dockerfile.worker:28` adds `apk add ... ffmpeg`; `Dockerfile.api` does not — directly reflects that only the worker invokes `exec.Command("ffmpeg", ...)`. This keeps the API image smaller and reduces its CVE/attack surface (no video-codec-parsing C library stack, historically a rich source of security vulnerabilities, present in an image that never needs it). |
| **Why non-root** | `addgroup -S pulsegrid && adduser -S pulsegrid -G pulsegrid` + `USER pulsegrid` (both Dockerfiles) — running containers as a non-root UID is a baseline container-security practice: a container-escape or arbitrary-code-execution vulnerability in the app has a much smaller blast radius as an unprivileged user than as root (can't modify most of the filesystem, can't bind privileged ports, harder to escalate to node-level compromise). |
| **Alternatives** | `FROM scratch` (zero base image, just the static binary) — even smaller, but loses `ca-certificates` (needed for TLS to AWS/Kafka/Postgres) and `tzdata` (needed for correct timezone handling) unless manually copied in — Alpine's tiny footprint plus these two packages is a pragmatic middle ground over `scratch`'s maximal minimalism. Distroless base images (Google's `gcr.io/distroless/static`) are another popular middle ground, not used here. |
| **Tradeoffs** | Alpine's musl libc has historically had subtle DNS-resolution behavior differences from glibc (a well-known Go-on-Alpine gotcha, particularly around `CGO_ENABLED=0`'s pure-Go DNS resolver vs. glibc's `getaddrinfo`) — not a bug in this repo, but a real "have you hit this before" interview topic given the Alpine + Go combination. |
| **Common mistakes** | Not pinning base image versions (`alpine:latest`) — avoided here (`alpine:3.19` pinned explicitly), preventing surprise breakage/behavior drift from an upstream `latest` tag changing underneath a rebuild. The version-drift issue noted earlier (Dockerfiles pin `golang:1.22-alpine`, `go.mod`/`USER_GUIDE.md` state Go 1.25) is worth restating here as a build-reproducibility concern: whatever `go.mod`'s `go` directive states as its minimum version, if it's newer than 1.22, **the Docker build would fail outright** (older toolchain refusing a newer minimum-version directive) — a very concretely testable claim (`docker build -f Dockerfile.api .` would either fail or Ignore the mismatch depending on exact versions) and a good "let's actually check this" moment in an interview. |
| **Performance implications** | Smaller final images pull faster on pod scheduling (relevant for KEDA's scale-from-zero worker pods — faster image pull = faster time-to-first-job after a cold start) and have a smaller attack surface to scan. |
| **Production considerations** | `ldflags="-s -w"` (both Dockerfiles + `Makefile`) strips debug symbols and DWARF info — smaller binary, but means a production panic's stack trace has less symbol information for post-mortem debugging (a real, standard tradeoff between binary size and debuggability). |
| **Security considerations** | Non-root user, minimal package set, static linking (no dynamic library CVE surface at runtime beyond what's `apk add`-ed) — solid baseline container hardening. No explicit `USER` numeric UID (uses named user `pulsegrid` from `adduser -S`, which Alpine assigns a system UID to, typically in a safe non-privileged range) — fine, though some hardened-image guides prefer explicit numeric UIDs for Kubernetes `securityContext.runAsNonRoot` verification without needing to resolve the image's `/etc/passwd`. |
| **Scalability considerations** | Image size directly affects cold-start latency at scale-from-zero (KEDA `minReplicaCount: 0`) — smaller is better here specifically because of the 0-30 scaling model. |
| **Failure scenarios** | A build-time `go build` failure (e.g., the Go-version mismatch above) fails the entire CI pipeline's `docker` job (`.github/workflows/ci.yml`'s `docker` job depends on `build`, which would already have failed first if the version mismatch caused a compile error) — the layered `needs: [lint, test]` / `needs: [build]` dependency chain (`.github/workflows/ci.yml:42,55`) means a broken build never reaches the image-push stage, a reasonable safety property regardless of this specific version concern. |
| **How this repo implements it** | Two-stage builds, Alpine 3.19 runtime, `CGO_ENABLED=0` static compilation, non-root user, stripped binaries. |
| **Relevant files** | `Dockerfile.api`, `Dockerfile.worker`, `Makefile:74-87`, `.github/workflows/ci.yml:40-96` |

### Kubernetes, KEDA & Autoscaling

| | |
|---|---|
| **What it is** | Deployments (API: fixed 2 replicas; worker: KEDA-managed 0-30), a `ScaledObject` custom resource (KEDA's autoscaling primitive), RBAC (ServiceAccounts + Roles + RoleBindings), a ConfigMap + Secret for configuration/credentials. No Helm charts anywhere in this repo — plain YAML manifests applied directly via `kubectl apply -f`. |
| **Why KEDA specifically (not HPA)** | Kubernetes' built-in Horizontal Pod Autoscaler (HPA) scales on CPU/memory utilization by default, or on a `metrics.k8s.io`/custom-metrics-API-exposed value — it has no native understanding of "Kafka consumer group lag." KEDA is purpose-built to bridge external event sources (Kafka, SQS, RabbitMQ, and dozens more) into the HPA mechanism, translating "lag on topic X for consumer group Y" into a scaling decision. This is exactly the signal that matters for PulseGrid's worker tier — CPU utilization is a lagging, indirect proxy for "how much work is queued," while consumer lag is the direct signal. |
| **Why min=0** | Scale-to-zero during idle periods is impossible with plain HPA (HPA's `minReplicas` floor is meaningful but doesn't include the "scale from literally zero pods back up on new signal" cold-start mechanic KEDA provides) — KEDA's Kafka scaler can watch lag even with zero active pods (it polls the broker directly, not through the pods themselves) and trigger the first pod's creation, closing the loop from "system fully idle, costing nothing in worker compute" to "job arrives, worker scales up" without any pod running to *notice* the job arrived. |
| **Why RBAC as configured** | API's Role grants `get/list/watch` on `configmaps`/`secrets` and `get/list` on `pods` (`kube/rbac.yaml:29-35`) — the `pods` access is a little unusual for a stateless HTTP API and worth asking about in an interview (what does the API actually *do* with pod-list access? nothing visible in `cmd/api/main.go` — this may be vestigial/over-provisioned access, a real least-privilege smell). Worker's Role is narrower — only `configmaps`/`secrets` (`kube/rbac.yaml:44-48`) — no pod access, appropriately scoped to "read my own config" only. |
| **Alternatives** | Plain HPA + a custom-metrics-adapter exposing Kafka lag as a Kubernetes custom metric (e.g., via `prometheus-adapter` reading the very `pulsegrid_queue_depth_jobs` gauge this repo already emits) — would work, but KEDA is purpose-built for exactly this and handles the scale-to-zero case HPA fundamentally cannot. Helm — not used; plain manifests are simpler to read/audit for a small, stable manifest set, at the cost of no templating (every environment-specific value is either hardcoded in the YAML or supplied via the ConfigMap/Secret, not parameterized at the chart level). |
| **Tradeoffs** | No Helm means no built-in versioned-release/rollback tracking at the Kubernetes-tooling level (Helm's `helm rollback` concept) — rollback here would rely on `kubectl rollout undo` (Deployment-native, works fine for the Deployments themselves) or re-applying a previous manifest version from git, both perfectly workable but less turnkey than Helm's release history. |
| **Common mistakes** | Worker taint/toleration mismatch (discussed under EKS above) — worth re-flagging here as a Kubernetes-scheduling-specific concern: a `NoSchedule` taint with no corresponding `tolerations` block in the pod spec means the scheduler will refuse to place that pod on the tainted nodes at all, a very common "why are my pods stuck Pending" root cause. |
| **Performance implications** | `pollingInterval: 15`/`cooldownPeriod: 300` (discussed under Backpressure above) — the asymmetric scale-up/scale-down cadence. |
| **Production considerations** | `terminationGracePeriodSeconds: 1800` on the worker (`kube/worker-deployment.yaml:25`) — critical for KEDA scale-down not to kill an in-flight 25-minute transcode; readiness/liveness probes differ meaningfully between tiers: API's liveness/readiness both hit `/health` (which actively checks Postgres/Kafka/S3, `cmd/api/main.go:159-222`) — meaning a downstream outage can mark the API pod `NotReady` (removed from Service load-balancing) or even `Unhealthy` (restarted) even though the API process itself is fine, a real "does a dependency outage cause unnecessary pod churn" question worth considering; worker's liveness is a bare `tcpSocket` check on the metrics port (`kube/worker-deployment.yaml:50-56`, weaker than API's — doesn't verify the worker is actually able to reach Kafka/S3, only that *a* TCP listener exists on 8081), and its readiness is an HTTP GET on `/metrics` (line 57-64) — neither actually gates on job-processing health, only on the metrics server being alive, which could be true even if the main polling loop has somehow wedged. |
| **Security considerations** | RBAC scoping discussed above; Secrets delivered via `envFrom: secretRef` land in every container's environment variables — visible via `kubectl exec ... env` or a process listing on the node, a real (if standard-for-k8s-Secrets) exposure surface vs. mounting Secrets as files with tighter filesystem permissions (also visible, but at least not in `/proc/[pid]/environ` for every process to read). |
| **Scalability considerations** | The node-group-vs-KEDA-maxReplicaCount mismatch flagged under EKS applies here directly — KEDA's `maxReplicaCount: 30` is aspirational without either a larger/autoscaling node group or a reconciled lower KEDA max. |
| **Failure scenarios** | Worker liveness probe passing (metrics server up) while the polling loop is wedged (e.g., blocked indefinitely on a misbehaving `FetchMessage` call that somehow ignores its context timeout) would leave a "zombie" worker pod — alive per Kubernetes, not actually processing jobs, invisible to both the liveness probe and (initially) to consumer-lag-based alerting until the session timeout (30 min) eventually evicts it from the consumer group. |
| **How this repo implements it** | Standard Deployment + Service + ConfigMap + Secret + RBAC manifests; KEDA `ScaledObject` CRD for the worker tier only (API scales via a fixed replica count, not KEDA — appropriate since request-driven HTTP load doesn't have a "lag" signal the way a queue consumer does; a more complete setup might use HPA-on-CPU for the API tier, not present here — API replica count is static at 2, with no autoscaling configured for it at all). |
| **Relevant files** | `kube/*.yaml` |

### CI/CD (GitHub Actions)

| | |
|---|---|
| **What it is** | A four-job pipeline: `lint` (go vet) → `test` (race-enabled + coverage) → `build` (compile both binaries) → `docker` (build+push images, main-branch-push only). |
| **Why this staged approach** | `needs: [lint, test]` on `build`, `needs: [build]` on `docker` (`.github/workflows/ci.yml:42,55`) — a strict linear gate: nothing gets built if lint/tests fail, nothing gets pushed as a container image if the build fails. This is the textbook "fail fast, fail cheap" CI structure — a lint failure (seconds to run) blocks before the more expensive test run, which blocks before the build, which blocks before the most expensive/consequential step (pushing an image to a registry other things might pull). |
| **Why `docker` is gated to `main` push only** | `if: github.event_name == 'push' && github.ref == 'refs/heads/main'` (`.github/workflows/ci.yml:56`) — PRs run lint/test/build (verifying the change compiles and passes tests) but never push images, preventing untrusted/unmerged code from ever producing a deployable artifact, and avoiding registry clutter from every PR's throwaway commits. |
| **Alternatives** | GitLab CI, CircleCI, Jenkins — functionally equivalent staged-pipeline concepts; GitHub Actions is the natural choice given the repo is (evidently) GitHub-hosted, with zero additional infrastructure to run the pipeline (GitHub-hosted runners). |
| **Tradeoffs** | `go vet` alone (no `golangci-lint` or similar aggregated linter) is a fairly shallow lint gate — catches genuine correctness issues (suspicious constructs, printf-format mismatches, unreachable code) but not style/complexity/security-lint categories a more comprehensive linter would (e.g., `gosec` for security-pattern scanning, notably absent — see [Supply Chain](#supply-chain--dependency-security) below). |
| **Common mistakes** | Not running tests with `-race` in CI — avoided here (`go test ./... -v -race -count=1`, line 31) — this is genuinely important given the goroutine/atomic-flag patterns in the worker; without `-race` in CI, a data race could sit undetected in local dev (where it may never manifest) until a production incident. |
| **Performance implications** | `cache-from: type=gha` / `cache-to: type=gha,mode=max` on the Docker build steps (`.github/workflows/ci.yml:82-83,94-95`) — leverages GitHub Actions' native layer caching, meaningfully speeding up repeated builds where only application-layer code (not `go.mod`/`go.sum`) changed, since the `COPY go.mod go.sum ./` + `go mod download` layer (`Dockerfile.api:15-16`) is cached independently of source changes. |
| **Production considerations** | Images are tagged both `${{ github.sha }}` and `latest` (`.github/workflows/ci.yml:79-81,90-92`) — the SHA tag gives an immutable, traceable-to-exact-commit deployable artifact (what you'd actually reference in a Kubernetes manifest for reproducible deploys), while `latest` is a convenience/moving-target tag; **`kube/*-deployment.yaml` currently reference `image: pulsegrid/api:latest`/`pulsegrid/worker:latest`** (`kube/api-deployment.yaml:49`, `kube/worker-deployment.yaml:28`), not a pinned SHA — meaning a rolling restart of an existing Deployment (without a new manifest apply) would **not** pick up a newer `latest` push (Kubernetes doesn't re-pull `latest` on its own without `imagePullPolicy: Always` set, which isn't visible in these manifests, defaulting instead to `IfNotPresent` for non-`latest` tags but actually defaulting to `Always` specifically *for* the `latest` tag per Kubernetes' documented default behavior — worth knowing this specific Kubernetes nuance) — a real, checkable deployment-reproducibility gap: using `:latest` in production manifests is a widely-known anti-pattern for exactly the ambiguity it creates about "what code is actually running right now." |
| **Security considerations** | `permissions: contents: read, packages: write` scoped explicitly on the `docker` job (`.github/workflows/ci.yml:57-59`) — good least-privilege practice for the GITHUB_TOKEN used to authenticate to `ghcr.io`, rather than relying on a broader default token scope. |
| **Scalability considerations** | N/A — a build-time concern. |
| **Failure scenarios** | A flaky test (network-dependent, timing-sensitive) failing intermittently in CI would block legitimate merges — no retry-on-failure or flaky-test-quarantine mechanism visible in the workflow. |
| **How this repo implements it** | Four sequential/dependent jobs, GitHub-hosted Ubuntu runners, `actions/setup-go@v5` pinned to Go 1.22 (matching the Dockerfiles, **not** matching the "Go 1.25" stated elsewhere — the version-drift issue triangulated across three sources: `.github/workflows/ci.yml:10`, `Dockerfile.api:8`/`Dockerfile.worker:8`, and `USER_GUIDE.md`'s tech stack table — a genuinely good "trace inconsistencies across the repo" exercise). |
| **Relevant files** | `.github/workflows/ci.yml`, `Makefile:96-97` (`ci` target mirrors the pipeline for local runs) |

### Observability — Prometheus, Grafana, Health Checks

| | |
|---|---|
| **What it is** | Two `/metrics` endpoints (API `:8080`, worker `:8081`) in Prometheus exposition format; a pre-built Grafana dashboard JSON; five `PrometheusRule` alert rules; `GET /health` for active dependency checking. No distributed tracing anywhere in the repo (no OpenTelemetry/Jaeger/X-Ray integration — confirmed absent). |
| **Why this three-pillar (partial) approach** | Metrics (Prometheus) answer "how much/how often" cheaply at scale; logs (structured zap JSON, see [Part 1](#structured-logging--graceful-shutdown)) answer "what exactly happened" for a specific event; traces would answer "where did time go across service boundaries for *this specific request*" — the third pillar's absence means there's no way to see a single job's full cross-service timeline (API submission → Kafka publish → worker pickup → S3 download → transcode → S3 upload) as one correlated trace; today, reconstructing that requires manually correlating `job_id` across API logs, worker logs, and metrics by timestamp — possible, but much more manual than a trace view would provide. |
| **Why these specific metrics** | Each metric maps to a specific operational question: `pulsegrid_jobs_submitted_total`/`pulsegrid_job_completed_total` (throughput), `pulsegrid_upload_duration_seconds`/`pulsegrid_transcode_duration_seconds` (latency, with the transcode histogram usefully labeled by `rendition` — letting you see if 1080p transcodes are disproportionately slow vs. 480p), `pulsegrid_queue_depth_jobs` (backlog/backpressure signal), `pulsegrid_transcode_failures_total{error_type}` (failure-mode breakdown, directly usable to see if permanent-vs-retryable-vs-constraint failures are trending up), `pulsegrid_pod_resource_constrained` (capacity-sizing signal). This is a well-chosen, minimal metric set that covers the golden-signals-adjacent bases (throughput, latency, errors — missing explicit "saturation," though queue depth is a reasonable proxy). |
| **Alternatives** | OpenTelemetry (traces + metrics + logs in one SDK/collector pipeline) — would unify all three pillars and add the missing tracing dimension, at real integration cost (instrumenting every I/O call site with spans, running/managing a collector). Given this repo's current single-hop-at-a-time processing model (no complex fan-out per request), tracing's marginal value is real but not as dramatic as in a deeply fan-out microservices architecture — a fair, defensible reason it might not have been prioritized yet, distinct from "it was simply forgotten." |
| **Tradeoffs** | The `DLQBacklog` alert referencing a never-populated metric (`pulsegrid_queue_depth_dlq_jobs`, flagged earlier under DLQ) is the standout example of "alert rules and actual instrumentation drifting apart" — a very real production-observability failure mode: an alert that looks like coverage but provides none. |
| **Common mistakes** | Unbounded label cardinality on a metric (e.g., labeling by `job_id`, which is unbounded/unique-per-request and would blow up Prometheus' memory) — correctly avoided here; the only labeled metrics use bounded label sets (`rendition` — a handful of known values; `error_type` — exactly three possible values). This is a real, easy-to-get-wrong Prometheus best practice this repo gets right. |
| **Performance implications** | Prometheus histograms pre-declare bucket boundaries (`DefaultHistogramBuckets`, `TranscodeDurationBuckets`, `pkg/metrics.go:24,27`) — chosen sensibly (upload buckets span sub-second to 5 minutes; transcode buckets span 1 second to 30 minutes, matching `DefaultTranscodeTimeout`) — bucket choice directly affects both metric storage cost (more buckets = more time series) and the precision of derived percentile queries (`histogram_quantile`, used in `HighP99Latency`'s alert expression, `kube/prometheus-rules.yaml:36-37`). |
| **Production considerations** | `GET /health`'s 5-second per-check timeout (`cmd/api/main.go:165`) bounds worst-case health-check latency to ~15s if all three checks (Postgres/Kafka/S3) are independently slow — actually, they run **sequentially**, not concurrently (`cmd/api/main.go:172-205` is a straight-line sequence of three `if`/`Ping` blocks, no goroutines/`errgroup`) — meaning a slow Postgres check delays the Kafka and S3 checks from even starting, a real "this could be parallelized with `errgroup` to cut worst-case health-check latency by up to 3x" improvement, good to spot and suggest in an interview. |
| **Security considerations** | `/metrics` endpoints are unauthenticated on both API and worker — standard for internal-cluster-only Prometheus scraping (protected by network boundary, not app-level auth), but worth naming as a deliberate reliance on network-level trust rather than defense-in-depth. |
| **Scalability considerations** | Metrics cardinality stays bounded regardless of job volume (discussed above) — this design scales cleanly to arbitrary job throughput without Prometheus storage growing unboundedly. |
| **Failure scenarios** | A Prometheus scrape failure (pod unreachable, metrics endpoint down) is only indirectly detectable (via Prometheus' own `up` meta-metric, not surfaced in any alert rule reviewed here) — combined with the DLQ-metric gap, this repo's alerting has real, specific holes despite a generally well-designed metric set. |
| **How this repo implements it** | `promhttp.Handler()` mounted at `/metrics` on both binaries; `pkg.Metrics`/`pkg.WorkerMetrics` structs registering typed Prometheus collectors; `kube/grafana-dashboard.json` for visualization; `kube/prometheus-rules.yaml` for alerting; `GET /health` for active (not just passive-metric-based) dependency verification. |
| **Relevant files** | `pkg/metrics.go`, `cmd/api/main.go:159-222`, `kube/grafana-dashboard.json`, `kube/prometheus-rules.yaml` |

## Part 6 — API Design

### REST, Status Codes & Error Responses

| | |
|---|---|
| **What it is** | Four resource-ish endpoints (`POST /videos/upload`, `GET /jobs/{id}`, `GET /jobs`, `GET /health`) plus `/metrics`; a consistent `ErrorResponse` JSON shape for all non-2xx responses. |
| **Why this approach** | `202 Accepted` (not `201 Created`) on successful upload (`cmd/api/main.go:378`) is a deliberately precise status-code choice: `201` implies the resource (the *transcoded output*) now exists at a location; `202` correctly signals "request accepted for asynchronous processing, not yet complete" — exactly this system's semantics (the response includes a `status_uri` to poll, not a completed-resource URI). This is a genuinely good, easy-to-get-wrong REST detail. |
| **Status code choices, evaluated** | `400` for validation errors (missing fields, bad JSON, invalid query params) — correct. `413 Payload Too Large` for the 10GB cap — correct and specific (not a generic `400`). `404` for unknown `job_id` — correct. `503 Service Unavailable` when `DATABASE_URL` isn't configured (`handleListJobs`/`handleGetJob`, `cmd/api/main.go:475,531`) — arguably correct (the *service* — DB-backed job querying — really is unavailable, distinct from a `500` which would imply an unexpected failure rather than an expected, known configuration state) though a case could be made for `501 Not Implemented` instead, depending on how you frame "not configured" vs. "not supported." `500` for genuine internal failures (S3 upload failure, DB query error) — correct, generic fallback. |
| **Alternatives** | A formal API spec (OpenAPI/Swagger) generated from or validated against the handler code — would give you a machine-checkable contract, auto-generated client SDKs, and request/response validation middleware; this repo has none (no `openapi.yaml`/`swagger.json` in the reviewed files) — `docs/api.md` is the only spec, and it's hand-written prose/markdown, meaning it can drift from the actual handler behavior with no automated check catching the drift (a real, common documentation-rot risk). |
| **Tradeoffs** | Hand-written JSON response structs (`ErrorResponse`, `UploadResponse`, etc.) give full control with zero framework dependency, at the cost of every field name / status code choice being manually kept consistent across handlers — there's no shared "APIError" middleware auto-generating these; `writeError`/`writeErrorWithRequestID` (`cmd/api/main.go:602-620`) are the closest thing to that consistency mechanism, and they *are* used consistently across all handlers, a good sign of discipline even without framework enforcement. |
| **Common mistakes** | Returning `200` for an error (forces clients to parse the body to detect failure) — not present, all error paths use appropriate non-2xx codes. Inconsistent error body shape across endpoints — not present, `ErrorResponse` is used uniformly. A real, present gap: `handleHealth`'s `503` response body (`HealthResponse`) has a **different shape** than `ErrorResponse` (no `error_code`/`request_id`/`timestamp` fields, just `status`+`checks`) — a client written to generically parse "any non-2xx as `ErrorResponse`" would fail to parse a `503` from `/health` correctly. Minor, but a real API-consistency gap worth naming. |
| **Performance implications** | N/A beyond general HTTP overhead. |
| **Production considerations** | `request_id` on every error response (`cmd/api/main.go:44,613`) is generated fresh per-request via `pkg.GenerateJobID()` — useful for correlating a client-reported error with server-side logs, **except this `request_id` is never actually logged anywhere on the server side** (no `log.Printf` or structured log call includes it) — meaning the correlation ID exists in the response but has no corresponding server-side breadcrumb to correlate *against*, undermining its main purpose. Another good "spot the incomplete feature" finding. |
| **Security considerations** | See [Error Handling](#error-handling--error-wrapping) — raw internal error text (`err.Error()`) is surfaced in the `detail` field on some paths, a mild information-disclosure concern for a public-facing deployment. |
| **Scalability considerations** | N/A. |
| **Failure scenarios** | N/A beyond what's covered under specific endpoints. |
| **How this repo implements it** | Manual `http.ServeMux` routing, hand-written response structs, a small set of shared error-writing helpers. Full endpoint-by-endpoint reference: `docs/api.md` / `USER_GUIDE.md`'s API Documentation section (not duplicated here). |
| **Relevant files** | `cmd/api/main.go`, `docs/api.md` |

### Input Validation, Idempotency-at-the-API-Layer & Versioning

| | |
|---|---|
| **What it is** | Layered validation: `http.MaxBytesReader` (size), multipart form parsing, required-field checks, JSON structural validation on `renditions`, query-parameter parsing/bounds-checking on `GET /jobs`. No API versioning of any kind (no `/v1/` prefix, no `Accept` header version negotiation, no version field in any payload). |
| **Why this approach** | Validation happens as early and as cheaply as possible at each layer: size is enforced by `MaxBytesReader` before any parsing happens at all (rejects oversized bodies mid-stream, not after fully receiving them) — `handleVideoUpload`'s ordering is deliberate: `MaxBytesReader` wraps the body (line 240) *before* `ParseMultipartForm` is called (line 243), so a too-large upload is cut off during parsing, not after. |
| **Query-parameter validation specifics** | `limit` is bounded `[1, 1000]` (`cmd/api/main.go:450-459`), `offset` bounded `≥0` (`cmd/api/main.go:465-470`), `status` values checked against a fixed allow-list (`validJobStatuses`, `cmd/api/main.go:391-396,438-444`), timestamp range checked for logical consistency (`submitted_after` must precede `submitted_before`, `cmd/api/main.go:429-432`) — this is genuinely thorough input validation, worth citing as a strength: every user-controllable query parameter has an explicit bound or allow-list, not just a type check. |
| **Why no versioning** | For a two-consumer system (the API's only "clients" today are: end users/services calling it directly, and `cmd/load-test`'s harness) with a single deploying team controlling both API and its consumers, versioning overhead (maintaining `/v1/` and `/v2/` in parallel, deprecation policies) isn't yet paying for itself — a defensible "not needed at current scale" omission rather than an oversight, though it's a real gap the moment any external, independently-deployed client starts depending on this API's exact shape. |
| **Alternatives** | URL-path versioning (`/v1/videos/upload`) — simplest, most visible, what most public APIs do; header-based versioning (`Accept: application/vnd.pulsegrid.v1+json`) — cleaner URLs, less discoverable/harder to test with a browser/curl casually; no versioning + strict backward-compatible-only evolution (additive fields only, never remove/rename) — what this repo implicitly relies on today without ever stating it as policy. |
| **Tradeoffs** | Skipping versioning now means the *first* breaking change (e.g., renaming `output_files` or changing `status` enum values) has no migration path — every consumer breaks simultaneously on deploy, with no ability to run old/new API shapes side-by-side during a transition. |
| **Common mistakes** | Trusting `Content-Length` alone for size enforcement (a client can lie about it) — correctly avoided: `MaxBytesReader` enforces the limit by actually counting bytes read, not by trusting a header, with `header.Size` (`cmd/api/main.go:269`) checked only as a *secondary* sanity check, explicitly commented as such ("secondary check — MaxBytesReader is primary," line 268). |
| **Performance implications** | Validating before expensive work (S3 upload, DB write) avoids wasted I/O on a request that's going to be rejected anyway — the ordering in `handleVideoUpload` (validate `source_name` → validate `video` file presence → check size → validate `renditions` JSON → *then* generate job ID and upload to S3) is correctly validation-before-side-effects throughout. |
| **Production considerations** | No idempotency-key support on `POST /videos/upload` — a client retrying a timed-out request (e.g., their own client-side timeout expired even though the server eventually succeeded) has no way to signal "this is a retry of the same logical request, don't create a second job" — each retry creates a genuinely new job with a new `job_id`, a real, if fairly standard-for-this-kind-of-API, gap (idempotency keys are a well-known pattern — e.g., Stripe's `Idempotency-Key` header — not implemented here). |
| **Security considerations** | Structural JSON validation on `renditions` (checking `id`/`type` presence) doesn't validate *semantic* correctness (e.g., a nonsensical `VideoBitrate: "999999999M"` would pass validation and be handed directly to `ffmpeg` as a CLI argument) — see [ffmpeg command construction](#ffmpeg-pipeline--command-construction) below for why this specific gap matters more than it might first appear. |
| **Scalability considerations** | N/A directly. |
| **Failure scenarios** | A client double-submitting the same video (e.g., a buggy retry loop, or a user double-clicking "upload") produces two independent jobs, two S3 source uploads, two full transcode runs — no deduplication anywhere in the pipeline. |
| **How this repo implements it** | See above — validation is thorough at the *structural* level, absent at the *semantic*/business-logic level for rendition parameters, and there's no idempotency-key or versioning mechanism anywhere. |
| **Relevant files** | `cmd/api/main.go:224-280,398-472,567-591` |

## Part 7 — ffmpeg Pipeline

### Transcoding, HLS, Manifests & Storage Layout

| | |
|---|---|
| **What it is** | The worker shells out to `ffmpeg` as an external subprocess (`os/exec`) for both single-rendition MP4 transcoding and HLS (HTTP Live Streaming) segment+playlist generation; a `manifest.json` summarizing all outputs is generated and uploaded alongside them. |
| **Why shell out to ffmpeg rather than a Go video library** | ffmpeg is *the* industry-standard, battle-tested, format-and-codec-exhaustive transcoding engine — no Go-native library comes close to its codec/container coverage, hardware-acceleration support, or performance (it's a highly optimized C/assembly codebase built over two decades). Shelling out via `exec.Command` is the standard, correct way to use it from a language that isn't C — this is not a shortcut, it's the industry-normal integration pattern (nearly every video platform's backend does exactly this, whether from Go, Python, Node, or Java). |
| **Why two separate transcode functions (`TranscodeSingleRendition` vs. `TranscodeHLS`)** | MP4 rendition output and HLS segment/playlist output have meaningfully different ffmpeg invocation shapes (`-c:v`/`-b:v`/`-s`/`-c:a`/`-b:a` args producing one output file, vs. `-f hls -hls_time -hls_segment_filename -hls_list_size 0` producing a playlist + N segment files) and different result types (`TranscodeResult` — one file, one size, one duration — vs. `HLSResult` — a playlist path, a segment count, a segment-path list) — separating them keeps each function's argument-building and result-parsing logic simple and directly matched to its output shape, rather than one function branching internally on a `rendition.Type == "hls_segments"` flag throughout. |
| **ffmpeg command construction — a real security-adjacent finding** | Both `TranscodeSingleRendition` (`pkg/transcode.go:40-58`) and `TranscodeHLS` (`pkg/transcode.go:123-136`) build the `ffmpeg` argument list by directly interpolating **client-supplied rendition fields** (`rendition.VideoCodec`, `rendition.VideoBitrate`, `rendition.Resolution`, `rendition.AudioCodec`, `rendition.AudioBitrate`, `rendition.BaseResolution`) as individual `exec.Command` arguments. Because `exec.CommandContext(ctx, "ffmpeg", args...)` passes each string as a **separate argv element** (not through a shell), this is **not** classic shell-injection-via-string-concatenation (there's no `sh -c "ffmpeg ... " + userInput` pattern here, which would be trivially exploitable) — Go's `exec.Command` with an args slice does not invoke a shell, so metacharacters like `;`, `|`, `` ` `` in a rendition field are passed literally to ffmpeg as a single argument, not interpreted by a shell. **However**, this does **not** mean the input is safe: a client can still supply an arbitrary string as, say, `VideoCodec`, and ffmpeg itself interprets some of its own arguments in ways that can be abused — for example, ffmpeg's `-i` (or, relevantly, filter/output-related flags) has historically had documented cases where crafted input to certain options can trigger unexpected file reads/writes or resource exhaustion (ffmpeg's own CVE history includes such issues). Because `parseRenditions` only validates that `id` or `type` is non-empty (`cmd/api/main.go:583-587`) and never validates that `VideoCodec` is one of an allow-listed set of known-safe codec names, or that `Resolution` matches a `WIDTHxHEIGHT` numeric pattern, or that bitrate strings match ffmpeg's expected `[0-9]+[kKmM]?` format — **arbitrary attacker-controlled strings reach the ffmpeg argv directly**. This is a real, valuable finding: shelling out safely (no shell metacharacter risk) is necessary but not sufficient — the *arguments themselves*, even passed safely, still need allow-list validation against the receiving program's expected grammar. |
| **Alternatives** | Allow-list validation on every rendition field before it ever reaches `exec.Command` (e.g., `VideoCodec` checked against `{"libx264", "libx265", "vp9"}`, `Resolution` checked against a `^\d{2,5}x\d{2,5}$` regex, bitrate checked against `^\d+[kKmM]$`) — the correct fix, straightforward to add, not present today. A stricter alternative still: don't accept free-form codec/bitrate strings from clients at all — accept only a `rendition_preset` enum (`"720p_h264"`, `"1080p_h265"`, etc.) mapped server-side to a fixed, known-safe ffmpeg argument set, eliminating this entire class of risk by construction (this is closer to what `defaultRenditions()`, `cmd/api/main.go:594-600`, already implicitly is — the gap is only in the *custom* renditions path). |
| **Tradeoffs** | Free-form rendition specs (current design) give clients maximum flexibility (any codec/bitrate/resolution ffmpeg supports) at the cost of the input-validation gap above; a preset-only model closes the gap but limits client flexibility to whatever presets are pre-defined server-side. |
| **Common mistakes** | Not setting a timeout on a long-running subprocess — avoided here (`DefaultTranscodeTimeout = 30 * time.Minute`, enforced via `context.WithTimeout` wrapping `exec.CommandContext`, `pkg/transcode.go:61-64,139-142`) — without this, a hung/looping ffmpeg process (malformed input triggering an edge case, or a deliberately crafted "decompression bomb"-style input designed to make ffmpeg spin) would hold a worker pod (and its Kafka partition, per the earlier discussion) indefinitely. Not capturing/bounding subprocess output — avoided via the 500-char stderr truncation discussed earlier, though `CombinedOutput()`'s unbounded *collection* (before truncation) remains a latent memory risk as noted in [Memory](#memory-escape-analysis--garbage-collection). |
| **Performance implications** | ffmpeg's own internal multi-threading (codec-dependent, e.g., libx264 uses multiple threads by default) means a single `TranscodeSingleRendition` call can already saturate multiple CPU cores — this is why the worker pod's resource request is `cpu: "2"` / limit `"4"` (`kube/worker-deployment.yaml:44,48`), sized for ffmpeg's own parallelism, not for Go-level goroutine concurrency (reinforcing the earlier point that this system's concurrency model is pod-level, not intra-process). |
| **Production considerations** | `parseDuration`'s regex (`durationRegex`, `pkg/transcode.go:190`) parses ffmpeg's human-readable `Duration: HH:MM:SS.xx` line from its stderr/stdout output — this is inherently **fragile**: it depends on ffmpeg's exact log-line format not changing across ffmpeg versions (a real "works today, silently breaks on an ffmpeg upgrade" risk), versus using ffprobe's structured `-print_format json` output (a more robust, purpose-built alternative for extracting exactly this kind of metadata) — not used here. |
| **Security considerations** | Beyond the argument-injection-adjacent finding above: the worker downloads and processes **arbitrary client-uploaded video files** as input to a large, historically CVE-rich C codebase (ffmpeg) — running ffmpeg as a non-root container user (`Dockerfile.worker`) is the primary containment measure; there's no additional sandboxing (no seccomp profile, no gVisor/Kata, no separate untrusted-input-processing namespace) visible in the reviewed Kubernetes manifests, meaning a hypothetical ffmpeg RCE vulnerability triggered by a malicious input video would have the blast radius of "arbitrary code execution as the `pulsegrid` non-root user inside the worker container" — contained by the container boundary and non-root user, but not further sandboxed. |
| **Scalability considerations** | Transcode duration is the dominant per-job cost driver and the primary input to capacity planning (`TranscodeDurationBuckets` spans up to 1800s/30min, matching the timeout) — worker pod count × (1/avg transcode duration) is the throughput model discussed earlier. |
| **Failure scenarios** | ffmpeg exits non-zero → `TranscodingError` with truncated stderr → classified via `ClassifyError`'s substring matching (e.g., "unsupported codec" → permanent → DLQ; a generic crash → retryable → re-enqueue up to 3x). A subtle failure mode: **a successful ffmpeg exit (code 0) with zero actual output** — `TranscodeHLS` explicitly guards against this ("no segments found after successful ffmpeg run," `pkg/transcode.go:174-179`), a good defensive check; `TranscodeSingleRendition` similarly verifies the output file exists via `os.Stat` after the ffmpeg call (`pkg/transcode.go:83-90`) before declaring success — both functions correctly treat "process exited 0" as necessary but not sufficient for success. |
| **Storage layout** | `s3://{bucket}/{jobID}/original.mp4` (source); `s3://pulsegrid-output/{jobID}/{renditionID}/{filename}` for MP4 renditions and HLS playlist+segments; `s3://pulsegrid-output/{jobID}/manifest.json` (`pkg/s3client.go:280-338`) — a clean, predictable, job-scoped prefix structure that makes both the idempotent-overwrite property (discussed earlier) and straightforward manual/CLI browsing of a job's outputs (`aws s3 ls s3://pulsegrid-output/{jobID}/ --recursive`) trivial. |
| **How this repo implements it** | `pkg/transcode.go` (single-rendition + HLS), `pkg/manifest.go` (manifest generation + ffmpeg-version detection via `ffmpeg -version`), `pkg/s3client.go`'s `UploadOutputsToS3` (storage layout). |
| **Relevant files** | `pkg/transcode.go`, `pkg/manifest.go`, `pkg/s3client.go:270-349`, `pkg/transcode_test.go`, `pkg/manifest_test.go` |

## Part 9 — Security

### Secrets & Credential Management

| | |
|---|---|
| **What it is** | `kube/secrets.yaml` — a plain Kubernetes `Secret` holding `DATABASE_URL` (with embedded password), `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`, and Kafka credentials (if any — the ConfigMap covers broker addresses, not credentials, suggesting the in-cluster Kafka may not require auth at all, itself worth flagging). |
| **Why this is a real, notable weakness** | Kubernetes `Secret` objects are base64-*encoded*, not encrypted, by default — base64 is trivially reversible, not a security control. At-rest encryption of Secrets in etcd requires separately configuring KMS-backed envelope encryption on the EKS control plane, which is **not evidenced anywhere in `terraform/main.tf`** (no `encryption_config` block on `aws_eks_cluster.main`). Combined with static, long-lived AWS credentials (discussed in depth under [IAM & IRSA](#iam--irsa)) and a DB password embedded directly in a connection-string env var (readable by anything with `kubectl exec` access or node-level process-list visibility), this is the weakest link in the whole system's security posture. |
| **Why this approach was (presumably) taken** | Simplicity — `envFrom: secretRef` is the path of least resistance for getting credentials into a container's environment, and it's a extremely common pattern precisely because it's the *default*, low-effort option, not because it's the most secure one. |
| **Alternatives** | AWS Secrets Manager or Parameter Store + the AWS Secrets and Configuration Provider (ASCP) for the Kubernetes Secrets Store CSI Driver — mounts secrets as files (not env vars), supports automatic rotation, and centralizes audit logging (CloudTrail) of every secret access. Sealed Secrets / External Secrets Operator — encrypts secrets at rest in git (for GitOps workflows) and/or syncs from an external secret store into native k8s Secrets. HashiCorp Vault with the Vault Agent Injector — enterprise-grade dynamic-secret issuance (e.g., short-lived DB credentials minted per-pod, auto-expiring). Any of these would be a meaningful upgrade; none are implemented here. |
| **Tradeoffs** | Every alternative above adds operational complexity (a new system to run/trust) in exchange for meaningfully reduced blast radius on credential leak and easier rotation — a very standard "security posture vs. operational simplicity" tradeoff, and this repo has clearly chosen simplicity, appropriate for an early-stage/demo system, a real gap for anything handling production data. |
| **Common mistakes** | Committing secrets to git — `kube/secrets.yaml` as checked into this repo presumably holds placeholder/example values (not real credentials, per standard practice and `USER_GUIDE.md`'s note that "`kube/secrets.yaml` must have real values populated before applying") — worth explicitly confirming this file doesn't contain live credentials if this were a real audit, since a checked-in Secret manifest with real values is one of the most common real-world credential leaks. |
| **Performance implications** | N/A. |
| **Production considerations** | No credential rotation mechanism visible anywhere — static keys/passwords, presumably rotated manually and infrequently if at all, which is both an availability risk (a compromised credential stays valid indefinitely until someone notices and manually rotates) and a compliance gap for most real-world security standards (SOC 2, PCI-DSS, etc. typically mandate rotation cadences this setup can't automatically satisfy). |
| **Security considerations** | This whole section *is* the security consideration — see also [IAM & IRSA](#iam--irsa) for the AWS-credential-specific half of this problem. |
| **Scalability considerations** | N/A. |
| **Failure scenarios** | A leaked `DATABASE_URL` (logged accidentally, exposed via a debug endpoint, captured in a crash dump) grants full read/write access to the entire `jobs`/`job_status_events` tables indefinitely until manually rotated — no scoped, time-limited alternative exists in the current design. |
| **How this repo implements it** | Plain `Secret` + `envFrom: secretRef`, referenced identically by both API and worker deployments. |
| **Relevant files** | `kube/secrets.yaml`, `kube/api-deployment.yaml:57-58`, `kube/worker-deployment.yaml:36-37`, `terraform/main.tf` (absence of KMS/encryption_config) |

### Authentication & Authorization

| | |
|---|---|
| **What it is** | **There is none.** No API key, no OAuth/OIDC bearer token, no mTLS, no session cookie — every endpoint (`POST /videos/upload`, `GET /jobs/{id}`, `GET /jobs`) is fully open to anyone who can reach the API's network address. |
| **Why this is worth naming plainly, not softening** | This is the most significant security gap in the entire repository, more consequential in a real deployment than the secrets/IRSA gaps above, because it means **any client that can route to the API can upload arbitrary video files (up to 10GB, consuming real S3/compute cost) and read every job's status** (`GET /jobs` has no per-user/per-tenant scoping at all — any caller can list *all* jobs in the system, not just their own). There is no concept of "whose job is this" anywhere in the data model (`pkg.Job` has no `owner`/`user_id`/`tenant_id` field). |
| **Why this might be the current state** | Plausibly: PulseGrid is deployed today only behind a network boundary that itself provides the trust control (e.g., only reachable from an internal VPC, or fronted by something not shown in these manifests that handles auth upstream) — the `ClusterIP`-only Service type (no external LoadBalancer/Ingress, discussed under [CloudWatch & Cost](#cloudwatch-cost-optimization--why-some-aws-services-were-not-used)) supports this reading: today, the API literally isn't reachable from outside the cluster at all in this configuration, making app-level auth *currently* redundant with network-level isolation. This is a defensible interim state, but a **hard blocker** the moment any external entry point (ALB, Ingress, public LoadBalancer) is added — auth needs to land *before or simultaneously with* external exposure, not after. |
| **Alternatives** | API keys (simplest — a per-client static key checked against a header, stored hashed) — a good minimum bar. OAuth2/OIDC bearer tokens (if there's a broader identity provider to integrate with) — better for multi-tenant, revocable, scoped access. mTLS (if this is purely service-to-service, internal-only traffic) — strong identity without a token-issuance system, a good fit if PulseGrid is *only* ever called by other internal services, never end users directly. |
| **Tradeoffs** | Every auth mechanism adds a request-path cost (token validation, a lookup or crypto-verify) and an operational burden (key/token issuance and revocation) — genuinely justified the moment there's more than one trust domain touching this API, which is almost certainly already true (at minimum, "PulseGrid operators" vs. "PulseGrid's actual end users/clients" are different trust levels, and today both would have identical access). |
| **Common mistakes** | Adding authentication without authorization (verifying *who* the caller is, but not *what they're allowed to do*) — would be the natural next mistake if auth were bolted on naively (e.g., adding an API key check but still returning *all* jobs from `GET /jobs` regardless of which key was used) — a real design question worth having a clear answer to ("authn tells you who; authz — which this data model doesn't support at all yet, given no `owner` field — tells you what they can see/do; both are currently entirely absent, and adding one without the other only solves half the problem"). |
| **Performance implications** | N/A — not implemented. |
| **Production considerations** | This is a **blocking** production-readiness gap for any real-world (non-fully-air-gapped) deployment — worth stating unambiguously in an interview: "before this goes anywhere near being reachable outside a fully trusted internal network, it needs authentication, and given `GET /jobs` currently returns every job in the system to any caller, it needs authorization/multi-tenancy scoping too." |
| **Security considerations** | This entire section is a security consideration. |
| **Scalability considerations** | N/A directly, though multi-tenant scoping (once added) would need to be reflected in the `QueryJobs` SQL (an additional `WHERE owner = $N` clause) and indexed accordingly (a new index on an `owner`/`tenant_id` column) — a real schema-and-query change, not just an API-layer bolt-on. |
| **Failure scenarios** | Trivial denial-of-service: any anonymous caller can submit unlimited 10GB uploads, exhausting S3 storage budget, Kafka queue capacity, and worker compute (real AWS cost) with zero rate limiting or per-client accounting anywhere in the system. This is the single most severe, concretely exploitable weakness in the repository as reviewed. |
| **How this repo implements it** | Not implemented. |
| **Relevant files** | `cmd/api/main.go` (absence, throughout) |

### Least Privilege & Supply Chain Security

| | |
|---|---|
| **What it is** | Least privilege: granting each component only the permissions it strictly needs (RBAC Roles, IAM policies, container users). Supply chain security: trusting/verifying the provenance and integrity of third-party dependencies (Go modules, base container images, GitHub Actions). |
| **Where least privilege is done well** | Non-root container users (both Dockerfiles); RDS security group scoped to `eks_nodes` SG only, not `0.0.0.0/0` (`terraform/main.tf:268-274`); worker's RBAC Role limited to `configmaps`/`secrets` only, no `pods` access (`kube/rbac.yaml:38-48`); S3 bucket public-access-block on both buckets (`terraform/main.tf:533-540,609-616`). |
| **Where least privilege is violated or questionable** | The API's RBAC Role includes `get/list` on `pods` (`kube/rbac.yaml:29-35`) with no corresponding code in `cmd/api/main.go` that uses the Kubernetes API to list pods — apparent over-provisioning (violates least privilege by granting capability the workload doesn't exercise, widening blast radius if the API's ServiceAccount token were ever compromised). The node-level IAM policy (`eks_nodes_s3`) grants S3 permissions to the **entire node**, meaning any pod scheduled there (not just PulseGrid's own pods, if the cluster is ever shared with other workloads) inherits S3 access via the instance metadata service — directly the problem IRSA solves and this repo hasn't implemented (see [IAM & IRSA](#iam--irsa)). |
| **Supply chain — dependencies** | `go.mod`/`go.sum` pin exact dependency versions with cryptographic checksums (`go.sum`) — Go's module system verifies downloaded module content against these checksums by default, a solid baseline supply-chain integrity guarantee "for free" from the toolchain. **No automated dependency vulnerability scanning is configured anywhere** — no `govulncheck`, no Dependabot/Renovate config visible, no `go.mod`-scanning step in `.github/workflows/ci.yml`. This means a known CVE in, say, the AWS SDK, `kafka-go`, `pgx`, or `zap` would go undetected by this repo's CI indefinitely unless someone manually runs `govulncheck` or a similar tool. |
| **Supply chain — base images** | `golang:1.22-alpine` / `alpine:3.19` are pinned to specific tags (good — avoids `latest`-tag drift) but **not pinned to a specific image digest** (`@sha256:...`) — a tag can technically be repointed to different underlying content by the upstream publisher (rare for official images, but a known supply-chain attack vector — "tag mutation") that digest-pinning would eliminate entirely. No image vulnerability scanning step in CI (no `trivy`/`grype`/`docker scan` invocation visible in `.github/workflows/ci.yml`). |
| **Supply chain — GitHub Actions** | Third-party actions (`actions/checkout@v4`, `actions/setup-go@v5`, `docker/build-push-action@v5`, etc.) are pinned to major-version tags, not to commit SHAs — the widely-recommended hardened practice (per GitHub's own security guidance and tools like `zizmor`) is pinning to a full commit SHA (`uses: actions/checkout@<full-sha>`) specifically because a compromised/malicious action publisher *can* move a version tag to point at different, malicious code without changing the visible `@v4` reference in your workflow file — this repo doesn't do that, a real, if industry-common, supply-chain gap. |
| **Alternatives** | `govulncheck` in CI (official Go tooling, purpose-built, low-effort to add as one more CI step); Dependabot/Renovate for automated dependency-update PRs; `trivy`/`grype` container scanning as a CI step or admission-controller gate; SHA-pinning both base images and GitHub Actions. All are well-established, low-to-moderate-effort additions, none currently present. |
| **Tradeoffs** | Every supply-chain hardening measure adds CI time and (for SHA-pinning) some maintenance friction (updating a pinned SHA requires looking up the new commit, not just bumping a version number) — real but modest costs relative to the risk reduction, making this a genuinely good "quick wins" answer if asked "what would you add to this CI pipeline first." |
| **Common mistakes** | Trusting `go.sum` alone as "supply chain security handled" — it verifies *integrity* (the code you downloaded matches what was recorded), not *safety* (whether that code has known vulnerabilities) — these are different properties, and conflating them is a common misunderstanding worth being able to correct in an interview. |
| **Performance implications** | Vulnerability scanning adds CI time (typically seconds to low minutes) — a worthwhile tradeoff against the alternative of shipping known-vulnerable dependencies silently. |
| **Production considerations** | Without automated scanning, this repo's actual current vulnerability exposure (across Go deps, base images, and Action dependencies) is genuinely unknown from the repository alone — a real, checkable "run `govulncheck ./...` right now and see what it finds" action item. |
| **Security considerations** | This entire section is security-focused. |
| **Scalability considerations** | N/A. |
| **Failure scenarios** | A known-critical CVE in a pinned dependency (e.g., a future AWS SDK or `pgx` vulnerability) ships silently into production because nothing in the pipeline would flag it — the failure mode isn't a crash, it's an unnoticed, exploitable weakness persisting indefinitely. |
| **How this repo implements it** | `go.sum` integrity checking (automatic, via the Go toolchain); no vulnerability scanning, no SHA-pinning, mixed IAM/RBAC scoping (mostly good, two notable exceptions above). |
| **Relevant files** | `go.mod`, `go.sum`, `.github/workflows/ci.yml`, `kube/rbac.yaml`, `terraform/main.tf:371-396` |

## Part 10 — Production Engineering

### Monitoring, Incident Debugging & Performance Tuning

Covered in depth under [Observability](#observability--prometheus-grafana-health-checks) — this section adds the *incident-response* and *tuning* angle specifically.

**A realistic incident walkthrough** (good to have ready verbatim for a "walk me through debugging a production issue" prompt): *"Jobs are stuck in `submitted` and never reach `completed`."* Step 1: check `GET /jobs/{id}` — if `status` never changes, first hypothesis given this specific codebase should immediately be the known worker→DB write-back gap (`RecordStatusEvent`/status updates are never called from the worker), **not** a processing failure — this is the single most likely explanation given what's actually implemented, and a strong interview answer demonstrates recognizing that immediately rather than chasing Kafka/ffmpeg red herrings first. Step 2: confirm via `pulsegrid_job_completed_total` (worker metric) — if it's incrementing, jobs *are* completing successfully, just not reflected in Postgres, confirming hypothesis 1. Step 3: if `pulsegrid_job_completed_total` is *not* incrementing, check `pulsegrid_transcode_failures_total{error_type}` — a spike in one label tells you which failure class (retryable/permanent/constraint) to chase; cross-reference `pulsegrid_queue_depth_jobs` (is Kafka backing up — a worker capacity problem?) and worker pod count/`Pending` status (is KEDA/the node group failing to scale, per the capacity-mismatch finding under [EKS](#eks-networking--security-groups)?).

**Performance tuning surface, ranked by likely impact:**
1. **`pgxpool` connection pool sizing** — currently unconfigured (library defaults), the single most impactful and easiest DB-layer tuning knob, given the pool is contended for both fast reads (`GetJobByID`) and the held-open-across-Kafka-publish write transaction (`InsertJobTx`) discussed under [Postgres](#postgres-transactions--consistency).
2. **`http.Server` timeouts** — unconfigured (discussed under [HTTP](#http-server--middleware)), directly affects connection-slot exhaustion resilience under slow/malicious clients.
3. **Health check parallelization** — sequential Postgres→Kafka→S3 checks (discussed under Observability) triples worst-case `/health` latency unnecessarily.
4. **ffmpeg's own thread/preset tuning** — not exposed anywhere in `pkg.Rendition` (no `-preset`/`-threads` control surface for clients or operators) — a real transcode-speed-vs-quality lever currently fixed implicitly by ffmpeg's defaults.

### Deployment, Rollbacks & Disaster Recovery

| | |
|---|---|
| **Deployment strategy** | Standard Kubernetes rolling update (the implicit default `strategy` on both Deployments — no `strategy.type: Recreate` or custom `maxSurge`/`maxUnavailable` specified, meaning Kubernetes' defaults apply: 25% max unavailable, 25% max surge). For the **worker** tier specifically, a rolling update combined with `terminationGracePeriodSeconds: 1800` means an old-version worker pod being replaced during a deploy will **finish its current job** (up to 30 minutes) before actually terminating — deploys are not disruptive to in-flight transcodes, a genuinely good property directly enabled by the grace-period design discussed throughout this guide. |
| **Rollback** | `kubectl rollout undo deployment/pulsegrid-api` (or `-worker`) — standard Kubernetes Deployment rollback to the prior ReplicaSet, works cleanly for application-code rollbacks. **Database rollback is not supported** — migrations are forward-only (no down-migrations in `db/migrate.go`), so rolling back the API binary to a version predating a schema change does *not* revert the schema; if the older binary's queries are incompatible with the newer (already-applied) schema, rollback alone doesn't fix the incident — a real, standard, but important-to-articulate constraint: **application rollback and schema rollback are not the same operation, and this system only supports the former.** |
| **Disaster recovery** | RDS: `skip_final_snapshot = var.environment != "prod"` and `backup_retention_period = var.environment == "prod" ? 7 : 1` (`terraform/main.tf:651-654`) — prod gets a 7-day automated-backup retention window and a final snapshot on deletion; non-prod deliberately does not (faster teardown, no accidental snapshot cost accumulation in throwaway environments) — a sensible environment-differentiated policy. **`db_multi_az = false` by default** (`terraform/variables.tf:171-174`) even in the reviewed default — meaning a single AZ's RDS outage is a real availability event with no automatic failover, recoverable only via restore-from-backup (minutes-to-hours RTO) rather than Multi-AZ's typical sub-minute automatic failover. S3: versioning is **off by default** (`enable_s3_versioning = false`, `terraform/variables.tf:185-189`) — meaning an accidental/malicious object overwrite or delete is **not recoverable** in the default configuration (no previous-version fallback) — a real DR gap for the *output* bucket especially, where a bug in `UploadOutputsToS3` re-running against an already-successful job (the idempotency property discussed earlier) would silently and permanently overwrite prior output with no way to recover the original if something went wrong in the reprocessing. Kafka: as a self-hosted, 3-broker cluster, its own durability/replication settings aren't visible in the reviewed `pkg`/`kube` files — a real open question for anyone operating this system ("what's the Kafka topic's replication factor, and does the DLQ topic have the same durability guarantees as the main topic?"). |
| **Capacity planning** | The throughput model discussed under [Backpressure](#backpressure--horizontal-scaling) (`worker pod count × 1/avg transcode duration`) combined with the node-group-vs-KEDA-max mismatch (`worker_node_max_size: 5` vs. `maxReplicaCount: 30`) means **today's Terraform-provisioned infrastructure cannot actually deliver the autoscaling ceiling the KEDA config advertises** — real capacity planning work here means either reconciling those two numbers, or configuring cluster-autoscaler so the node group itself scales in response to unschedulable pods (not evidenced anywhere in the reviewed files). This is genuinely one of the best "what would you fix first" answers available from this repo, because it's concrete, cross-file, and directly load-bearing on the system's advertised scaling story. |

## How to Explain PulseGrid in an Interview

### The 30-Second Explanation

"PulseGrid is a distributed video transcoding platform — clients upload a video over HTTP, the API stores it in S3 and enqueues a Kafka job, and a fleet of Kubernetes worker pods — autoscaled from zero based on queue depth via KEDA — pull jobs, run ffmpeg to produce multiple renditions plus HLS streaming segments, and upload the results back to S3. It's built in Go, with Postgres for job-state tracking, and it's designed so the API and worker tiers scale completely independently."

### The 2-Minute Explanation

"PulseGrid has two Go services sharing a common library: a stateless HTTP API and a pool of Kafka-consumer worker pods. When a client uploads a video, the API streams it directly to S3 — no local buffering, even for files up to 10GB — then does something I think is the most interesting design decision in the system: it opens a Postgres transaction, inserts the job row, publishes the job to Kafka, and only *then* commits the transaction and returns `202 Accepted` to the client. If the Kafka publish fails, it rolls back the DB insert, so you never get an orphaned job that exists in the queue but not in the database, or vice versa. It's not a fully atomic distributed transaction — there's a small documented window where Kafka succeeds but the DB commit itself could still fail — but it eliminates the much more common orphan case.

Workers are decoupled entirely — they poll Kafka, download the source from S3, shell out to ffmpeg for each requested rendition plus HLS segmentation, upload results, and generate a manifest. Failures get classified into three buckets — retryable, permanent, or resource-constraint — which drives three different responses: re-enqueue with a bumped retry count, send straight to a dead-letter topic, or in the resource-constraint case, actually exit the pod so Kubernetes reschedules it fresh. Worker pod count autoscales from zero to thirty based on Kafka consumer lag via KEDA, which is a nice fit because ffmpeg jobs are long — up to 30 minutes — so the whole system is tuned around that: Kafka session timeouts, pod termination grace periods, and the ffmpeg process timeout are all set to roughly the same 30-minute window so nothing gets killed mid-job."

### The 10-Minute Deep Dive

Use this as a structure, not a script — pull from the tables above for specifics as you go:

1. **Start with the request lifecycle** (30s explanation, expanded) — walk the ASCII diagram from [System Overview](#system-overview).
2. **The DB-Kafka atomicity pattern** — explain the transaction-held-open-across-a-network-call design, why it's unusual, what it buys (no orphans in the common case), what it doesn't fully solve (the documented "ALERT" log-and-hope path), and what the textbook alternative (outbox pattern) would look like instead.
3. **Failure handling as the system's real complexity center** — three-way error classification, the retry-vs-DLQ-vs-pod-exit decision tree, why commit-after-outcome (not commit-after-fetch) gives at-least-once delivery, and why that's the right choice here given S3's natural write-idempotency.
4. **Scaling story** — KEDA + Kafka lag, why pod-level (not goroutine-level) concurrency is the right model for this workload, the 30-minute timeout/grace-period/session-timeout alignment, and — if you want to show real depth — the node-group-vs-KEDA-max capacity mismatch you found by cross-referencing Terraform and the KEDA manifest.
5. **Be ready to volunteer the honest gaps** — this is what separates a mid-level "I read the code" answer from a senior/staff "I understand the system's actual state" answer: the worker never writes completion status back to Postgres (a genuine, documented, unresolved architectural gap), the `jobs.status` CHECK constraint doesn't include `'submitting'` (a real bug no test catches because nothing tests against a real migrated schema), there's no authentication anywhere, and static AWS credentials sit in a plain Secret despite IRSA infrastructure being half-provisioned. Naming these unprompted, accurately, and with a clear sense of "here's what I'd fix and why" is far stronger than waiting to be asked "any concerns?"

## Interview Questions About This Project

Ideal answers below are intentionally terse — expand using the relevant table above when practicing out loud.

### Junior (Q1–Q25)

**Q1. What does `POST /videos/upload` return, and why that status code specifically?**
`202 Accepted` with a `job_id` and `status_uri` — `202` because processing is asynchronous and not yet complete, unlike `201 Created` which would imply the final resource already exists.

**Q2. Why does the worker have no `net/http` routes except `/metrics`?**
It's not a request-driven service — it's a Kafka consumer polling loop; `/metrics` exists only for Prometheus scraping and Kubernetes probes, not for application traffic.

**Q3. What happens if you upload a video without setting `AWS_REGION`?**
The API runs in "local mode" — `s3Uploader` stays `nil`, and the handler synthesizes a fake `s3://pulsegrid-source/{jobID}/original.mp4` URI without actually uploading anything (`cmd/api/main.go:300-302`).

**Q4. What's the maximum upload size, and how is it enforced?**
10GB, enforced primarily via `http.MaxBytesReader` wrapping the request body (cuts the read off as bytes stream in), with `header.Size` checked as a secondary sanity check after parsing.

**Q5. Where are Kafka topic names configured?**
Environment variables — `KAFKA_TOPIC`/`KAFKA_DLQ_TOPIC` (API) and `JOB_TOPIC`/`DLQ_TOPIC` (worker) — note these are *different variable names for the same concept* across the two binaries, both set in `kube/configmap.yaml`.

**Q6. What are the three job statuses a job can be in after submission but before it's fully done?**
`submitting` (transient, inside the open transaction), `submitted` (committed, in queue), `processing` (implied by the model but never actually set by any code path today).

**Q7. Why does the worker container image include ffmpeg but the API's doesn't?**
Only the worker invokes ffmpeg; keeping it out of the API image keeps that image smaller and reduces its attack surface.

**Q8. What port does the worker expose, and what's on it?**
`:8081`, serving only `/metrics` (Prometheus exposition format) — no application routes.

**Q9. How does a client check the status of a submitted job?**
`GET /jobs/{job_id}`.

**Q10. What does `GET /jobs` support for filtering?**
`submitted_after`/`submitted_before` (ISO 8601 timestamps), `status` (comma-separated allow-listed values), `limit` (1-1000, default 100), `offset`.

**Q11. What happens on a malformed `renditions` JSON string?**
`400 VALIDATION_ERROR` — `parseRenditions` returns an error if the JSON doesn't parse, is empty, or any entry lacks both `id` and `type`.

**Q12. What are the default renditions if a client submits none?**
720p H.264, 480p H.264, and an HLS variant based on the 720p resolution — defined in `defaultRenditions()`.

**Q13. What does `GET /health` check?**
Postgres, Kafka, and S3 connectivity, each with a 5-second timeout, returning `200` if all configured checks pass and `503` otherwise.

**Q14. Why might `GET /health` show `"kafka": "not_configured"` instead of `"healthy"` or `"unhealthy"`?**
Because `KAFKA_BROKERS` isn't set — the check distinguishes "not wired up" from "wired up but failing," a three-state (not two-state) health signal per dependency.

**Q15. What's the difference between the `jobs` table and the `job_status_events` table?**
`jobs` is the current-state record (one row per job, mutated in place); `job_status_events` is meant to be an append-only time-series log of lifecycle transitions (a TimescaleDB hypertable) — though in practice it only ever records a single `"submitted"` event today.

**Q16. Which service runs database migrations?**
Only `cmd/api`, on startup, when `DATABASE_URL` is set. The worker never runs migrations.

**Q17. What Go package does this repo use for the HTTP server?**
The standard library's `net/http`, with `http.ServeMux` for routing — no third-party framework.

**Q18. What logging library does the worker use, and why not `log.Printf`?**
`go.uber.org/zap`, for structured, low-allocation JSON logging — `log.Printf` produces unstructured plain-text lines that are much harder to parse/query/filter programmatically at scale.

**Q19. How is a `job_id` generated?**
A UUID v4, hand-rolled via `crypto/rand` plus manual version/variant bit-setting (`pkg.GenerateJobID`) rather than a third-party UUID library.

**Q20. What is `pkg.RetryWithBackoff`, and what's its delay formula?**
A generic retry helper used for S3/Kafka/Postgres calls; delay is `baseDelay * 2^attempt`, capped at 16 seconds.

**Q21. Why does `handleVideoUpload` check `header.Size` *after* already wrapping the body in `MaxBytesReader`?**
Defense in depth — `MaxBytesReader` is the actual enforcement mechanism (can't be lied about), `header.Size` is a secondary, client-reported value checked as a fast-path sanity check, explicitly commented as secondary in the code.

**Q22. What does `docker-compose` do in this repo?**
Nothing — there is no `docker-compose.yml` in this repository; local multi-service orchestration isn't provided (a real gap versus the "Running Locally" instructions, which have you run `go run ./cmd/api` and `go run ./cmd/worker` manually in separate terminals).

**Q23. What HTTP status does `GET /jobs/{job_id}` return for an unknown job?**
`404`, with `error_code: "NOT_FOUND"`.

**Q24. Where does the worker read its pod identity from for logging?**
The `HOSTNAME` environment variable, which Kubernetes automatically sets to the pod's name.

**Q25. What's the significance of `terminationGracePeriodSeconds: 1800` on the worker Deployment?**
It gives an in-flight transcode (up to the 30-minute ffmpeg timeout) time to finish before Kubernetes force-kills the pod during a scale-down or rolling deploy.

### Mid-Level (Q26–Q55)

**Q26. Walk through exactly what happens, in order, when `handleVideoUpload` is called with Postgres and Kafka both configured.**
Generate request ID → enforce size limit → parse multipart form → validate `source_name`/`video`/`renditions` → generate `job_id` → upload to S3 (if configured) → build `Job` struct → `InsertJobTx` (opens tx, inserts row with `status=submitting`) → `EnqueueJob` to Kafka (outside the tx) → on Kafka success, `UpdateStatusAndCommit` (updates status to `submitted`, commits) → best-effort `RecordStatusEvent` → emit metrics → return `202`.

**Q27. What happens if the Kafka publish fails after the DB insert but before commit?**
`txHandle.Rollback` is called — the job row is never persisted, so from any external observer's point of view, the job submission never happened; the client gets a `500`.

**Q28. What happens if the Kafka publish *succeeds* but the subsequent DB commit fails?**
The job is now live in Kafka (a worker will pick it up) but has no corresponding Postgres row — an orphan. The code logs an `"ALERT"`-prefixed message and still returns `202` to the client, since the job genuinely will be processed. This is a documented, accepted edge case, not silently ignored.

**Q29. Why is this DB-then-Kafka pattern not a true distributed transaction?**
Because there's no two-phase commit or saga coordinating the Postgres commit and the Kafka publish as one atomic unit — it's an ordered sequence with a rollback safety net on one specific failure ordering (Kafka fails first), not a general solution to "keep two systems in sync under all failure combinations."

**Q30. What's the textbook fix for the "Kafka succeeds, DB commit fails" gap?**
The outbox pattern — write the job row and a "pending message" row in a single local transaction, then have a separate relay process publish to Kafka asynchronously and mark the outbox row as sent, eliminating the cross-system network call from inside the DB transaction boundary entirely.

**Q31. Why does the worker hold a Kafka `SessionTimeout` of 30 minutes instead of the more typical 10-45 seconds?**
Because a consumer that goes 30+ seconds without heartbeating gets evicted from the consumer group under typical settings — but this worker can be legitimately busy (blocked in an ffmpeg call) for up to 30 minutes at a time, so the timeout has to exceed the longest expected single-job processing time.

**Q32. What determines whether a failed job is retried, DLQ'd, or causes a pod exit?**
`pkg.ClassifyError`, which checks for specific typed errors (`ResourceConstraintError` → constraint/pod-exit; `PermanentError`/`SourceNotFoundError` → permanent/DLQ) first, then falls back to substring-matching the error message against known permanent and constraint patterns, defaulting to retryable otherwise.

**Q33. Why does a resource-constraint error cause `os.Exit(1)` instead of a normal error return?**
Because the failure indicates the *pod itself* is out of a resource (memory, disk) — retrying in-place on the same constrained pod is expected to fail identically; exiting lets Kubernetes reschedule onto (potentially) healthier capacity, and the Kafka offset is deliberately left uncommitted so another consumer picks up the message.

**Q34. What's the maximum number of retries before a job goes to the DLQ, and where is that enforced?**
3, via the `maxRetries` constant in `cmd/worker/main.go`, checked against `kafkaMsg.RetryCount` in `handleJobOutcome`.

**Q35. Why is the Kafka offset committed in the retry case, not just the success/DLQ cases?**
Because a retry means the message has been *re-published* as a new message with an incremented retry count — the original message's processing is "done" (in the sense that its outcome has been handled, either by re-enqueueing or DLQ'ing), so its offset should be committed to avoid reprocessing the stale original.

**Q36. What guarantees does this system's delivery model actually provide — exactly-once, at-least-once, or at-most-once — and why?**
At-least-once. Offsets commit only after processing completes (success, DLQ, or re-enqueue), so a crash between fetch and commit causes redelivery — meaning a message can be processed more than once, but never silently dropped without being handled somehow.

**Q37. Given at-least-once delivery, how does this system avoid producing duplicate output on redelivery?**
S3 output keys are deterministic (`{jobID}/{rendition}/{filename}`, no random component) — reprocessing the same job overwrites the same keys with equivalent content, an idempotent-by-construction side effect rather than an explicitly deduplicated one.

**Q38. What partitioning strategy does the Kafka producer use, and why?**
Hash-partition by `job_id` (`kafka.Hash{}` balancer) — guarantees all messages for one job (including retries) land on the same partition and are processed in order relative to each other.

**Q39. What's the practical ceiling on useful worker parallelism, independent of pod count?**
The number of partitions on the `transcoding-jobs` topic — a consumer group cannot have more actively-consuming members than there are partitions; excess pods beyond the partition count sit idle.

**Q40. Why might KEDA's `maxReplicaCount: 30` never actually be reached in practice?**
Because the worker node group's Terraform-configured `max_size` is only 5 — without cluster-autoscaler reconciling node capacity with pod demand, the node group physically can't schedule anywhere near 30 pods at the current per-pod resource requests.

**Q41. What is `pgxpool`, and what isn't configured on it in this codebase?**
The Postgres connection pool used by `pgx/v5`; this repo never sets `MaxConns`/`MinConns` explicitly, relying entirely on library defaults rather than values tuned for expected concurrent request volume.

**Q42. Why does `InsertJobTx` hold a database transaction open across a Kafka network call — isn't that a well-known anti-pattern?**
Yes, generally — holding a DB connection/transaction open across a slow external call risks pool exhaustion and extended lock duration. It's accepted here deliberately to get "no Kafka message without a corresponding DB row" as a guarantee in the common failure case, a documented tradeoff, not an oversight.

**Q43. What does `errors.As` do differently from a plain type assertion, and where does this codebase rely on that difference?**
`errors.As` walks the `Unwrap()` chain to find a matching type even through multiple layers of `fmt.Errorf("...: %w", err)` wrapping; a plain type assertion only checks the immediate value. `ClassifyError` relies on this to find, e.g., a `*SourceNotFoundError` several wrap-layers deep inside `processJob`'s returned error.

**Q44. What would happen if a caller used `%v` instead of `%w` when wrapping an error in this codebase?**
The wrapped error would become an opaque string — `errors.As`/`errors.Is` downstream (specifically `ClassifyError`) would no longer be able to find the underlying typed error, and classification would fall through to string-pattern-matching only (or default to retryable), potentially misclassifying a permanent failure as retryable.

**Q45. Why does the worker use `context.Background()` for `processJob` instead of a context derived from the shutdown signal?**
So that an in-flight job is never cancelled by a shutdown request — the design deliberately lets current work finish rather than abandoning it mid-transcode, relying on the bounded `terminationGracePeriodSeconds` to cap total shutdown time instead.

**Q46. Where in this codebase would a data race be most likely if `sync/atomic` weren't used correctly?**
The `shutdownRequested` flag — read by the main polling loop, written by a separate signal-handling goroutine; using a plain `bool` instead of `atomic.Bool` would be a textbook data race, caught by `go test -race` (run in CI).

**Q47. What's the actual concurrency model for job processing — goroutine worker pool, or something else?**
Pod-level, not goroutine-level: each worker pod processes exactly one job at a time in a sequential polling loop; concurrency comes from KEDA scaling the *number of pods*, not from an in-process worker pool of goroutines.

**Q48. Why might pod-level concurrency be a better fit here than an in-process goroutine pool?**
ffmpeg's CPU/memory usage is substantial per job — running N jobs concurrently within one pod would require N× the CPU/memory anyway, complicate OOM attribution (which job caused it?), and per-job resource isolation is much cleaner at the pod/container boundary than within a shared process.

**Q49. What does `MaxAttempts: 1` on the Kafka writer configuration mean, and why is it set that way?**
It disables kafka-go's own internal retry so that all retry logic is centralized in `pkg.RetryWithBackoff`, avoiding two independent, potentially-compounding retry mechanisms operating on the same publish call.

**Q50. Why does the manifest generation code unmarshal the JSON it just marshaled, right after marshaling it?**
As a belt-and-suspenders correctness check — verifying the manifest is genuinely valid, round-trippable JSON before writing it to disk/S3, at the cost of doing the encode work twice.

**Q51. What test type is conspicuously (and admittedly, per the audit doc) under-implemented relative to the original spec?**
True property-based tests using `testing/quick` — only 3 files (`postgres_test.go`, `kafka_test.go`, `jobid_test.go`) use it, versus roughly 8 subsystems the original task spec called for; the rest are thorough example-based tests covering similar logic without randomized-input generation.

**Q52. Why does `test/integration` reimplement `handleUpload`/`handleGetJob` locally instead of importing them from `cmd/api`?**
Because `cmd/api` is a `package main`, and Go doesn't allow importing symbols from another `main` package — there's no way to reuse those handler functions directly from an external test package.

**Q53. What single missing piece of test infrastructure would have caught the `jobs.status` CHECK-constraint bug before it shipped?**
A test running against a real Postgres instance (e.g., via `testcontainers-go`) with `db.RunMigrations` actually applied — every existing DB-related test uses a mock `DBClient`, so nothing ever attempts an `INSERT` against the real, migrated schema.

**Q54. What's the actual bug, precisely?**
`db/migrations/001_create_jobs_table.sql`'s CHECK constraint only allows `status IN ('submitted', 'processing', 'completed', 'failed')`, but `InsertJobTx` inserts `status = 'submitting'` (the `JobStatusSubmitting` constant) as the very first write for every job — against a real database, this `INSERT` would fail its CHECK constraint on every single submission.

**Q55. Given that bug, what does that imply about how this codebase has actually been tested/run so far?**
Almost certainly never against a real Postgres instance with the migrations applied end-to-end in a way that exercised `InsertJobTx` — otherwise this would have failed immediately and loudly on the very first submission attempt.

### Senior (Q56–Q80)

**Q56. If you had to add authentication to this API today, where exactly would you put it, and what would you need to change beyond just the HTTP layer?**
Ideally as middleware wrapping every handler in `http.ServeMux` (currently absent entirely, so this would be new infrastructure, not a modification of existing middleware) — but authentication alone is insufficient given `GET /jobs` has no per-caller scoping; you'd also need an `owner`/`tenant_id` column on the `jobs` table, a corresponding index, and a `WHERE owner = $N` clause added to `QueryJobs`/`GetJobByID` — a schema change and query change, not just an API-layer bolt-on.

**Q57. The AWS review recommends replacing self-hosted Kafka with SQS. What would you actually need to verify before doing that, and what's the risk if you skip verification?**
Whether anything genuinely depends on Kafka-specific guarantees this system currently gets "for free" — specifically per-job ordering via hash-partitioning (does out-of-order retry processing matter?) and consumer-group-lag as KEDA's exact scaling signal (SQS's queue-depth signal is coarser — it doesn't distinguish "messages waiting" from "messages currently being processed slowly" the way per-consumer-group lag can). Skipping verification risks silently breaking retry-ordering guarantees or getting materially different (likely worse-tuned, since it wasn't designed against this metric) autoscaling behavior.

**Q58. Explain the tradeoff between this repo's migrate-on-API-boot approach and a dedicated migration Job/init-container, specifically in terms of failure modes under a 2-replica rolling deploy.**
Migrate-on-boot means both API replicas independently attempt migrations on every restart; the `schema_migrations` table plus per-migration transactions prevent double-application, but a genuine race exists where both replicas see a migration as pending, both begin, and one loses on the `schema_migrations` primary-key insert, causing that replica to fail its transaction, `log.Fatalf`, and crash-loop until retry. A dedicated migration Job run to completion before the Deployment rollout begins eliminates this race entirely by making "apply schema" and "serve traffic" strictly sequential phases rather than concurrent ones.

**Q59. Why is holding a Postgres transaction open across a Kafka publish call risky under load, specifically in terms of connection pool behavior?**
Every concurrent `handleVideoUpload` call checks out a pool connection for the *entire* duration of `InsertJobTx` through `UpdateStatusAndCommit` — which spans an uncontrolled-duration external network call (up to ~31 seconds under Kafka's worst-case retry backoff). Under concurrent load, this can exhaust the (currently unconfigured, defaulted) pool faster than a workload where DB connections are held only for the duration of pure-DB work, potentially queuing or failing unrelated DB reads (`GetJobByID`/`QueryJobs`) that only need a connection briefly.

**Q60. What's the actual capacity mismatch between Terraform and the KEDA configuration, and how would you reconcile it?**
`worker_node_max_size` in Terraform defaults to 5 (t3.large instances), while KEDA's `ScaledObject` sets `maxReplicaCount: 30` — at the worker Deployment's resource requests (2 CPU/4Gi per pod), 5 t3.large nodes (2 vCPU/8GB each) can schedule at most roughly 5-10 pods, nowhere near 30. Reconciliation options: raise `worker_node_max_size` to match realistic demand at 30 pods' resource requests, lower `maxReplicaCount` to what the node group can actually support, or (better) deploy cluster-autoscaler so the node group scales dynamically in response to unschedulable pods rather than being capped by a static Terraform value.

**Q61. Walk through what "effectively-once" processing means here, as distinct from Kafka's own exactly-once semantics feature, and why this repo relies on the former instead of the latter.**
Kafka's exactly-once semantics (transactional producers, read-committed consumers) guarantee exactly-once *delivery* within Kafka itself, but don't extend to external side effects like S3 writes or DB writes without additional coordination (e.g., an outbox/two-phase mechanism). This repo instead accepts at-least-once *delivery* and makes its S3 side effect naturally idempotent (deterministic, overwriting keys) — achieving an equivalent *end-state* outcome ("effectively-once") without the complexity of Kafka transactions, which also wouldn't have covered the Postgres side of the system anyway.

**Q62. If the worker's DB write-back gap were fixed today, what idempotency risk would that introduce, and how would you mitigate it?**
If `RecordStatusEvent`/a status-update call were added to `handleJobOutcome`, a redelivered message (from a rebalance-during-processing scenario) could cause that DB write to happen twice — for `RecordStatusEvent` specifically, this means a duplicate row in `job_status_events` (not corrupting, since it's an append-only log, but skewing counts/analytics); for a hypothetical `UPDATE jobs SET status = 'completed'` call, a duplicate execution is actually harmless (idempotent by nature — setting the same value twice). The append-only event table is the actual risk; mitigation would be either a unique constraint on `(job_id, event_type, some_dedup_window)` with `ON CONFLICT DO NOTHING`, or including the Kafka offset/a message-level idempotency key in the event row to detect and skip duplicates explicitly.

**Q63. Why does `ClassifyError` combine typed-error checks with string-pattern matching instead of using one approach exclusively?**
Typed errors work well for failure modes the application code itself constructs and controls (a `SourceNotFoundError` raised explicitly on S3 404). String matching is the fallback for failure signals originating from code this application doesn't control — `exec.Command`'s stderr output, syscall-level errors surfaced as plain Go error strings — where there's no practical way to get a typed Go error from, e.g., ffmpeg's own text output. The combination trades some brittleness (message-text changes silently misclassify) for coverage the typed-error-only approach couldn't achieve without wrapping every possible external failure text in a bespoke type.

**Q64. What's the risk of `ClassifyError`'s string-matching approach specifically with respect to a future dependency upgrade?**
If the AWS SDK, Kafka client, or any other dependency changes the exact wording of an error message that a pattern (e.g., `"no space left on device"`) currently matches, that failure would silently reclassify — most dangerously, a resource-constraint failure reclassifying as retryable would cause the worker to retry in place on a still-resource-constrained pod rather than exiting for rescheduling, potentially looping ineffectively rather than recovering.

**Q65. Explain precisely why `exec.Command`-based ffmpeg invocation in this codebase is not vulnerable to shell injection, but is still not fully safe from malicious rendition input.**
`exec.CommandContext(ctx, "ffmpeg", args...)` passes each argument as a distinct argv element without invoking a shell — so shell metacharacters (`;`, `|`, backticks) in a client-supplied field are passed literally to ffmpeg as one argument, not interpreted for command chaining/substitution. However, the arguments themselves are not validated against an allow-list before reaching ffmpeg's own argument parser — ffmpeg itself has historically had vulnerabilities triggerable via crafted argument/input combinations, so unvalidated free-form strings reaching its argv (even safely, non-shell-interpreted) is still a real, if narrower, risk surface than shell injection would be.

**Q66. What concrete fix would close that gap, and what's the tradeoff of the stricter alternative?**
Allow-list validation on `VideoCodec`/`Resolution`/bitrate fields against known-safe patterns (a fixed codec set, a `WIDTHxHEIGHT` regex, a `\d+[kKmM]` bitrate pattern) closes the gap while preserving client flexibility. The stricter alternative — accepting only a fixed set of named presets mapped server-side to hardcoded ffmpeg arguments — eliminates the risk class entirely but removes clients' ability to request arbitrary custom encoding parameters.

**Q67. Why might `CombinedOutput()` on the ffmpeg subprocess call be considered a latent memory risk, even though stderr is truncated to 500 characters before logging?**
Truncation happens *after* `CombinedOutput()` has already fully buffered the entire stdout+stderr stream into memory — the 500-character cap only limits what's logged, not what's collected. An unusually verbose or looping ffmpeg process could buffer an arbitrarily large amount of output in memory before the call even returns, independent of the eventual log truncation.

**Q68. What does the presence of both `pkg/kafka.go`'s `KafkaProducer` interface and `pkg/queue/kafka.go`'s `MessageQueue` interface suggest, and what would you want to know before treating one as dead code?**
It suggests either an in-progress refactor toward a cleaner, unified producer+consumer abstraction (`MessageQueue` covers both publish and consume/commit in one interface, versus `KafkaProducer`'s producer-only surface used alongside a separately-constructed `kafka.Reader` in `cmd/worker`), or an abandoned exploration. Before calling it dead code, you'd want to check git history/blame for recency and intent, confirm no other package/test imports `pkg/queue`, and ask whoever owns the roadmap whether a migration to the newer abstraction is planned.

**Q69. Why does the health check's sequential (not parallel) checking of Postgres, Kafka, and S3 matter more than it might first appear?**
Because `GET /health` backs both liveness and readiness probes for the API — if any one dependency check is slow (approaching its 5-second timeout), the *total* health check latency can approach 15 seconds in the worst case (three sequential 5-second timeouts), which could interact with probe `timeoutSeconds`/`failureThreshold` settings in ways that cause Kubernetes to mark the pod unhealthy for reasons unrelated to the pod's actual serving capability — parallelizing with `errgroup` would cut worst-case latency roughly 3x and reduce this false-unhealthy risk.

**Q70. What's the practical difference between the API's liveness probe and readiness probe as configured, given both hit the same `/health` endpoint?**
They hit the same endpoint with different consequences: a failing readiness probe removes the pod from Service load-balancing (traffic stops routing to it) without restarting it; a failing liveness probe restarts the pod. Since `/health` actively checks Postgres/Kafka/S3, a genuine downstream outage (not an API-process problem at all) would eventually fail *both* probes — meaning the API pod gets restarted for a problem restarting it cannot fix, a real "does this probe design cause unnecessary churn during a dependency outage" question worth raising.

**Q71. Given the worker's liveness probe is a bare TCP check on the metrics port, what failure mode could this probe fail to detect?**
A "zombie" worker — the metrics HTTP server goroutine is alive (satisfying the TCP check) while the main polling loop itself is wedged (blocked indefinitely on something that ignores its context timeout, or panicked in a way that didn't crash the whole process). The probe would report healthy while the pod does no actual job processing.

**Q72. Why is S3 versioning off by default, and what's the actual risk that introduces given the idempotent-overwrite design elsewhere in this system?**
Versioning is off (`enable_s3_versioning = false` by default) to avoid its storage-cost and lifecycle-complexity overhead in non-critical environments. The risk: the same idempotent-overwrite property that makes duplicate processing safe (same key, equivalent content, harmless overwrite) also means a *buggy* reprocessing run (e.g., a code bug that generates wrong output for a previously-successful job) would silently and irrecoverably destroy the correct prior output, since there's no previous-version fallback without versioning enabled.

**Q73. Why does `db_multi_az` defaulting to `false` matter for disaster recovery specifically, separate from the backup_retention_period setting?**
Backup retention governs how far back you can *restore from* (recovery point objective, RPO) but a restore is a manual, minutes-to-hours operation (recovery time objective, RTO). Multi-AZ provides automatic, typically sub-minute failover to a standby replica on primary failure — without it, a single AZ's RDS outage is a real, non-instantaneous incident requiring manual restore, not an automatically-mitigated one, regardless of how good the backup retention policy is.

**Q74. What's the argument for keeping self-hosted Kafka despite the AWS review's SQS recommendation, articulated as strongly as possible?**
Kafka's consumer-group-lag metric is a more precise autoscaling signal than SQS queue depth (it reflects *per-consumer-group* backlog, not just total messages in flight, and distinguishes messages genuinely waiting from those currently being slowly processed by an existing consumer); per-partition ordering guarantees retry-in-order-relative-to-original-message semantics that SQS FIFO would only replicate at real throughput cost; and Kafka's replay capability supports future use cases (backfill, DLQ-recovery-via-replay) that SQS's delete-on-receive model doesn't naturally support without extra plumbing. If any of these properties are load-bearing for correctness (not just convenience), the ops cost of self-hosting may be justified.

**Q75. What single change would most improve this system's testability of the DB layer, and why does it matter more than adding more example-based unit tests?**
Introducing `testcontainers-go` to run tests against a real, migrated Postgres instance. It matters more than additional example-based tests because the entire class of "does the code's assumptions about the schema actually hold against the real schema" bugs (the CHECK-constraint mismatch being the concrete, shipped example) is structurally invisible to any test that only exercises a mock `DBClient` — no amount of additional mock-based test cases can catch a mismatch between application code and the actual database schema.

**Q76. Explain why `context.Background()` inside `processJob`, rather than a context derived from the poll loop's per-fetch timeout, is the correct choice here — and identify the one scenario where it's arguably *not* correct.**
It's correct because it decouples in-flight job processing from the unrelated 10-second poll-timeout context, ensuring a job isn't cancelled just because its *originating fetch's* context happened to have a short timeout. It's arguably not correct in the case of a pod that needs to shut down *faster* than 30 minutes for an urgent reason (e.g., a critical security patch requiring immediate redeploy) — there's currently no mechanism to force-cancel an in-flight job faster than SIGKILL after the full grace period, since nothing downstream of `processJob` ever observes a cancellable, shutdown-aware context.

**Q77. If you needed to add per-tenant rate limiting to this API, where would it fit most naturally given the current architecture, and what would it need to depend on that doesn't exist yet?**
Most naturally as middleware wrapping the mux (new infrastructure, since none exists today), likely token-bucket or sliding-window based, keyed by whatever caller identity authentication would introduce. It fundamentally depends on authentication existing first — you cannot rate-limit "per tenant" without first being able to identify which tenant a given request belongs to, which this system currently has no mechanism for at all.

**Q78. What GitHub Actions / CI hardening gap is most worth prioritizing, and why that one first?**
Adding `govulncheck` as a CI step — it's official Go tooling (zero new trust dependency), directly answers "are we shipping any dependency with a known vulnerability," and is the cheapest (in engineering effort) of the available supply-chain improvements (SHA-pinning actions/images is more mechanical toil for comparatively narrower risk reduction against a less likely attack vector for most teams).

**Q79. Why is `image: pulsegrid/api:latest` in the Kubernetes manifests a deployment-reproducibility risk specifically, distinct from it being a general best-practice violation?**
Because it makes "what code is actually running in this cluster right now" unanswerable from the manifest alone — two different clusters both running `:latest` could be running entirely different commits depending on when each last pulled the image, and a `kubectl rollout restart` without a corresponding new build/push wouldn't deploy new code at all (the tag hasn't moved from that cluster's point of view until the next `docker push` to `latest` happens and something triggers a fresh pull) — reproducibility and auditability both suffer versus referencing the CI-produced SHA-tagged image explicitly.

**Q80. Staff-level framing: if you had one week to spend improving this system before it took production traffic, what would you do and in what order, and why that order?**
(1) Add authentication + basic per-caller job scoping — currently a complete open door, the single most severe risk, and blocking for literally any real external traffic. (2) Fix the `jobs.status` CHECK constraint bug — a one-line migration fix that currently makes every real-database submission fail outright; trivial to fix, catastrophic if missed. (3) Wire the worker's DB write-back (`RecordStatusEvent`/status updates) so job status actually reflects reality — currently the system's core "is my job done" promise is broken for real users relying on `GET /jobs/{id}` rather than metrics. (4) Move static AWS credentials to IRSA — meaningfully reduces blast radius on any future credential leak, and the OIDC groundwork is already provisioned, making this comparatively cheap. (5) Reconcile the KEDA-max/node-group-max capacity mismatch, so the advertised autoscaling ceiling is real. This order prioritizes correctness-blocking and security-blocking issues (things that make the system either not work or actively dangerous) ahead of capacity/completeness issues (things that make it not scale as advertised) — a deliberate "safe and correct before impressive" sequencing.

### Staff (Q81–Q100)

**Q81. The system advertises "atomic DB-Kafka write ordering" in its own code comments. Is that terminology accurate, and if not, what would you call it instead in a design review?**
Not fully accurate in the strict distributed-systems sense — there's no atomicity across the DB and Kafka as a single indivisible unit; there's an *ordered sequence with a partial rollback safety net* covering exactly one failure ordering (Kafka-fails-after-DB-insert). I'd call it "orphan-preventing ordered writes with a documented consistency gap" in a review, and I'd push for either implementing the outbox pattern to actually close that gap, or explicitly widening the on-call runbook to cover the "ALERT" log line's remediation, since today it's logged but not automatically actioned.

**Q82. If you were asked to make this system's job-submission path a true saga (with compensating actions), what would the steps and compensations look like?**
Steps: (1) insert job row (compensate: delete row), (2) upload source to S3 (compensate: delete S3 object), (3) publish to Kafka (compensate: none possible — Kafka publish can't be un-published; this is the crux of why sagas are awkward here). Because step 3 has no meaningful compensation (a consumer might already have read the message before you'd try to "compensate"), a saga isn't actually a clean fit for this specific ordering — it's a good illustration of why the simpler ordered-writes-with-rollback-of-the-still-compensable-steps approach this repo already uses (rollback the DB row, which *is* fully compensable, before the Kafka step, which isn't) is actually a reasonable practical simplification of full saga semantics, not a naive shortcut.

**Q83. Argue for and against moving this system fully to an outbox-pattern architecture, considering the relay process's own failure modes.**
For: eliminates the held-open-transaction-across-network-call anti-pattern, closes the "Kafka succeeds, DB commit fails" gap entirely (the DB write and the outbox-row write are the same local transaction, always atomic), and decouples DB write latency from Kafka availability. Against: introduces a new component (the relay process/CDC mechanism) with its own failure modes — if the relay itself lags or crashes, jobs sit "submitted in DB" but not yet actually in Kafka, an entirely new kind of latency/staleness window that didn't exist before, and now requires monitoring/alerting on relay lag specifically. It trades one well-understood, documented consistency gap for a differently-shaped one, plus additional operational surface area — a legitimate, non-obvious tradeoff, not a strict improvement.

**Q84. This system currently cannot answer "which specific worker pod, at which point in a multi-hour incident, processed job X" without manually correlating logs by job_id and timestamp across potentially dozens of ephemeral pods. Design the minimal addition that would fix this, and justify why you'd choose it over full distributed tracing.**
Minimal fix: ensure every log line and every DLQ/status-event write consistently includes `job_id`, and ship logs to a centralized, queryable store (e.g., Loki, CloudWatch Logs, or an ELK stack) — none of which exists today (logs currently only reach `kubectl logs`/stdout, ephemeral per pod). This alone (structured logs + centralized storage + query-by-`job_id`) would answer the stated question without needing span-level distributed tracing, because this system's call chain per job is shallow (no deep service-to-service fan-out) — tracing's marginal value over "grep centralized structured logs by job_id and sort by timestamp" is real but smaller here than in a genuinely fan-out-heavy microservices architecture, making the simpler fix the better first investment.

**Q85. Critique the choice to use `testing/quick` (rather than `gopter`) for the three property tests that exist, from a staff-engineer code-review perspective.**
`testing/quick` is stdlib-only (zero new dependency, a real plus for a codebase already minimizing third-party surface) but has no shrinking — when a property test fails, you get whatever random input happened to trigger the failure, not a minimized reproducing case, making root-causing a failure slower than `gopter` would allow. For a codebase this size with only three property tests, the shrinking gap probably isn't yet worth the dependency; if property-based testing were meaningfully expanded (closer to the ~8 subsystems the original task spec called for), I'd revisit `gopter` specifically for its shrinking, since debugging efficiency compounds in value as property-test surface area grows.

**Q86. The worker node group has a `NoSchedule` taint (`workload=transcoding`). Trace through what you'd need to verify in the actual cluster (not just these files) to confirm the worker Deployment can schedule at all, and what the failure mode looks like if it can't.**
I'd check `kube/worker-deployment.yaml`'s pod spec for a matching `tolerations` entry (`key: workload, value: transcoding, effect: NoSchedule`) — the reviewed file doesn't show one, which would mean the Kubernetes scheduler refuses to place worker pods on the tainted nodes at all. Failure mode: pods stay `Pending` indefinitely (`kubectl get pods` shows `0/1 Running`, `kubectl describe pod` shows a scheduling-failure event citing the untolerated taint), Kafka lag grows unbounded despite KEDA "requesting" more replicas, and `HighQueueDepth` eventually fires with no corrective action ever actually taking effect — a silent, fully-masked capacity failure unless someone is specifically watching pod scheduling state, not just the queue-depth alert.

**Q87. Design a schema and write-path for making `job_status_events` a true, complete event-sourced audit log (every lifecycle transition, from both API and worker), and identify the biggest design risk in doing so.**
Schema: keep the existing hypertable, but ensure every transition (`submitted`, `processing_started`, `completed`, `failed`, `retried`, `dlq_dead_lettered`) is written from the single authoritative point where that transition happens — meaning the worker needs a DB client (closing the existing gap) and every `handleJobOutcome` branch needs a corresponding `RecordStatusEvent` (or richer event) call. Biggest risk: making event-writes required (not best-effort) for correctness would mean a DB outage now blocks job processing entirely (today, worker processing is fully DB-independent) — a real availability-coupling tradeoff introduced by "fixing" the audit-log gap; the safer design keeps these writes best-effort/fire-and-forget (as the API side already does) so a DB blip degrades observability, not core processing capability.

**Q88. This system's worker pods run arbitrary client-supplied video files through ffmpeg with no sandboxing beyond a non-root container user. As a staff engineer responsible for the security posture, what additional layer would you push for first, and what's the cost/benefit?**
A seccomp profile restricting the syscalls available to the ffmpeg subprocess (or the whole container) — meaningfully reduces the kernel-level attack surface an ffmpeg RCE could exploit, at relatively low implementation cost (define/test an allow-list profile, apply via pod `securityContext.seccompProfile`) compared to heavier options like gVisor/Kata (stronger isolation, real performance overhead for a CPU-intensive workload like transcoding, likely not worth it here). I'd push for seccomp first as the best cost/benefit point on this specific curve, and treat gVisor-level sandboxing as a "only if a specific incident or compliance requirement demands it" escalation.

**Q89. If this system needed to support 100x its current job volume, name the three components most likely to become bottlenecks first, in order, and why.**
(1) The single-primary Postgres instance (no read replicas) — `GetJobByID`/`QueryJobs` read load scales with client polling frequency × job volume, and at 100x volume this is the first hard wall without read-replica fan-out. (2) Kafka partition count — if under-provisioned for 100x throughput, this caps useful consumer parallelism regardless of how many worker pods KEDA tries to add, a purely configuration-level (not architectural) fix, but a real one to catch before it bites. (3) The `maxReplicaCount: 30` ceiling itself, combined with the underlying node-group capacity — 100x volume almost certainly needs a KEDA max well beyond 30 and a node group (or cluster-autoscaler policy) actually capable of supporting it.

**Q90. Defend or critique this design decision: the API validates rendition *structure* (fields present) but not rendition *semantics* (codec/bitrate/resolution values are meaningful and safe), pushing semantic validation entirely to ffmpeg's own argument parsing at process-invocation time.**
Critique: this pushes a validation responsibility to the worst possible point in the pipeline — after a job has already been accepted, queued, and picked up by a worker, meaning a client submitting garbage rendition values wastes a full round-trip through the queue and a worker pod's time before failing, rather than failing fast at submission with an immediate, actionable `400`. It's also the root cause of the ffmpeg-argument-injection-adjacent finding discussed earlier. Defense: doing full semantic validation at the API requires the API to encode knowledge of ffmpeg's entire valid-argument grammar, which is a real maintenance burden (every new supported codec/format needs an API-layer update too) — a middle ground (allow-list the handful of codec/format families actually supported, validate structural patterns for bitrate/resolution) captures most of the safety benefit without requiring the API to fully mirror ffmpeg's argument surface.

**Q91. The load-test harness computes p50/p95/p99 latency and scale-up/scale-down timing by scraping `/metrics` on an interval and parsing raw Prometheus text format with string matching, rather than using a Prometheus client library. Evaluate this choice.**
It avoids adding a Prometheus client-library dependency purely for a read-and-parse use case (the harness is a producer of load, a consumer of metrics text, not itself exposing metrics) — reasonable minimalism. The tradeoff: hand-rolled text parsing (`strings.HasPrefix` matching against specific metric name prefixes, `cmd/load-test/harness.go:807-820`) is brittle against metric-name or format changes in exactly the way a proper Prometheus text-format parser (e.g., `prometheus/common/expfmt`) wouldn't be — a real "this will silently break if someone renames a metric" risk for comparatively little dependency-avoidance benefit, since `expfmt` is a lightweight, already-transitively-available (via `client_golang`) parsing library.

**Q92. Design an incident-response runbook entry for "the `ALERT: DB commit failed after Kafka publish` log line has appeared in production." What would an on-call engineer need to do, and what tooling gap would make that harder than it should be?**
The on-call engineer needs to: identify the specific `job_id` from the log line, check whether a corresponding `jobs` row exists (it may not, given the failure), and if not, manually insert a reconciling row (or accept the job will process successfully but never be independently queryable via `GET /jobs/{id}` until manually fixed) — essentially manual saga compensation for a scenario the system itself can't self-heal. The tooling gap: there's no automated reconciliation job that periodically diffs "jobs known to Kafka/worker completion metrics" against "jobs known to Postgres" to *find* these orphans proactively — today, discovery relies entirely on someone noticing this specific log line in real time, which is a weak, easy-to-miss detection mechanism for a rare-but-real failure mode.

**Q92b (bonus follow-up).** *What would that reconciliation job's query logic look like, at a high level?* Periodically compare the set of `job_id`s the worker has emitted `pulsegrid_job_completed_total`/DLQ events for (or, better, a durable worker-side log/table of processed job IDs, which doesn't fully exist today either) against the set of `job_id`s present in Postgres, flagging any processed-but-absent-from-DB job_id as an orphan needing manual backfill — itself dependent on the worker gaining some durable record of what it's processed, tying back to the broader worker-DB-write-back gap discussed throughout this guide.

**Q93. Is the choice to make `RecordStatusEvent` "best-effort" (failure only logged, not surfaced to the client or retried) the right call? Argue both sides.**
For: a submission's core promise (job is durably queued and will be processed) is fully satisfied by the DB-insert-plus-Kafka-publish sequence; the audit-event write is a secondary, non-critical-path concern, and failing the entire request over it would be over-strict — you'd be telling a client "your job failed" when it actually didn't. Against: silently swallowing this failure means `job_status_events` can systematically under-report even its one current event type with zero visibility beyond a log line nobody may be watching — if this audit trail is ever relied upon for compliance or billing purposes, "best-effort, unmonitored" is a real, understated data-integrity risk that deserves at least a dedicated metric/alert on `RecordStatusEvent` failure rate, which doesn't currently exist.

**Q94. This repo has zero Go benchmarks (`testing.B`). Identify the one function in this codebase where you'd most want a benchmark, and explain what decision a benchmark result would actually inform.**
`pkg.ClassifyError` — it's called on every single failure path in the worker's hot loop, involves multiple `errors.As` calls plus a linear scan over string patterns via `strings.Contains`; a benchmark would tell you whether this function's cost is negligible (likely, given failure classification isn't the dominant per-job cost — ffmpeg is) or whether, at very high failure-rate/high-throughput scenarios, the pattern-matching approach itself becomes a measurable tax, informing whether a smarter data structure (e.g., a precompiled matcher) is ever worth the complexity versus the current simple loop.

**Q95. Critique the decision to make both `KafkaClient` (pkg/kafka.go) and `KafkaQueue` (pkg/queue/kafka.go) coexist in the same repository, doing substantially overlapping work.**
This is a real code-health smell worth naming directly rather than glossing over: two implementations of nearly the same responsibility (Kafka publish/DLQ, with `KafkaQueue` additionally unifying consume/commit) increase the surface area for behavioral drift — a bug fix or config change applied to one (e.g., adjusting `RetryAttempts` defaults) has no guarantee of being mirrored in the other, and any engineer encountering this repo cold has to spend time figuring out which one is "the real one" (currently `pkg.KafkaClient`, based on what `cmd/api`/`cmd/worker` actually import) versus which is either aspirational, abandoned, or in-progress. I'd flag this explicitly in a code review and push for either completing a migration to `pkg/queue.KafkaQueue` (the cleaner interface) across both binaries, or deleting it if it's confirmed dead.

**Q96. Staff-level system-design framing: if you were redesigning PulseGrid's job-submission consistency model from scratch today, knowing everything this codebase reveals about its actual failure modes, what would you build instead, and why?**
I'd build the outbox pattern from day one (DB insert of job + outbox row, single local transaction; a separate lightweight relay polling the outbox and publishing to Kafka, marking rows sent) rather than the current held-open-transaction-across-Kafka approach — it fully closes the orphan gap this codebase currently only partially mitigates and documents as a known risk, at the cost of one additional, fairly simple relay component whose own health (lag, crash-looping) needs monitoring. I'd pair it with making the worker's Postgres write-back mandatory-but-idempotent from the start (unique-constrained event rows, or upsert-based status updates) rather than retrofitting it later as this codebase currently needs to, since retrofitting a previously-absent write path onto an already-running at-least-once-delivery system is exactly where the idempotency risk discussed in Q62 originates.

**Q97. Evaluate whether this system's current architecture would survive a full AZ outage in its RDS's AZ, and identify every place in the stack that would be affected, not just RDS itself.**
With `db_multi_az = false` (default), RDS itself is unavailable until manual restore — a hard outage for the DB tier specifically. Cascading effects: the API's `/health` would report `postgres: unhealthy`, likely failing readiness probes and removing API pods from load-balancing (or failing liveness and restarting them repeatedly, per the earlier probe-design critique) even though the API process itself is fine; `POST /videos/upload` would fail at the `InsertJobTx` step for every request (a hard dependency, no fallback path once `dbClient != nil`); `GET /jobs`/`GET /jobs/{id}` would return `500`/`503`. Notably, the **worker tier would be entirely unaffected** — it has no DB client at all, so job processing (download, transcode, upload) would continue uninterrupted, silently accumulating jobs whose completion the system has no way to record or for clients to query, a strange but real "processing continues, visibility doesn't" partial-outage shape unique to this architecture's current DB-write-back gap.

**Q98. This system emits `pulsegrid_pod_resource_constrained` as a counter on `os.Exit(1)`. From a staff-level capacity-planning perspective, what would sustained growth in this metric actually tell you, and what wouldn't it tell you?**
It would tell you that worker pods are hitting OOM/disk-exhaustion often enough to require forced exits — a signal to either raise per-pod resource limits, reduce the number of renditions/HLS work attempted concurrently within a single job (currently sequential per-rendition within one job, but all sharing one pod's resource ceiling for the job's total duration), or investigate whether specific job characteristics (very large source files, very many requested renditions) correlate with constraint events. What it wouldn't tell you: *which specific resource* (memory vs. disk) was constrained — `ResourceConstraintError.Resource` carries this distinction in code (`"disk"` or `"memory"`) but the Prometheus counter itself has no corresponding label to preserve it (`PodResourceConstrained` is an unlabeled `prometheus.Counter`, `pkg/metrics.go:19,80-83`) — a real, fixable observability gap: the information exists in the error type but is discarded before it reaches the metric.

**Q99. Given everything in this guide, write the single most important sentence you'd want a hiring manager to remember about your understanding of this codebase.**
There's no single "correct" answer here — a strong response demonstrates synthesis, not recall. A good example: "PulseGrid's core engineering story is a deliberately pragmatic consistency model — ordered writes with a documented, accepted gap instead of a full outbox pattern — that works well for its actual failure modes, but the codebase's most valuable lesson is that its test suite, however thorough at the unit/mock level, never validated against a real migrated database, and that single gap let a CHECK-constraint bug that breaks every real submission ship undetected — which is a sharper, more durable lesson about testing strategy than any specific bug fix."

**Q100. If you were mentoring the engineer who wrote this codebase, what's the one piece of process (not code) feedback you'd give, based on the pattern of gaps found throughout this guide?**
The pattern across nearly every gap identified in this guide — the CHECK-constraint bug, the never-populated DLQ backlog metric, the worker's missing DB write-back, the KEDA/node-group capacity mismatch — is that each one is a **seam between two independently-true-looking pieces that were never validated together**: the schema and the insert code, the alert rule and the metric it depends on, the API's data model and the worker's actual write path, the autoscaler's config and the infrastructure's config. The process fix isn't "write more tests" in the abstract — it's specifically: **whenever you add a component that depends on another component's exact current state (a query depending on a schema, an alert depending on a metric, a scaler depending on node capacity), write one integration-level check that exercises both together**, not just each in isolation. That single habit would have caught every headline gap in this codebase.
